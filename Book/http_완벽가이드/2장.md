# 제2장: URL과 리소스

URL의 문법, 컴포넌트, 단축 URL, 인코딩 규칙, 주요 스킴, 그리고 URN을 통한 미래 방향을 다룬다.

---

### 1. 인터넷의 리소스 탐색하기

URL(Uniform Resource Locator)은 인터넷의 리소스를 가리키는 **표준 이름**이다. 리소스가 어디에 있고 어떻게 접근할 수 있는지 알려준다.

URL 이전에는 애플리케이션마다 접근 방식이 달랐다. FTP 서버에 접속해 익명 로그인하고 디렉터리를 이동한 뒤 바이너리 모드로 전환해서 파일을 받아야 했다. URL 덕분에 `ftp://ftp.lots-o-books.com/pub/complete-catalog.xls` 한 줄이면 된다.

**URI, URL, URN의 관계:**

```
         URI (Uniform Resource Identifier)
        ┌──────────┴──────────┐
       URL                   URN
   (위치 기반 식별)      (이름 기반 식별)
```

- **URI**: 리소스 식별자의 가장 일반화된 개념
- **URL**: 리소스가 **어디에** 있는지 설명하여 식별 (가장 흔한 형태)
- **URN**: 리소스의 위치와 상관없이 **이름**으로 식별

> HTTP 명세에서는 URI를 사용하지만, 실제로 대부분 URL을 가리킨다. 이 책에서도 URI와 URL을 같은 의미로 사용한다.

### 2. URL 문법

대부분의 URL은 `스킴://서버위치/경로` 구조를 따른다. 일반적인 URL 문법의 9개 컴포넌트는 다음과 같다.

```
<스킴>://<사용자 이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>
```

가장 중요한 세 가지는 **스킴, 호스트, 경로**이다.

| 컴포넌트 | 설명 | 기본값 |
|----------|------|--------|
| **스킴** | 리소스에 접근하기 위한 프로토콜 (예: `http`, `ftp`) | 없음 |
| **사용자 이름** | 리소스 접근에 필요한 사용자 이름 | `anonymous` |
| **비밀번호** | 사용자의 비밀번호. 사용자 이름에 `:`으로 이어서 기술 | (이메일 주소) |
| **호스트** | 리소스를 호스팅하는 서버의 호스트 명 또는 IP 주소 | 없음 |
| **포트** | 서버가 열어놓은 포트 번호 | 스킴에 따라 다름 (HTTP: 80) |
| **경로** | 서버 내 리소스의 위치. `/`로 경로조각을 구분 | 없음 |
| **파라미터** | 특정 스킴에서 사용하는 이름/값 쌍. `;`으로 구분 | 없음 |
| **질의(쿼리)** | 게이트웨이/DB 등에 파라미터를 전달. `?` 뒤에 기술, `&`로 구분 | 없음 |
| **프래그먼트** | 리소스의 특정 부분을 가리키는 이름. `#` 뒤에 기술 | 없음 |

#### 2.1 스킴

리소스에 어떻게 접근하는지 알려주는 중요한 정보. URL의 나머지 부분과 첫 번째 `:`로 구분된다. 대소문자를 가리지 않는다.

```
http://www.joes-hardware.com  ==  HTTP://www.joes-hardware.com
```

#### 2.2 호스트와 포트

- **호스트**: 호스트 명(`www.joes-hardware.com`) 또는 IP 주소(`161.58.228.45`)
- **포트**: 서버가 열어놓은 네트워크 포트. HTTP의 기본 포트는 80

```
http://www.joes-hardware.com:80/index.html
http://161.58.228.45:80/index.html       ← 같은 리소스
```

#### 2.3 사용자 이름과 비밀번호

FTP 등 일부 스킴은 사용자 인증이 필요하다. `@`로 호스트와 구분, `:`로 사용자 이름과 비밀번호를 구분한다.

```
ftp://ftp.prep.ai.mit.edu/pub/gnu                       ← 사용자 이름: anonymous (기본값)
ftp://anonymous@ftp.prep.ai.mit.edu/pub/gnu              ← 사용자 이름 명시
ftp://anonymous:my_passwd@ftp.prep.ai.mit.edu/pub/gnu    ← 사용자 이름 + 비밀번호
```

#### 2.4 경로

서버 내 리소스의 위치를 가리킨다. 유닉스 파일 시스템과 유사한 계층 구조(`/`로 구분).

```
http://www.joes-hardware.com:80/seasonal/index-fall.html
                                ──────────┬─────────────
                                        경로
```

각 경로조각은 자체 **파라미터**를 가질 수 있다.

```
http://www.joes-hardware.com/hammers;sale=false/index.html;graphics=true
                                     ──────┬──────  ──────────┬────────
                                  hammers의 파라미터   index.html의 파라미터
```

#### 2.5 질의 문자열

데이터베이스·검색 엔진 등의 게이트웨이에 요청 범위를 좁히기 위한 파라미터를 전달한다.

```
http://www.joes-hardware.com/inventory-check.cgi?item=12731&color=blue&size=large
                                                 ──────────────┬──────────────────
                                                         질의 컴포넌트
```

편의상 많은 게이트웨이가 `&`로 나뉜 `이름=값` 쌍 형식을 사용한다.

#### 2.6 프래그먼트

리소스의 **특정 부분**을 가리킨다. URL 끝에 `#`으로 구분.

```
http://www.joes-hardware.com/tools.html#drills
                                        ──┬──
                                      프래그먼트
```

> **핵심**: 프래그먼트는 **서버에 전달되지 않는다**. HTTP 서버는 객체 전체를 전송하고, 브라우저가 전체 리소스를 받은 후 프래그먼트를 사용하여 해당 부분을 표시한다.

### 3. 단축 URL

#### 3.1 상대 URL

**절대 URL**은 리소스 접근에 필요한 모든 정보를 담고 있다. **상대 URL**은 기저(base) URL을 기준으로 불완전한 경로만 기술한다.

```html
<!-- http://www.joes-hardware.com/tools.html 문서 내의 링크 -->
<A HREF="./hammers.html">hammers</A>
```

변환 과정:
1. **기저 URL** 결정: `http://www.joes-hardware.com/tools.html`
2. 상대 URL `./hammers.html`에서 빠진 스킴·호스트를 기저 URL로부터 **상속**
3. 최종 절대 URL: `http://www.joes-hardware.com/hammers.html`

**기저 URL을 찾는 방법:**
- HTML `<BASE>` 태그로 명시적 제공
- 해당 리소스의 URL을 기저 URL로 사용
- 기저 URL이 없으면 절대 URL만으로 구성된 것 (또는 불완전/깨진 URL)

> 상대 URL을 사용하면 문서 집합의 위치를 변경하더라도 새로운 기저 URL에 의해 해석되므로, 리소스 집합을 쉽게 이동·미러링할 수 있다.

#### 3.2 URL 확장

브라우저가 URL 입력을 자동으로 완성해주는 기능.

| 확장 방식 | 설명 |
|-----------|------|
| **호스트 명 확장** | `yahoo` 입력 → `www.yahoo.com`으로 자동 확장. 프락시 사용 시 문제 발생 가능 |
| **히스토리 확장** | 과거 방문 URL 기록에서 일치하는 URL을 자동 제안. `http://www.joes-` 입력 → `http://www.joes-hardware.com` 제시 |

### 4. 안전하지 않은 문자와 인코딩

#### 4.1 설계 배경

URL은 여러 인터넷 프로토콜을 통해 **안전하게 전송**될 수 있도록 설계되었다. SMTP 같은 프로토콜은 특정 문자를 제거할 수 있으므로, URL은 상대적으로 작고 안전한 알파벳 문자만 포함하도록 제한한다.

그러나 이진 데이터나 비-라틴 문자를 포함해야 하는 경우를 위해 **이스케이프 인코딩**을 지원한다.

#### 4.2 URL 문자 집합

URL은 기본적으로 **US-ASCII** 문자 집합을 사용한다. US-ASCII에서 지원하지 않는 문자(유럽 언어 변형 문자, 비-라틴계 문자, 이진 데이터 등)는 **이스케이프 문자열**로 인코딩한다.

#### 4.3 인코딩 체계

안전하지 않은 문자를 `%` + **두 자리 16진수 ASCII 코드**로 변환한다.

| 문자 | ASCII 코드 | 인코딩 예시 |
|------|-----------|------------|
| `~` | 126 (0x7E) | `http://www.joes-hardware.com/%7Ejoe` |
| 빈 문자 | 32 (0x20) | `http://www.joes-hardware.com/more%20tools.html` |
| `%` | 37 (0x25) | `http://www.joes-hardware.com/100%25satisfaction.html` |

#### 4.4 문자 제한

URL 내에서 특별한 의미로 **예약된 문자**들:

| 문자 | 선점/제한 이유 |
|------|---------------|
| `%` | 이스케이프 토큰 |
| `/` | 경로 세그먼트 구분 |
| `.` | 경로 컴포넌트에서 선점 |
| `#` | 프래그먼트 구획 문자 |
| `?` | 질의 문자열 구획 문자 |
| `;` | 파라미터 구획 문자 |
| `:` | 스킴, 사용자 이름/비밀번호, 호스트/포트 구획 |
| `@` `&` `=` | 특정 스킴에서 특별한 의미 |
| `{` `}` `\|` `\` `^` `~` `[` `]` `` ` `` | 게이트웨이 등에서 불안전하게 처리될 수 있음 |
| `<` `>` `"` | URL 범위 밖에서 역할이 있으므로 반드시 인코딩 |

> 클라이언트 애플리케이션에서 안전하지 않거나 제한된 문자를 URL로 보내기 전에 인코딩하는 것이 좋다.

### 5. 스킴의 바다

주요 URL 스킴:

| 스킴 | 기본 포트 | 기본 형식 | 설명 |
|------|----------|-----------|------|
| **http** | 80 | `http://<호스트>:<포트>/<경로>?<질의>#<프래그먼트>` | 하이퍼텍스트 전송 프로토콜. 사용자 이름/비밀번호 없음 |
| **https** | 443 | `https://<호스트>:<포트>/<경로>?<질의>#<프래그먼트>` | HTTP + **SSL/TLS** 암호화. 문법은 HTTP와 동일 |
| **ftp** | 21 | `ftp://<사용자>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>` | 파일 전송 프로토콜. 파일 업로드/다운로드, 디렉터리 목록 조회 |
| **mailto** | — | `mailto:<RFC-822-addr-spec>` | 이메일 주소. 표준 URL과 다른 형식 |
| **rtsp/rtspu** | 554 | `rtsp://<사용자>:<비밀번호>@<호스트>:<포트>/<경로>` | 실시간 스트리밍 프로토콜. `rtspu`는 UDP 사용 |
| **file** | — | `file://<호스트>/<경로>` | 로컬 또는 네트워크 파일 시스템의 파일. 호스트 생략 시 localhost |
| **news** | — | `news:<newsgroup>` 또는 `news:<news-article-id>` | 뉴스 그룹/기사 접근. 호스트 정보를 포함하지 않는 특이한 구조 |
| **telnet** | 23 | `telnet://<사용자>:<비밀번호>@<호스트>:<포트>/` | 대화형 서비스 접근 |

### 6. URL의 한계와 URN의 미래

**URL의 근본적 한계**: URL은 **주소**이지 실제 **이름**이 아니다. 리소스가 옮겨지면 기존 URL은 더 이상 유효하지 않다.

**URN(Uniform Resource Name)**: 객체의 위치와 상관없이 항상 해당 객체를 가리킬 수 있는 이름을 제공한다.

**PURL(Persistent Uniform Resource Locator)**: URL로 URN의 기능을 제공하는 중간 해법. 리소스의 실제 URL 목록을 관리하는 **리소스 위치 중재 서버**를 두고, 클라이언트는 영구적인 URL을 요청하면 실제 URL로 리다이렉트된다.

```
클라이언트 ──(1) PURL 요청──→ 리소스 리졸버 ──→ 현재 위치(실제 URL) 반환
           ←─(2) 리다이렉트──┘
           ──(3) 실제 URL로 리소스 요청──→ 실제 서버
```

> URL에서 URN으로의 전환은 표준화, HTTP 애플리케이션 수정, 벤더 합의 등 막대한 작업이 필요하기 때문에 아직 완전히 이루어지지 않았다. URL은 한계를 가지고 있지만, 당분간 인터넷 리소스를 명명하는 주된 방법으로 계속 사용될 것이다.

### 보충: 책 이후의 변화와 실무 지식

#### B1. URI 표준의 진화

| 표준 | 연도 | 비고 |
|------|------|------|
| RFC 2396 | 1998 | 이 책이 참조하는 URI 문법 표준 |
| **RFC 3986** | **2005** | RFC 2396을 대체한 **현행 URI 표준**. 문법 규칙 명확화, 상대 참조 해석 알고리즘 개선 |
| **WHATWG URL Standard** | 지속 갱신 | 브라우저가 실제로 따르는 **Living Standard**. RFC 3986과 미묘한 차이가 있으며, 브라우저 호환성이 목표 |

> 실무에서는 RFC 3986(서버/API 개발)과 WHATWG URL Standard(브라우저/프론트엔드)를 모두 알아두는 것이 좋다.

#### B2. IRI — 국제화 리소스 식별자

책에서는 URL이 US-ASCII만 지원한다고 설명하지만, **RFC 3987 (2005)**에서 **IRI(Internationalized Resource Identifier)**가 도입되었다.

```
IRI:  http://예시.한국/경로/문서.html      ← 유니코드 직접 사용
URI:  http://xn--9n2bp8q.xn--3e0b707e/%EA%B2%BD%EB%A1%9C/%EB%AC%B8%EC%84%9C.html
```

- IRI는 유니코드(UTF-8)를 허용하며, 전송 시 자동으로 퍼센트 인코딩된 URI로 변환된다.
- 현대 브라우저는 주소창에 IRI를 표시하지만, 내부적으로는 인코딩된 URI로 처리한다.

#### B3. Punycode와 국제화 도메인(IDN)

호스트 명에 비-ASCII 문자를 사용하기 위한 **IDN(Internationalized Domain Name)** 체계.

```
사용자가 보는 도메인:  한국인터넷정보센터.한국
실제 DNS에 질의되는 도메인:  xn--3e0bx5euxnj7fuaij0a82atq0b7sa.xn--3e0b707e
```

- 도메인의 각 레이블을 **Punycode**(RFC 3492)로 인코딩하고 `xn--` 접두사를 붙인다.
- **보안 주의**: 키릴 문자 `а`(U+0430)가 라틴 `a`와 시각적으로 동일하여 **호모그래프 공격**(유사 도메인 피싱)에 악용될 수 있다. 현대 브라우저는 혼합 스크립트 도메인을 Punycode로 표시하여 방어한다.

#### B4. 현대 웹의 프래그먼트 활용 — SPA 라우팅

책에서는 프래그먼트를 "HTML 문서의 특정 부분을 가리키는 용도"로만 설명하지만, 현대 SPA(Single Page Application)에서는 **클라이언트 사이드 라우팅**에 핵심적으로 활용된다.

**해시 기반 라우팅 (Hash Router):**
```
http://myapp.com/#/users/123
http://myapp.com/#/settings
```
- 프래그먼트는 서버에 전달되지 않으므로, `#` 뒤의 경로가 바뀌어도 **페이지 새로고침 없이** 클라이언트에서 뷰를 전환한다.
- 서버는 항상 같은 HTML을 응답하면 되므로 서버 설정이 간단하다.

**History API 기반 라우팅 (Browser Router):**
```
http://myapp.com/users/123
http://myapp.com/settings
```
- HTML5 `history.pushState()`를 사용해 프래그먼트 없이 깔끔한 URL을 제공한다.
- 단, 서버에서 모든 경로에 대해 같은 HTML을 반환하도록 **fallback 설정**이 필요하다.

#### B5. `data:` URL 스킴

책에서 다루지 않는 현대적 스킴. 리소스를 URL **자체에 인라인으로 내장**한다 (RFC 2397).

```
data:text/html;charset=utf-8,<h1>Hello</h1>
data:image/png;base64,iVBORw0KGgoAAAANSUhEUg...
```

- 별도의 HTTP 요청 없이 작은 리소스를 직접 포함할 수 있다.
- CSS의 `background-image`, HTML의 `<img src>` 등에서 작은 아이콘을 인라인할 때 사용한다.
- 주의: Base64 인코딩 시 원본 대비 약 33% 크기 증가, 캐싱 불가.

#### B6. URL 인코딩 — JavaScript 실무

| 함수 | 인코딩 범위 | 용도 |
|------|------------|------|
| `encodeURI()` | URI 구분자(`:`, `/`, `?`, `#` 등)는 **인코딩하지 않음** | 전체 URL을 인코딩할 때 |
| `encodeURIComponent()` | URI 구분자를 **포함하여 인코딩** | 쿼리 파라미터 값 등 URL의 일부분을 인코딩할 때 |

```javascript
const query = '검색어=hello world&lang=ko';

encodeURI(`http://example.com/search?${query}`);
// http://example.com/search?%EA%B2%80%EC%83%89%EC%96%B4=hello%20world&lang=ko
// → &, = 등 구분자가 보존됨

encodeURIComponent(query);
// %EA%B2%80%EC%83%89%EC%96%B4%3Dhello%20world%26lang%3Dko
// → &, = 등도 모두 인코딩됨
```

> **실수하기 쉬운 점**: 쿼리 파라미터 **값**에 `&`나 `=`이 포함될 수 있을 때 반드시 `encodeURIComponent()`를 사용해야 한다. `encodeURI()`를 쓰면 구분자가 보존되어 파싱 오류가 발생한다.

현대 브라우저에서는 `URL` 및 `URLSearchParams` API를 사용하면 인코딩을 자동 처리할 수 있다.

```javascript
const url = new URL('http://example.com/search');
url.searchParams.set('q', 'hello world');
url.searchParams.set('lang', 'ko');
console.log(url.toString());
// http://example.com/search?q=hello+world&lang=ko
```

#### B7. URL 단축 서비스 — PURL 개념의 현대적 구현

책에서 소개한 PURL(영구 URL) 개념은 현대의 **URL 단축 서비스**(bit.ly, t.co 등)와 원리가 유사하다.

```
https://bit.ly/3xAbCdE  →  301/302 리다이렉트  →  https://www.example.com/very/long/path
```

- 단축 서비스 서버가 **리졸버** 역할을 하여 원래 URL로 리다이렉트한다.
- PURL과의 차이: 단축 서비스는 편의성·분석(클릭 통계)이 주 목적이고, PURL은 영속적 식별이 목적이다.
- **보안 주의**: 단축 URL은 목적지를 숨기므로 피싱에 악용될 수 있다. 신뢰할 수 없는 출처의 단축 URL은 미리보기 기능(`+` 붙이기 등)으로 확인하는 것이 좋다.

#### B8. HTTPS 기본화 — 현대 웹의 필수

책에서는 HTTPS를 "SSL을 사용하는 변형"으로 간단히 소개하지만, 현대 웹에서 HTTPS는 **사실상 필수**가 되었다.

- **2014**: Google이 HTTPS를 검색 랭킹 시그널로 채택
- **2017**: Chrome이 HTTP 페이지에 "주의 요함" 경고 표시 시작
- **2018**: Chrome 68부터 모든 HTTP 사이트를 "안전하지 않음"으로 표시
- **현재**: Let's Encrypt 등 무료 인증서 보급으로 HTTPS 도입 장벽이 크게 낮아짐

최신 브라우저의 주요 웹 API(Geolocation, Service Worker, Web Bluetooth 등)는 **HTTPS 컨텍스트에서만** 사용 가능하다.

#### B9. 경로 탐색 공격 (Path Traversal)

URL 경로에서 `../`(상위 디렉터리 참조)를 악용한 보안 공격.

```
http://example.com/files/../../../etc/passwd
```

- 웹 서버가 경로를 그대로 파일 시스템에 매핑하면, 의도하지 않은 시스템 파일에 접근할 수 있다.
- **방어**: 서버에서 경로 정규화(canonicalization) 후 허용된 디렉터리 범위 내인지 검증해야 한다. 대부분의 현대 웹 서버/프레임워크는 이를 기본적으로 차단한다.
