# 제1장: HTTP 개요

HTTP의 기본 개념, 작동 원리, 웹을 구성하는 핵심 요소를 다룬다.

---

### 1. HTTP — 인터넷의 멀티미디어 배달부

HTTP(HyperText Transfer Protocol)는 웹 브라우저와 서버가 통신하는 데 사용하는 **애플리케이션 계층 프로토콜**이다. JPEG 이미지, HTML 페이지, 텍스트, MPEG 동영상, WAV 음성 파일 등 수십억 개의 멀티미디어 리소스를 빠르고 간편하고 정확하게 전달한다.

HTTP는 **신뢰성 있는 데이터 전송 프로토콜**(TCP)을 사용하기 때문에, 데이터가 지구 반대편에서 오더라도 전송 중 손상되거나 꼬이지 않음을 보장한다. 덕분에 개발자는 전송 중 파괴·중복·왜곡을 걱정하지 않고 애플리케이션 고유 기능 구현에 집중할 수 있다.

### 2. 웹 클라이언트와 서버

- **웹 서버**: 웹 콘텐츠를 저장·제공하며, HTTP로 통신하므로 HTTP 서버라고도 부른다. 인터넷의 데이터를 저장하고 클라이언트가 요청한 데이터를 제공한다.
- **웹 클라이언트**: 대표적으로 웹 브라우저. 서버에 HTTP 요청을 보내고, 응답으로 받은 객체를 화면에 렌더링한다.

### 3. 리소스

웹 서버가 관리하고 제공하는 콘텐츠를 **웹 리소스**라 한다. 어떤 종류의 콘텐츠 소스든 리소스가 될 수 있다.

#### 3.1 리소스의 종류

| 종류 | 설명 |
|------|------|
| **정적 리소스** | 서버 파일 시스템에 저장된 HTML, 이미지, 텍스트, Word, PDF 등의 파일 |
| **동적 리소스** | 사용자·요청 내용·시간 등에 따라 실시간으로 콘텐츠를 생성하는 프로그램 (예: 라이브 영상, 주식 거래, 온라인 쇼핑몰) |

#### 3.2 MIME 타입

HTTP는 전송되는 각 객체에 **MIME 타입**(Multipurpose Internet Mail Extensions)이라는 데이터 포맷 라벨을 붙인다. 원래 각기 다른 전자메일 시스템 간 메시지 호환을 위해 설계되었으나, 워낙 잘 동작하여 HTTP에서도 멀티미디어 콘텐츠를 기술하는 데 채택되었다.

웹브라우저는 서버로부터 객체를 받을 때 MIME 타입을 통해 다룰 수 있는 객체인지 확인한다.

형식: `주 타입(primary type)/부 타입(subtype)`

| MIME 타입 | 설명 |
|-----------|------|
| `text/html` | HTML 문서 |
| `text/plain` | 일반 ASCII 텍스트 |
| `image/jpeg` | JPEG 이미지 |
| `image/gif` | GIF 이미지 |
| `video/quicktime` | 애플 퀵타임 동영상 |
| `application/vnd.ms-powerpoint` | 파워포인트 프레젠테이션 |

#### 3.3 URI, URL, URN

| 개념 | 설명 |
|------|------|
| **URI** (Uniform Resource Identifier) | 리소스를 고유하게 식별하는 통합 자원 식별자. URL과 URN을 포괄하는 상위 개념 |
| **URL** (Uniform Resource Locator) | 리소스의 구체적 **위치**를 서술. 스킴(프로토콜) + 서버 주소 + 리소스 경로로 구성 |
| **URN** (Uniform Resource Name) | 리소스의 위치에 독립적인 **유일한 이름**. 리소스를 옮기더라도 동작함 (예: `urn:ietf:rfc:2141`) |

```
http://www.joes-hardware.com/specials/saw-blade.gif
──┬─   ─────────┬────────── ────────┬───────────
스킴        서버 주소           리소스 경로
```

> URN은 리소스 위치를 분석하기 위한 인프라 지원이 필요하여 아직 널리 채택되지 않았다. 특별한 언급이 없으면 URI와 URL을 같은 의미로 사용한다.

### 4. HTTP 트랜잭션

하나의 HTTP 트랜잭션 = **요청 명령**(클라이언트 → 서버) + **응답 결과**(서버 → 클라이언트). 이 상호작용은 **HTTP 메시지**라는 정형화된 데이터 덩어리를 이용해 이루어진다.

#### 4.1 메서드

모든 HTTP 요청 메시지는 한 개의 메서드를 가지며, 서버에게 어떤 동작을 취해야 하는지 알려준다.

| 메서드 | 설명 |
|--------|------|
| `GET` | 서버에서 클라이언트로 지정한 리소스를 보내라 |
| `PUT` | 클라이언트에서 보낸 데이터를 지정한 이름의 리소스로 저장하라 |
| `DELETE` | 지정한 리소스를 서버에서 삭제하라 |
| `POST` | 클라이언트 데이터를 서버 게이트웨이 애플리케이션으로 보내라 |
| `HEAD` | 지정한 리소스에 대한 응답에서 HTTP 헤더 부분만 보내라 |

#### 4.2 상태 코드

모든 HTTP 응답 메시지는 **상태 코드**(세 자리 숫자)와 텍스트로 된 **사유 구절**(reason phrase)을 함께 반환한다. 사유 구절은 설명을 위한 것일 뿐, 실제 응답 처리에는 숫자 코드가 사용된다.

| 상태 코드 | 설명 |
|-----------|------|
| `200 OK` | 성공. 문서가 바르게 반환됨 |
| `302 Found` | 리다이렉션. 다른 곳에서 리소스를 가져가라 |
| `404 Not Found` | 리소스를 찾을 수 없음 |
| `500 Internal Server Error` | 서버 내부 오류 |

> 같은 상태 코드라도 사유 구절은 다를 수 있다: `200 OK`, `200 Document attached`, `200 Success` 등은 모두 같은 것으로 취급된다.

#### 4.3 복합 페이지

하나의 웹 페이지는 보통 하나의 리소스가 아닌 **리소스의 모음**이다. 브라우저는 HTML '뼈대'를 한 번의 트랜잭션으로 가져온 뒤, 첨부된 이미지·그래픽·자바 애플릿 등을 가져오기 위해 추가 HTTP 트랜잭션들을 수행한다. 이 리소스들은 서로 다른 서버에 위치할 수도 있다.

### 5. 메시지 구조

HTTP 메시지는 줄 단위의 **일반 텍스트** 구조이다. 이진 형식이 아니기 때문에 사람이 읽고 쓰기 쉽다.

```
시작줄(Start line)     ← 요청: 메서드 + URL + 버전 / 응답: 버전 + 상태코드 + 사유구절
헤더(Headers)          ← 이름: 값 쌍으로 메시지 속성 기술. 빈 줄로 종료
본문(Body)             ← 선택 사항. 텍스트 또는 임의의 이진 데이터(이미지, 비디오 등)
```

**요청 메시지 예시:**
```
GET /tools.html HTTP/1.1        ← 시작줄
Host: www.joes-hardware.com     ← 헤더
Accept: text/*
Accept-Language: en, fr
                                ← 빈 줄 (헤더 종료)
```

**응답 메시지 예시:**
```
HTTP/1.1 200 OK                 ← 시작줄
Content-type: text/html         ← 헤더
Content-length: 433

<HTML>...</HTML>                ← 본문
```

### 6. TCP 커넥션

HTTP는 애플리케이션 계층 프로토콜로, 네트워크 통신의 세부 사항은 **TCP/IP**에 맡긴다.

#### 6.1 TCP/IP

TCP는 다음을 제공한다:
- **오류 없는 데이터 전송**
- **순서에 맞는 전달** (데이터는 보낸 순서대로 도착)
- **조각나지 않는 데이터 스트림** (언제든 어떤 크기로든 보낼 수 있음)

일단 TCP 커넥션이 맺어지면, 클라이언트-서버 간 메시지가 없어지거나 손상되거나 순서가 뒤바뀌는 일은 결코 없다.

네트워크 계층 구조:
```
애플리케이션 계층   ← HTTP
전송 계층           ← TCP
네트워크 계층       ← IP
데이터 링크 계층
물리 계층
```

#### 6.2 접속 과정

1. **(a)** 웹브라우저가 서버의 URL에서 **호스트 명**을 추출한다.
2. **(b)** 웹브라우저가 호스트 명을 **DNS**를 통해 **IP 주소**로 변환한다.
3. **(c)** 웹브라우저가 URL에서 **포트 번호**(있다면)를 추출한다. (기본값 80)
4. **(d)** 웹브라우저가 웹 서버와 **TCP 커넥션**을 맺는다.
5. **(e)** 웹브라우저가 서버에 **HTTP 요청**을 보낸다.
6. **(f)** 서버가 웹브라우저에 **HTTP 응답**을 돌려준다.
7. **(g)** 커넥션이 닫히면, 웹브라우저가 **문서를 표시**한다.

#### 6.3 텔넷을 이용한 실습

HTTP는 이진 형식이 아닌 문자열 기반이므로, 텔넷으로 웹 서버와 직접 대화할 수 있다.

```bash
% telnet www.joes-hardware.com 80
Trying 128.121.66.211...
Connected to joes-hardware.com.
GET /tools.html HTTP/1.1          # 직접 타이핑
Host: www.joes-hardware.com       # 직접 타이핑
                                  # 빈 줄 입력 → 요청 완료
HTTP/1.1 200 OK                   # 서버 응답 시작
Content-Type: text/html
Content-Length: 433

<HTML>...</HTML>
```

> 더 유연한 도구로 `nc`(netcat)를 사용할 수도 있다. UDP/TCP 기반 트래픽을 조작하고 스크립트할 수 있다.

### 7. 프로토콜 버전

| 버전 | 주요 특징 |
|------|-----------|
| **HTTP/0.9** (1991) | `GET`만 지원, MIME 타입·헤더·버전 번호 없음. 간단한 HTML 객체 전용 프로토타입 |
| **HTTP/1.0** (1996) | 버전 번호, 헤더, 추가 메서드, 멀티미디어 객체 처리 도입. 첫 대중화 버전. 잘 정의된 명세라기보다 잘 동작하는 용례들의 모음에 가까움 |
| **HTTP/1.0+** | keep-alive 커넥션, 가상 호스팅, 프락시 연결 지원 등 비공식 확장 기능이 사실상 표준으로 추가된 버전 |
| **HTTP/1.1** (1997) | 구조적 결함 교정, 성능 최적화, 지속 커넥션, 파이프라이닝, 청크 전송 등. 오랫동안 사실상 표준 |
| **HTTP/2** (2015) | 구글 SPDY 기반. 바이너리 프레이밍, 멀티플렉싱, 헤더 압축(HPACK), 서버 푸시 도입 |
| **HTTP/3** (2022) | TCP 대신 **QUIC**(UDP 기반) 사용. 연결 설정 지연 최소화, HOL 블로킹 해소 |

> **책과 현실의 차이 — HTTP/2 · HTTP/3 보충**
>
> 이 책(원서 2002, 번역 2014)에서는 HTTP/2.0을 "구글 SPDY 기반으로 설계가 진행 중인 프로토콜"로만 소개한다. 이후 실제 표준화 과정은 다음과 같다.
>
> **HTTP/2 (RFC 7540, 2015)**
> - 구글이 2009년 발표한 **SPDY** 프로토콜이 직접적 기반이 되었다. SPDY는 HTTP/1.1의 성능 한계(HOL 블로킹, 헤더 중복 전송 등)를 해결하기 위해 만들어졌다.
> - 핵심 변화: 텍스트 기반 → **바이너리 프레이밍** 계층 도입. 하나의 TCP 커넥션 위에서 여러 요청/응답을 동시에 처리하는 **멀티플렉싱**, **HPACK** 헤더 압축, **서버 푸시**(클라이언트가 요청하기 전에 리소스를 선제 전송) 등이 추가되었다.
> - HTTP/1.1과 의미(메서드, 상태 코드, 헤더)는 동일하게 유지하면서 **전송 방식만** 개선한 것이 특징이다.
>
> **HTTP/3 (RFC 9114, 2022)**
> - HTTP/2는 멀티플렉싱을 도입했지만, 하위 계층인 TCP 자체의 **HOL(Head-of-Line) 블로킹** 문제는 해결하지 못했다. TCP에서 하나의 패킷이 유실되면 같은 커넥션의 모든 스트림이 대기해야 했다.
> - 이를 근본적으로 해결하기 위해 TCP 대신 구글이 개발한 **QUIC**(UDP 기반 전송 프로토콜)을 채택했다. QUIC은 각 스트림을 독립적으로 처리하여 한 스트림의 패킷 손실이 다른 스트림에 영향을 주지 않는다.
> - 추가로 **TLS 1.3이 프로토콜에 내장**되어 있어, TCP+TLS 핸드셰이크(최소 2~3 RTT)가 **1 RTT**(재연결 시 0-RTT)로 단축되었다.
>
> ```
> HTTP/1.1        HTTP/2          HTTP/3
> ─────────       ─────────       ─────────
>  HTTP            HTTP            HTTP
>  (텍스트)        (바이너리)      (바이너리)
>                  프레이밍         프레이밍
>  TCP             TCP             QUIC
>  TLS(선택)       TLS(사실상필수)  TLS 1.3(내장)
>  IP              IP              UDP / IP
> ```

### 8. 웹의 구성 요소

#### 8.1 프락시 (Proxy)

클라이언트와 서버 사이에 위치한 **HTTP 중개자**. 클라이언트의 모든 HTTP 요청을 받아 서버에 전달한다(대개 요청을 수정한 뒤). 주로 **보안**을 위해 사용되며, 요청과 응답을 **필터링**한다.

- 다운로드 시 애플리케이션 **바이러스 검출**
- 초등학교 학생들에 대한 **성인 콘텐츠 차단**
- 모든 웹 트래픽 흐름에서 **신뢰할 만한 중개자** 역할

#### 8.2 캐시 (Cache)

**웹 캐시**(캐시 프락시)는 자신을 거쳐 가는 문서들 중 자주 찾는 것의 사본을 저장해 두는 특별한 종류의 HTTP 프락시 서버다. 클라이언트는 멀리 떨어진 웹 서버보다 근처의 캐시에서 훨씬 더 빨리 문서를 다운받을 수 있다.

HTTP는 캐시를 효율적으로 동작하게 하고, 캐시된 콘텐츠를 최신 버전으로 유지하면서 동시에 **프라이버시도 보호**하기 위한 많은 기능을 정의한다.

#### 8.3 게이트웨이 (Gateway)

다른 서버들의 중개자로 동작하는 특별한 서버. 주로 **HTTP 트래픽을 다른 프로토콜로 변환**하기 위해 사용된다. 게이트웨이는 언제나 스스로가 리소스를 갖고 있는 **진짜 서버인 것처럼** 요청을 다루므로, 클라이언트는 게이트웨이와 통신하고 있음을 알아채지 못한다.

- 예: HTTP/FTP 게이트웨이는 HTTP 요청을 받아 FTP 프로토콜로 문서를 가져온 뒤, HTTP 응답에 담아 반환한다.

#### 8.4 터널 (Tunnel)

두 커넥션 사이에서 **날(raw) 데이터를 열어보지 않고 그대로 전달**해주는 HTTP 애플리케이션이다. 주로 비-HTTP 데이터를 하나 이상의 HTTP 연결을 통해 전송하기 위해 사용한다.

- 대표적 예: 암호화된 **SSL 트래픽**을 HTTP 커넥션으로 전송하여, 웹 트래픽만 허용하는 사내 **방화벽을 통과**시킬 수 있다.

#### 8.5 에이전트 (Agent)

사용자를 위해 HTTP 요청을 만들어주는 클라이언트 프로그램. 웹브라우저 외에도 사람의 통제 없이 스스로 웹을 돌아다니며 HTTP 트랜잭션을 일으키는 **자동화된 에이전트**가 있다.

- **스파이더(웹 로봇)**: 웹을 돌아다니며 검색 엔진의 데이터베이스나 가격 비교 로봇을 위한 제품 카탈로그 등 유용한 콘텐츠 보관소를 구축한다.
