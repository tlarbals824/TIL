# 10장. HTTP-NG

HTTP-NG(HTTP: The Next Generation)는 1997년 W3C가 주도한 차세대 HTTP 설계 프로젝트다. HTTP/1.1의 한계를 극복하기 위한 시도였으나 널리 채택되지는 못했다. 그러나 이후 HTTP/2, HTTP/3 등 현대 HTTP 발전 방향과 맥락을 같이 한다는 점에서 의미가 있다.

---

### HTTP의 성장통

HTTP/1.1이 원래 설계를 넘어서 과도하게 확장되면서 4가지 문제가 드러났다.

| 문제 | 설명 |
|------|------|
| **복잡성 (Complexity)** | 커넥션 관리, 메시지 처리, 서버 처리 로직이 뒤섞여 올바른 구현이 어렵고 오류가 많음 |
| **확장성 (Extensibility)** | 기존 HTTP 애플리케이션이 프로토콜 확장을 방해하는 비호환 동작을 포함. 자율적 기능 확장 메커니즘 부재 |
| **성능 (Performance)** | 성능 비효율이 내재되어 있어 고지연·저대역폭 무선 환경에서 더욱 심각해짐 |
| **전송 의존성 (Transport Dependence)** | TCP/IP에 종속적. 임베디드·무선 환경 등 대체 전송 스택 지원 부족 |

---

### HTTP-NG 활동

1997년 여름, W3C가 복잡성·확장성·성능·전송 의존성 문제를 해결할 차세대 HTTP 설계를 위한 특별 프로젝트를 시작했다. 1998년 12월 IETF 회의에서 제안서가 발표됐다.

---

### 핵심 철학: 모듈화와 개선 (Modularize and Enhance)

HTTP-NG의 핵심은 "**모듈화하고 개선하라**"다. HTTP/1.1이 단일 레이어에 모든 기능을 뒤섞은 것과 달리, HTTP-NG는 프로토콜을 3개 레이어로 분리했다.

```
┌─────────────────────────────────────┐
│  Layer 3: Web Application Functions │  ← HTTP/1.1 메서드, 헤더, WebDAV 등
├─────────────────────────────────────┤
│  Layer 2: Remote Operation Invoc.  │  ← Binary Wire Protocol
├─────────────────────────────────────┤
│  Layer 1: Message Transport        │  ← WebMUX
├─────────────────────────────────────┤
│  Underlying Network Transport      │  ← TCP/IP
└─────────────────────────────────────┘
```

#### Layer 1: 메시징 (Messaging)

**메시지 전송 레이어**는 메시지의 의미와 무관하게 효율적인 메시지 전달에만 집중한다. 무선 환경 등 다양한 네트워크 스택을 지원하도록 추상화된 API를 제공한다.

주요 기능:
- 파이프라이닝과 배칭으로 왕복 지연 감소
- 커넥션 재사용으로 지연 감소 및 대역폭 향상
- **단일 연결 위에서 여러 메시지 스트림을 병렬 다중화** → 슬로우 프로듀서로 인한 HOL(Head-of-Line) 블로킹 방지
- 효율적인 메시지 경계 구분

HTTP-NG 팀은 이 레이어를 위해 **WebMUX** 프로토콜을 제안했다.

#### Layer 2: 원격 호출 (Remote Invocation)

**원격 호출 레이어**는 클라이언트가 서버 리소스에 대한 연산을 호출하는 요청/응답 프레임워크를 제공한다.

- 특정 연산의 구현과 의미론(캐시, 보안, 메서드 로직 등)에는 무관심
- HTTP/1.1보다 확장 가능한 객체 지향 방식으로 서버 연산 호출
- CORBA, DCOM, Java RMI와 유사한 철학

HTTP-NG 팀은 이 레이어를 위해 **Binary Wire Protocol**을 제안했다.

#### Layer 3: 웹 애플리케이션 (Web Application)

**웹 애플리케이션 레이어**는 HTTP/1.1의 GET, POST, PUT 등 메서드와 헤더 파라미터, 그리고 WebDAV 같은 확장 기능을 포함한다. 원격 호출 레이어 위에서 동작하며 여러 애플리케이션이 공통 인프라를 공유하면서도 독립적으로 공존할 수 있다.

---

### WebMUX

WebMUX는 Layer 1을 위해 제안된 고성능 메시지 전송 프로토콜이다. 하나의 TCP 연결 위에 여러 메시지 스트림을 병렬로 실어나를 수 있다.

#### 주요 목표

| 특성 | 설명 |
|------|------|
| **단순한 설계** | 구현 복잡도 최소화 |
| **고성능** | 처리량과 응답속도 극대화 |
| **다중화 (Multiplexing)** | 여러 스트림을 단일 연결 위에서 동적으로 인터리빙 |
| **크레딧 기반 흐름 제어** | 수신자가 사전에 수신 의사를 표명해 데드락 방지 |
| **정렬 보존** | 멀티플렉스 스트림에서 데이터 정렬 유지 → 바이너리 데이터 효율 처리 |
| **풍부한 기능** | 소켓 API 지원 가능한 수준의 인터페이스 |

> 현대 HTTP/2의 **스트림 다중화**와 **흐름 제어**는 WebMUX의 아이디어와 매우 유사하다.

---

### Binary Wire Protocol

Binary Wire Protocol은 Layer 2를 위해 제안된 원격 연산 호출 프로토콜이다.

#### 핵심 개념

- **객체 타입(Object Type)**: 각 객체 타입에 URI가 할당되어 설명과 메서드 목록을 광고할 수 있음
- **연산 호출**: 클라이언트 → 서버, 연산 결과 응답: 서버 → 클라이언트
- **상태 유지 연결**: HTTP/1.1의 무상태성보다 효율적인 상태 유지 연결 제공
- **병렬 요청 순서 무관**: 응답에 요청 일련번호가 포함되어 비순차 응답 가능

---

### 분산 객체 (Distributed Objects)

HTTP-NG의 설계 철학은 CORBA, DCOM 등 구조적 분산 객체 시스템에서 많이 빌려왔다. HTTP와 분산 객체 시스템의 수렴을 주장하는 연구 커뮤니티가 1996년부터 활발히 활동했다.

---

### 현황과 의의

1998년 말, HTTP-NG 팀은 당시 업계가 HTTP/1.1에 막 적응하고 있는 상황에서 분산 객체 패러다임으로의 전면 재설계는 너무 급진적이라고 판단했다.

두 가지 대안 제안이 나왔다:
1. WebMUX 전송 기술에만 집중
2. XML 기반 유연한 프로토콜 타입 연구

결국 HTTP-NG 워킹 그룹은 해산됐고 공식 채택은 이루어지지 않았다.

#### 현대 HTTP와의 연결

HTTP-NG가 제안한 개념들은 이후 실제로 표준화됐다.

| HTTP-NG 제안 | 실제 구현 |
|-------------|----------|
| WebMUX 다중화 | HTTP/2 스트림 다중화 |
| 크레딧 기반 흐름 제어 | HTTP/2 흐름 제어 |
| 바이너리 프레이밍 | HTTP/2 바이너리 프레임 레이어 |
| 전송 독립성 | HTTP/3 (QUIC over UDP) |

> HTTP-NG는 실패한 프로젝트처럼 보이지만, 그 핵심 아이디어는 HTTP/2와 HTTP/3로 결실을 맺었다.

---

### 원문 기준으로 보완할 개념

#### HTTP-NG가 해결하려던 더 큰 문제

HTTP-NG는 단순한 성능 개선안이 아니라, HTTP를 **계층화된 범용 분산 애플리케이션 프로토콜**로 재구성하려는 시도였다.

- 전송 계층과 메시지 의미를 분리해 네트워크 환경 변화에 덜 민감하게 만들려 했다.
- 요청/응답을 더 일반적인 원격 연산 호출 모델로 추상화하려 했다.
- HTTP 확장을 "기존 문자열 기반 헤더를 조금 더 붙이는 방식"이 아니라 구조적으로 가능하게 만들려 했다.

#### 왜 실제 웹은 다른 길을 택했는가

| 이유 | 설명 |
|------|------|
| **배포 호환성** | 기존 브라우저, 서버, 프록시, 방화벽과의 호환이 너무 중요했음 |
| **중간 장비 현실** | 웹은 이미 수많은 프록시와 캐시에 둘러싸여 있었음 |
| **개발 모델 차이** | 웹은 문서/리소스 지향 모델이 강했고, 분산 객체 모델은 너무 급진적이었음 |
| **점진적 진화 선호** | 전체 교체보다 부분 개선이 실제 채택 가능성이 높았음 |

- 즉, HTTP-NG는 기술적으로 흥미로웠지만 웹 생태계의 배포 현실을 과소평가했다.

---

### 현대 HTTP와의 연결을 더 넓게 보기

#### HTTP/2, HTTP/3와의 비교

| 항목 | HTTP-NG의 방향 | 실제 표준의 선택 |
|------|----------------|------------------|
| **메시지 표현** | 바이너리 원격 호출 지향 | HTTP 의미는 유지하고 프레이밍만 바이너리화 |
| **다중화** | WebMUX | HTTP/2 스트림, HTTP/3 스트림 |
| **흐름 제어** | 크레딧 기반 제어 | HTTP/2/HTTP/3 스트림/연결 흐름 제어 |
| **전송 독립성** | 다양한 전송 스택 고려 | HTTP/3에서 QUIC 채택 |
| **진화 전략** | 재설계 | 하위 호환을 중시한 점진적 개선 |

#### 현대 프로토콜에서 실제로 추가된 요소

- HTTP/2는 **HPACK 헤더 압축**, 바이너리 프레임, 스트림 우선순위를 도입했다.
- HTTP/3는 QUIC 위에서 동작하며 TCP 수준의 HOL 블로킹을 줄였다.
- 브라우저 환경에서는 ALPN을 통한 프로토콜 협상과 TLS 사용이 사실상 표준 경로가 되었다.
- 반대로 CORBA류의 분산 객체 호출 모델은 웹의 주류 진화 경로가 되지 못했고, REST/리소스 지향 의미론이 유지되었다.

#### 실무적으로 이해해야 할 차이

| 오해하기 쉬운 점 | 실제 의미 |
|------------------|-----------|
| **HTTP/2면 HOL 블로킹이 없다** | 애플리케이션 계층의 블로킹은 줄지만 TCP 수준 HOL은 남아 있음 |
| **HTTP/3면 항상 더 빠르다** | 네트워크 특성, 중간 장비, 서버 구현에 따라 이점이 달라짐 |
| **바이너리 프로토콜이면 무조건 성공한다** | 성능 외에 배포, 디버깅, 호환성이 함께 맞아야 함 |

#### 오늘날 HTTP-NG를 읽는 의미

- 웹 프로토콜 진화는 "좋은 아이디어"보다 **점진적 배포 가능성**이 더 중요하다는 점을 보여준다.
- HTTP/2와 HTTP/3는 HTTP-NG의 문제의식을 이어받았지만, 의미론은 유지하고 전송 계층만 개선하는 쪽을 택했다.
- 최근의 `103 Early Hints`, 우선순위 힌트, 서버 푸시의 쇠퇴 같은 흐름도 "기능 추가"보다 "현실적인 운영성"이 더 중요하다는 교훈과 연결된다.
