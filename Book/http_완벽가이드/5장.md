# 5장. 웹 서버

웹 서버는 HTTP 요청을 처리하고 응답을 돌려주는 소프트웨어(또는 하드웨어)다. 이 장에서는 웹 서버의 종류, 요청 처리 흐름, 리소스 매핑, MIME 타입 처리, 그리고 리다이렉션까지 다룬다.

---

### 웹 서버의 종류

#### 범용 소프트웨어 웹 서버

표준 운영체제와 컴퓨터 하드웨어에서 동작하는 웹 서버 소프트웨어다. 가장 널리 쓰이는 오픈소스 웹 서버인 **Apache**와 마이크로소프트의 **IIS(Internet Information Services)**가 대표적이다.

#### 어플라이언스 웹 서버 (Web Server Appliance)

웹 서버 소프트웨어가 미리 설치된 전용 하드웨어 장비다. 별도의 OS 설치나 소프트웨어 설정 없이 바로 사용할 수 있어 간편하지만, 유연성이 낮다.

#### 내장형 웹 서버 (Embedded Web Server)

프린터, 가정용 네트워크 장비(공유기 등), IoT 기기처럼 소형 기기에 내장된 웹 서버다. 관리 인터페이스를 제공하는 목적으로 주로 사용된다.

---

### 웹 서버의 요청 처리 단계 (7단계)

웹 서버는 다음 7단계를 거쳐 클라이언트의 요청을 처리한다.

| 단계 | 설명 |
|------|------|
| 1. 커넥션 수립 | 클라이언트의 TCP 연결 요청을 수락하거나 원하지 않으면 닫는다 |
| 2. 요청 수신 | HTTP 요청 메시지(메서드, URL, 헤더, 바디)를 읽어 들인다 |
| 3. 요청 처리 | 요청 메시지를 파싱하여 메서드, 리소스, 헤더 등을 해석한다 |
| 4. 리소스 접근 | 요청한 리소스(파일, 동적 콘텐츠 등)에 접근한다 |
| 5. 응답 생성 | 올바른 헤더와 함께 HTTP 응답 메시지를 생성한다 |
| 6. 응답 전송 | 클라이언트에게 응답 메시지를 보낸다 |
| 7. 트랜잭션 로깅 | 처리한 요청과 응답에 대한 로그를 남긴다 |

---

### 리소스 매핑

#### docroot (문서 루트)

웹 서버는 요청 URI를 로컬 파일 시스템의 특정 디렉터리에 매핑한다. 이 기준 디렉터리를 **docroot**라고 한다.

```
# 예: Apache 설정
DocumentRoot /usr/local/httpd/files
```

클라이언트가 `/specials/saw-blade.gif`를 요청하면 서버는 `/usr/local/httpd/files/specials/saw-blade.gif` 파일을 찾는다.

#### 가상 호스팅 (Virtual Hosting)

하나의 물리적 서버에서 여러 웹 사이트를 호스팅하는 기능이다. 요청의 `Host` 헤더(HTTP/1.1) 또는 IP 주소를 기반으로 각각 다른 docroot에 매핑한다.

```
# 예: Host 헤더 기반 가상 호스팅
Host: www.joes-hardware.com → /docs/joe/
Host: www.marys-antiques.com → /docs/mary/
```

#### 디렉터리 목록과 기본 인덱스 파일

요청 URI가 파일이 아닌 디렉터리를 가리킬 때, 서버는 보통 `index.html` 또는 `index.htm`을 찾는다. 설정에 따라 디렉터리 목록을 보여주거나 에러를 반환하기도 한다.

---

### 동적 콘텐츠 리소스 매핑

#### CGI (Common Gateway Interface)

정적 파일이 아닌 **프로그램 실행 결과**를 응답으로 보내기 위해 개발된 표준 인터페이스다. 요청이 들어오면 서버가 외부 프로그램(Perl, Python, C 등)을 실행하고, 그 출력을 HTTP 응답으로 전달한다.

- URL에 `/cgi-bin/` 경로나 `?` 쿼리 문자열이 포함되어 있으면 CGI 요청임을 알 수 있다.
- 요청마다 새 프로세스를 생성하는 오버헤드가 있어 성능 문제가 있다.
- **FastCGI**: 프로세스를 데몬으로 유지해서 프로세스 생성 오버헤드를 제거한 개선판.

#### SSI (Server-Side Includes)

HTML 파일 안에 특수 태그를 삽입해두면 서버가 클라이언트에 전송하기 전에 동적 콘텐츠(날짜, 파일 내용, 다른 CGI 결과 등)를 삽입하는 기능이다.

### MIME 타입 설정

서버는 응답으로 보내는 리소스의 MIME 타입을 `Content-Type` 헤더에 담는다. MIME 타입은 주로 파일 확장자를 기준으로 결정된다.

| 설정 방식 | 설명 |
|----------|------|
| **확장자-타입 매핑** | `.html` → `text/html`, `.jpg` → `image/jpeg` 등 |
| **Magic typing** | 파일 내용(매직 바이트)을 분석해 타입을 결정 |
| **Explicit typing** | 특정 파일이나 디렉터리에 MIME 타입을 명시적으로 지정 |
| **Type negotiation** | 클라이언트가 수용 가능한 타입 목록을 보내면 서버가 최적 타입 선택 |

### 리다이렉션

서버는 클라이언트를 다른 URL로 안내해야 할 때 리다이렉션 응답을 보낸다.

| 상태 코드 | 의미 | 용도 |
|----------|------|------|
| **301 Moved Permanently** | 리소스가 영구적으로 이동 | URL 변경, 도메인 이전 |
| **302 Found (Moved Temporarily)** | 임시 이동 | A/B 테스트, 임시 유지보수 |
| **303 See Other** | 다른 URI로 GET 요청 유도 | POST 후 결과 페이지로 이동 |
| **307 Temporary Redirect** | HTTP/1.1의 임시 리다이렉션 | 메서드 보존 임시 이동 |

#### 리다이렉션의 주요 용도

- **사이트 이전**: 기존 URL을 새 URL로 영구 리다이렉션
- **부하 분산**: 요청을 여러 서버에 분산
- **친근한 URL 제공**: 짧고 기억하기 쉬운 URL → 실제 리소스 URL
- **서버 장애 처리**: 서버 다운 시 임시 대체 서버로 안내

```
# 301 리다이렉션 응답 예시
HTTP/1.1 301 Moved Permanently
Location: http://www.new-domain.com/new-path.html
```

### 원문 기준으로 보완할 개념

#### 웹 서버의 내부 아키텍처

웹 서버는 단순히 요청을 받아 파일을 읽는 프로그램이 아니라, **동시성 모델**에 따라 성능 특성이 크게 달라진다.

| 방식 | 특징 | 장단점 |
|------|------|--------|
| **단일 프로세스/단일 스레드** | 한 번에 하나의 요청 처리 | 구현은 단순하지만 동시성에 취약 |
| **멀티프로세스/멀티스레드** | 요청마다 프로세스/스레드 할당 | 이해하기 쉽지만 문맥 전환 비용이 큼 |
| **이벤트 기반** | 소수의 워커가 많은 연결을 비동기로 처리 | 대규모 keep-alive 연결에 유리 |

- 정적 파일 위주의 서버는 이벤트 기반 모델에서 강점을 보인다.
- 동적 애플리케이션 호출이 많으면 스레드/프로세스 모델 또는 별도 애플리케이션 서버와의 조합이 흔하다.

#### URL에서 로컬 리소스로 매핑할 때의 주의점

- 웹 서버는 URI를 파일 경로로 바꾸기 전에 `..`, 중복 슬래시, 인코딩된 경로 같은 요소를 정규화해야 한다.
- 이 과정이 부실하면 **디렉터리 탈출(Path Traversal)** 같은 취약점으로 이어질 수 있다.
- 디렉터리 인덱싱은 편리하지만, 설정을 잘못하면 내부 파일 목록이 외부에 노출될 수 있다.
- CGI/SSI 같은 동적 리소스와 일반 정적 파일을 같은 경로 체계에 섞어두면 권한 경계가 흐려질 수 있다.
- 실무에서는 `Alias`, `ScriptAlias`, 사용자 홈 디렉터리(`~user`) 같은 별도 매핑 규칙이 함께 쓰이며, 이 규칙이 docroot보다 우선 적용되기도 한다.
- 즉, "URI = docroot 아래 파일"은 가장 기본적인 형태일 뿐이고, 실제 서버는 여러 매핑 규칙을 조합한다.

#### 가상 호스팅의 전제 조건

- 이름 기반 가상 호스팅은 `Host` 헤더가 정확히 전달된다는 전제 위에서 동작한다.
- 현대 HTTPS 환경에서는 `Host`뿐 아니라 TLS의 **SNI(Server Name Indication)** 도 함께 고려해야 한다.
- 같은 IP를 공유하는 여러 사이트를 운영할 때는 인증서 선택, 리다이렉션 정책, 기본 호스트 설정이 함께 맞아야 한다.

### 현대 웹 환경에서의 보충

#### 오늘날의 웹 서버 배치 구조

현대 서비스에서는 애플리케이션 서버가 인터넷에 직접 노출되기보다, 웹 서버 또는 역방향 프록시가 앞단에 놓이는 구성이 일반적이다.

```
클라이언트
→ CDN / WAF
→ 역방향 프록시(Nginx, Envoy, Caddy 등)
→ 애플리케이션 서버
→ 데이터 저장소
```

- 앞단 서버는 정적 파일 제공, TLS 종료, 압축, 리다이렉션, 속도 제한, 보안 헤더 추가를 맡는다.
- 애플리케이션 서버는 비즈니스 로직에 집중한다.
- 대규모 서비스에서는 같은 역할을 CDN이 전 세계 엣지에서 일부 대신 수행한다.

#### HTTP/2, HTTP/3가 웹 서버에 미친 영향

| 항목 | HTTP/1.1 중심 환경 | HTTP/2, HTTP/3 환경 |
|------|--------------------|---------------------|
| **연결 사용** | 여러 TCP 연결을 병렬로 여는 경우가 많음 | 하나의 연결에서 다중 스트림 처리 |
| **성능 병목** | 헤더 중복, 연결 수 증가 | 다중화와 헤더 압축으로 완화 |
| **보안 배치** | 평문 HTTP 허용 사례 존재 | 실서비스는 TLS + ALPN 조합이 사실상 표준 |

- HTTP/2는 서버가 적은 연결로 더 많은 요청을 처리하도록 만들었다.
- HTTP/3는 QUIC 기반이므로 커널 TCP 튜닝보다 사용자 공간 네트워크 스택과 TLS 설정이 더 중요해졌다.

#### 실무에서 자주 함께 설정하는 보안/운영 항목

| 항목 | 이유 |
|------|------|
| `Strict-Transport-Security` | HTTPS 강제 |
| `X-Content-Type-Options: nosniff` | 브라우저의 MIME 스니핑 방지 |
| 디렉터리 목록 비활성화 | 내부 파일명 노출 방지 |
| 업로드 크기/시간 제한 | 대용량 요청과 느린 공격 완화 |
| 접근/에러 로그 분리 | 운영 진단과 보안 분석 용이 |

#### 현대적인 리다이렉션 사용 팁

- **308 Permanent Redirect**는 301과 비슷하지만 메서드와 바디를 보존하는 영구 리다이렉션이다.
- 로그인 후 이동, 결제 완료 후 결과 페이지 이동처럼 메서드 전환이 필요한 경우에는 303이 더 명확하다.
- 도메인 정규화(`example.com` → `www.example.com`, `http` → `https`)는 검색엔진 최적화와 쿠키 범위 관리 측면에서도 중요하다.
- 서버가 언어, 문자셋, 인코딩에 따라 다른 표현을 제공한다면 콘텐츠 협상 결과와 캐시 정책도 함께 설계해야 한다.
