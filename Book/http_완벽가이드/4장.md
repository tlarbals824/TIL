# 제4장: 커넥션 관리

HTTP 커넥션의 내부 동작, TCP 성능 고려사항, 병렬·지속·파이프라인 커넥션 최적화를 다룬다.

---

### 1. TCP 커넥션

HTTP 통신의 거의 대부분은 **TCP/IP** 위에서 이루어진다. TCP는 신뢰할 수 있는 데이터 전송을 보장하며, 한번 커넥션이 맺어지면 메시지가 손실·손상·순서 역전되는 일이 없다.

#### 1.1 TCP 신뢰성 있는 데이터 파이프

HTTP 커넥션 = TCP 커넥션 + 몇 가지 규칙. TCP는 **신뢰성 있는 비트 파이프**로, 한쪽에서 집어넣은 바이트는 반대편에서 순서 그대로 올바르게 나온다.

#### 1.2 TCP 스트림 — IP 패킷으로 세그먼트화

TCP는 데이터를 **IP 패킷(데이터그램)**이라는 작은 청크로 나눠서 전송한다. 각 IP 패킷은:
- IP 패킷 헤더 (보통 20바이트): 발신·수신 IP 주소, 크기, 기타 플래그
- TCP 세그먼트 헤더 (보통 20바이트): TCP 포트 번호, 시퀀스 번호, 제어 플래그
- TCP 데이터 청크 (0 이상의 바이트)

프로토콜 스택:
```
(a) HTTP                  (b) HTTPS
─────────                 ─────────
 HTTP (애플리케이션)       HTTP (애플리케이션)
 TCP  (전송)              TLS/SSL (보안)
 IP   (네트워크)           TCP (전송)
 네트워크 인터페이스       IP  (네트워크)
```

#### 1.3 TCP 커넥션 유지 — 포트 번호

컴퓨터는 여러 TCP 커넥션을 동시에 유지할 수 있다. TCP는 **포트 번호**로 커넥션을 구분한다. TCP 커넥션은 다음 네 값으로 고유하게 식별된다:

```
<발신지-IP, 발신지-포트, 목적지-IP, 목적지-포트>
```

네 값이 모두 같은 두 커넥션은 존재할 수 없다.

#### 1.4 TCP 소켓 프로그래밍

OS는 소켓 API를 통해 TCP 커넥션을 조작하는 기능을 제공한다.

| 소켓 API 호출 | 설명 |
|---------------|------|
| `s = socket(...)` | 새 소켓 생성 |
| `bind(s, <로컬IP:포트>)` | 소켓에 로컬 포트 할당 |
| `connect(s, <원격IP:포트>)` | 원격 호스트에 연결 |
| `listen(s, ...)` | 연결 수락 허용 |
| `s2 = accept(s)` | 연결 대기 |
| `n = read(s, buf, n)` | 소켓에서 읽기 |
| `n = write(s, buf, n)` | 소켓에 쓰기 |
| `close(s)` | 커넥션 완전 닫기 |
| `shutdown(s, <side>)` | 입력 또는 출력만 닫기 |

### 2. TCP 성능 고려사항

HTTP는 TCP 위에 계층화되어 있으므로 HTTP 트랜잭션의 성능은 **TCP 성능에 크게 의존**한다.

#### 2.1 HTTP 트랜잭션 지연 원인

직렬 HTTP 트랜잭션의 주요 지연 단계:

1. **DNS 조회**: URL의 호스트명을 IP로 변환. 캐시되지 않은 경우 수십 초 소요 가능
2. **TCP 커넥션 설정**: 매 새 커넥션마다 SYN/SYN+ACK 핸드셰이크 필요. 1~2초 걸릴 수 있으며, 수백 트랜잭션이면 누적됨
3. **HTTP 요청 전송**: 요청 메시지가 서버까지 도달하는 시간
4. **HTTP 응답 수신**: 서버 처리 + 응답 전송 시간

> 실제 처리 시간보다 **TCP 네트워크 지연**이 훨씬 큰 경우가 많다.

#### 2.2 TCP 커넥션 핸드셰이크 지연

새 TCP 커넥션을 열 때, 데이터를 보내기 전에 핸드셰이크가 필요하다:

```
클라이언트                    서버
    ──── (a) SYN ──────────▶
    ◀─── (b) SYN + ACK ────
    ──── (c) ACK + HTTP 요청▶   (데이터 전송 시작)
    ◀─── (d) HTTP 응답 ────
```

1. 클라이언트가 작은 TCP 패킷(40~60 바이트)에 `SYN` 플래그를 설정해 보낸다
2. 서버가 `SYN + ACK` 패킷으로 수락 의사를 알린다
3. 클라이언트가 `ACK`로 응답하며, 이 시점부터 데이터를 보낼 수 있다

작은 HTTP 트랜잭션은 설정 시간의 50% 이상을 TCP 핸드셰이크에 소비할 수 있다.

#### 2.3 지연 ACK (Delayed Acknowledgments)

TCP는 신뢰성을 위해 각 세그먼트에 **시퀀스 번호와 체크섬**을 붙이고, 수신자는 ACK 패킷을 보낸다. ACK는 작으므로 **같은 방향으로 보낼 데이터 패킷에 편승(piggyback)**시키면 효율적이다.

**지연 ACK 알고리즘**: ACK를 버퍼에 잠시 보류(보통 100~200ms)하고, 같은 방향으로 보낼 데이터가 있으면 그 패킷에 실어 보낸다. 없으면 그냥 ACK를 보낸다.

문제: HTTP는 요청-응답의 **양방향 교환** 패턴이라 피기백할 패킷이 없는 경우가 많다. 이로 인해 지연 ACK 알고리즘이 오히려 지연을 유발한다.

#### 2.4 TCP 슬로우 스타트 (Slow Start)

새 TCP 커넥션의 성능은 **"튜닝된"** 기존 커넥션보다 낮다. TCP는 처음에 전송 속도를 제한하고, 전송이 성공할수록 속도를 올린다(**혼잡 윈도우 확장**).

- 패킷 1개 전송 → ACK 수신 → 패킷 2개 가능 → ACK → 4개 → ...
- 이미 데이터를 많이 교환한 **"튜닝된" 커넥션**이 더 빠르다
- 따라서 기존 커넥션을 재사용(**지속 커넥션**)하면 슬로우 스타트를 피할 수 있다

#### 2.5 Nagle 알고리즘과 TCP_NODELAY

**Nagle 알고리즘**: 작은 패킷을 묶어서 보내는 방식. 각 TCP 세그먼트는 최소 40바이트의 헤더를 가지므로, 아주 작은 데이터를 자주 보내면 비효율적이다.

- 전체 크기 패킷(LAN: ~1500바이트)이 아니면, 전송 중인 패킷이 없을 때만 전송
- HTTP 성능에 악영향: 작은 HTTP 메시지가 패킷을 채우지 못하거나, 지연 ACK와 충돌해 200ms 지연 발생

해결: `TCP_NODELAY` 파라미터를 설정해 Nagle 알고리즘을 비활성화. 단, 이 경우 충분히 큰 청크를 써야 작은 패킷 폭풍을 방지할 수 있다.

#### 2.6 TIME_WAIT 누적과 포트 고갈

TCP 커넥션을 닫으면, 이전 커넥션과 같은 주소·포트 번호의 새 커넥션이 일정 시간(2MSL, 보통 2분) 내에 생성되지 않도록 제어 블록을 메모리에 유지한다. 이는 지연된 패킷이 새 커넥션에 혼입되는 것을 방지한다.

성능 테스트 환경에서 같은 클라이언트-서버 조합을 짧은 시간 내에 수천 번 연결하면, 사용 가능한 소스 포트가 부족해 연결 속도가 제한될 수 있다(포트 고갈). 대략 60,000포트 / 120초 ≈ 500 트랜잭션/초가 한계.

### 3. HTTP 커넥션 핸들링

#### 3.1 Connection 헤더

HTTP는 클라이언트-서버 사이에 여러 중개자(프락시, 캐시 등)가 있을 수 있다. 일부 헤더는 **인접한 두 애플리케이션 사이에서만** 유효하고 중개자를 거쳐 전달되면 안 된다.

`Connection` 헤더는 세 가지 종류의 토큰을 가질 수 있다:
- HTTP 헤더 필드명: 이 커넥션에만 해당하는 헤더 이름 (다음 홉으로 전달하기 전에 삭제)
- 비표준 옵션 값: 이 커넥션에 대한 비표준 설정
- `close`: 이 커넥션을 요청 처리 후 닫겠다는 선언

중개자는 `Connection` 헤더와, 그 안에 나열된 헤더들을 다음 홉으로 전달하기 전에 **반드시 삭제**해야 한다. Proxy-Authenticate, Transfer-Encoding, Upgrade 등도 프락시되어서는 안 된다.

#### 3.2 직렬 트랜잭션 지연

이미지 3개를 포함한 웹 페이지를 나이브하게 처리하면, 브라우저는 4번의 HTTP 트랜잭션을 순서대로 수행한다. 각 트랜잭션마다 새 커넥션을 열면 커넥션 설정 + 슬로우 스타트 지연이 4번 반복된다.

```
[Connect-1] [Req-1/Resp-1] [Connect-2] [Req-2/Resp-2] [Connect-3] ...
```

이를 해결하기 위한 네 가지 기법:
1. **병렬 커넥션**: 여러 TCP 커넥션을 통한 동시 HTTP 요청
2. **지속 커넥션**: TCP 커넥션 재사용 (connect/close 지연 제거)
3. **파이프라인 커넥션**: 하나의 공유 TCP 커넥션 위에서 병렬 HTTP 요청
4. **다중화 커넥션**: 요청·응답을 청크 단위로 교차 전송 (실험적)

### 4. 병렬 커넥션

HTTP는 클라이언트가 여러 커넥션을 열어 **복수의 HTTP 트랜잭션을 병렬로** 처리할 수 있게 한다.

#### 4.1 병렬 커넥션의 장점

복합 페이지(여러 임베디드 객체)를 로드할 때, 단일 커넥션의 유휴 시간과 대역폭을 다른 객체 로딩에 활용할 수 있다. 이미지들의 커넥션 설정 지연이 **겹쳐지므로(overlapped)** 전체 로딩 시간이 줄어든다.

#### 4.2 병렬 커넥션이 항상 빠르지는 않다

- **대역폭이 부족하면**: 각 객체가 제한된 대역폭을 나눠 쓰므로 순서대로 로드하는 것과 다를 바 없다
- **서버 부하 증가**: 100명의 사용자가 각각 100개 커넥션을 열면 서버에 10,000개 커넥션이 생긴다

실제 브라우저는 병렬 커넥션 수를 제한한다(보통 4~6개). 서버는 과도한 커넥션을 닫을 수 있다.

#### 4.3 병렬 커넥션이 "빠르게 느껴진다"

실제로 빠르지 않더라도, 여러 이미지가 동시에 로드되는 시각적 피드백이 주어지므로 사용자는 더 빠르다고 **느낀다**.

### 5. 지속 커넥션

웹 클라이언트는 같은 서버에 여러 요청을 연속으로 보내는 경우가 많다(한 페이지의 이미지들은 대부분 같은 서버에 있다). 이를 **사이트 지역성(site locality)**이라 한다.

HTTP/1.1(및 강화된 HTTP/1.0)은 트랜잭션이 끝난 후에도 **TCP 커넥션을 열린 상태로 유지**하고 이후 요청에 재사용할 수 있다. 이렇게 유지되는 커넥션을 **지속 커넥션**이라 한다.

지속 커넥션 = 커넥션 설정 지연 제거 + 슬로우 스타트 해소 (이미 "튜닝된" 커넥션)

#### 5.1 지속 커넥션 vs 병렬 커넥션

| 항목 | 병렬 커넥션 | 지속 커넥션 |
|------|-------------|-------------|
| 매 트랜잭션마다 설정 비용 | 있음 | 없음 (재사용) |
| TCP 슬로우 스타트 | 매번 발생 | 해소 |
| 동시 커넥션 수 | 제한 있음 | 소수의 지속 커넥션으로 충분 |
| 유휴 커넥션 관리 | 없음 | 주의 필요 (과도한 유휴 커넥션) |

두 기법을 조합하면 효과적이다. **소수의 병렬 커넥션을 지속 상태로 유지**하는 방식을 많이 사용한다.

#### 5.2 HTTP/1.0+ Keep-Alive 커넥션

HTTP/1.0 브라우저·서버는 1996년경부터 비공식적인 **keep-alive 커넥션**을 지원했다. 이는 이후 HTTP/1.1에서 개선되었지만 여전히 흔히 사용된다.

클라이언트가 keep-alive를 요청하려면:
```
GET /index.html HTTP/1.0
Connection: Keep-Alive
Host: www.joes-hardware.com
```

서버가 keep-alive를 지원하면 동일한 헤더로 응답:
```
HTTP/1.0 200 OK
Content-type: text/html
Content-length: 3104
Connection: Keep-Alive
...
```

`Keep-Alive` 헤더 옵션:
```
Connection: Keep-Alive
Keep-Alive: max=5, timeout=120
```
- `timeout`: 서버가 커넥션을 유지하려는 예상 시간(보장 아님)
- `max`: 이 커넥션으로 처리할 HTTP 트랜잭션 최대 수(보장 아님)

**Keep-Alive 제약사항:**
- HTTP/1.0에서는 기본값이 아님. 매 요청마다 `Connection: Keep-Alive` 헤더 전송 필요
- 본문의 길이를 알 수 있어야 함 (정확한 `Content-Length` 또는 청크 인코딩)
- 프락시는 Connection 헤더를 제거하고 전달해야 함
- 클라이언트는 커넥션이 닫힐 경우에 대비해 요청을 재시도할 준비가 필요

#### 5.3 Keep-Alive와 멍청한 프락시 (Dumb Proxies) 문제

**맹목적 릴레이(blind relay)** 역할을 하는 프락시가 `Connection: Keep-Alive` 헤더를 이해하지 못하고 그대로 전달하면 문제가 발생한다:

1. 클라이언트가 프락시에 `Connection: Keep-Alive` 헤더를 포함해 요청
2. 멍청한 프락시가 헤더를 그대로 서버에 전달
3. 서버는 keep-alive를 허용하고 커넥션을 유지하려 함
4. 프락시는 서버가 응답을 보내면 데이터를 클라이언트에 전달한 뒤, **서버가 커넥션을 닫기를 기다림**
5. 서버는 커넥션을 닫지 않고 대기 → **교착 상태(hang)**

해결책: `Proxy-Connection` 헤더 사용 (Netscape 제안). 스마트 프락시는 이를 `Connection: Keep-Alive`로 변환하고, 멍청한 프락시는 그냥 무시한다. 단, 프락시가 2개 이상 있으면 실패할 수 있다.

#### 5.4 HTTP/1.1 지속 커넥션

HTTP/1.1은 keep-alive를 **기본값으로 채택**했다. 별도 설정 없이 커넥션이 지속된다. 커넥션을 닫으려면 명시적으로 `Connection: close` 헤더를 보내야 한다.

```
HTTP/1.1 200 OK
...
Connection: close    ← 이 헤더가 없으면 커넥션이 유지된다
```

**HTTP/1.1 지속 커넥션 규칙:**
- `Connection: close`를 보내면 그 커넥션으로는 더 이상 요청할 수 없다
- 커넥션을 지속하려면 모든 메시지의 본문 길이가 명확해야 한다 (Content-Length 또는 청크 전송 인코딩)
- 서버·클라이언트 모두 언제든지 유휴 커넥션을 닫을 수 있다

---

### 6. 파이프라인 커넥션

HTTP/1.1은 **지속 커넥션 위에서 요청을 파이프라이닝**할 수 있다. 응답을 기다리지 않고 여러 요청을 연속해서 보낼 수 있어 성능이 향상된다.

```
직렬:        [요청1] → [응답1] → [요청2] → [응답2]
파이프라인:  [요청1][요청2][요청3] → [응답1][응답2][응답3]
```

**파이프라인 제약:**
- 지속 커넥션이 설정된 후에만 파이프라인 사용 가능
- 응답은 요청 순서대로 수신된다 (HTTP는 응답 순서를 보장해야 함)
- GET, HEAD 등 **안전한(side-effect 없는) 메서드만** 파이프라이닝해야 한다. POST와 같이 부작용이 있는 요청은 재시도 시 문제가 발생할 수 있음
- 커넥션이 끊어지면 완료되지 않은 요청을 재전송할 준비가 필요

---

### 7. 커넥션 닫기의 미스터리

#### 7.1 "마음대로" 커넥션 닫기

HTTP 클라이언트, 서버, 프락시는 언제든지 지속 커넥션을 닫을 수 있다. 서버는 클라이언트가 다음 요청을 보내는 도중에 커넥션을 닫을 수 있으며, 이는 정상적인 상황이다.

클라이언트는 커넥션이 닫힌 것을 감지하면 **요청을 재시도**해야 한다. 단, 요청에 부작용이 있어 재시도가 위험하다면 예외로 한다.

#### 7.2 Content-Length와 접속 닫힘 감지

서버가 응답 본문을 모두 보내고 커넥션을 닫으면, 클라이언트는 커넥션 닫힘으로 응답의 끝을 인식한다. 그러나 이 경우 클라이언트는 응답이 완전히 전달됐는지, 아니면 네트워크 오류로 잘렸는지 구별할 수 없다. 따라서 지속 커넥션에서는 **Content-Length를 정확히 포함**하거나 청크 인코딩을 사용해야 한다.

#### 7.3 TCP 닫기 동작 (절반 닫기)

보통 TCP 커넥션은 양방향 통신을 지원하는데, 한쪽 방향만 닫는 **절반 닫기(half close)**가 가능하다. `close()` 대신 `shutdown()`을 사용한다.

- **출력 채널 닫기**: 상대방에게 더 이상 데이터를 보내지 않겠다는 신호 (FIN 패킷). 상대방은 아직 데이터를 보낼 수 있음
- **전체 닫기**: `close()`로 입·출력 모두 닫기

HTTP 규격은 클라이언트나 서버가 예기치 않게 커넥션을 닫더라도 상대방이 이에 대응하고 복구할 수 있어야 한다고 규정한다.

### 8. 커넥션 최적화 정리

| 기법 | 핵심 | 장점 | 단점 |
|------|------|------|------|
| **병렬 커넥션** | 동시 여러 TCP 커넥션 | 지연 겹침, 빠르게 느껴짐 | 서버 부하, 대역폭 제한 시 무의미 |
| **지속 커넥션** | TCP 커넥션 재사용 | 핸드셰이크·슬로우 스타트 제거 | 유휴 커넥션 관리 필요 |
| **파이프라인** | 응답 기다리지 않고 요청 연속 전송 | 처리량 향상 | 안전한 메서드만, 순서 보장 필요 |

> 현대 실무: 소수의 병렬 지속 커넥션(예: 브라우저당 4~6개) 사용. HTTP/2는 한 개의 TCP 커넥션 위에서 **멀티플렉싱**으로 이 문제를 근본적으로 해결했다.

---

### 9. 현대에서 TCP 지연이 체감되지 않는 이유

4장에서 다룬 TCP 성능 이슈들이 오늘날 잘 느껴지지 않는 이유는 **스택의 모든 계층이 동시에 개선**됐기 때문이다.

#### 9.1 HTTP/2·3의 근본적 해결

| 문제 | HTTP/1.1 | HTTP/2 | HTTP/3 (QUIC) |
|------|----------|--------|----------------|
| HOL Blocking | 파이프라인도 순서 보장 필요 | 멀티플렉싱으로 스트림 독립 | UDP 기반으로 완전 해소 |
| 핸드셰이크 지연 | 커넥션마다 1-2 RTT | 커넥션 재사용 | 0-RTT 재연결 |
| 슬로우 스타트 | 커넥션 새로 열 때마다 반복 | 하나의 커넥션 장기 유지 | QUIC 자체 흐름 제어 |

특히 **HTTP/3(QUIC)** 은 TCP를 버리고 UDP 위에서 구현해 핸드셰이크 지연 자체를 없앴다.

#### 9.2 TLS 세션 재사용

TLS 1.3의 **0-RTT resumption**: 이전에 접속한 서버라면 핸드셰이크 없이 첫 패킷부터 데이터를 실어 보낼 수 있다.

#### 9.3 CDN과 물리적 거리 단축

- CDN이 사용자와 가까운 엣지 서버에서 TCP 커넥션을 종단 → RTT가 수십~수백 ms에서 **1~5 ms** 수준으로 단축
- RTT 자체가 줄어드니 핸드셰이크·슬로우 스타트의 절대적 지연도 함께 줄어든다

#### 9.4 서버·OS 수준 튜닝

- **RFC 6928**: 초기 혼잡 윈도우를 10 세그먼트로 확대 → 슬로우 스타트 초반 체감 불가
- 브라우저의 DNS Prefetch + TCP Preconnect: 사용자 클릭 전에 미리 커넥션을 열어둠

> TCP의 지연 문제가 "사라진" 것이 아니라, 프로토콜·인프라·브라우저가 그 지연을 **숨기거나 우회**하도록 진화한 것이다. 처음 접속하는 서버나 고레이턴시 환경(위성 인터넷 등)에서는 여전히 핸드셰이크 지연이 뚜렷하게 나타난다.
