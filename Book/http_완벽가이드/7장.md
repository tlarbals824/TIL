# 7장. 캐시

캐시(Cache)는 자주 사용되는 문서의 사본을 가까운 곳에 보관해 두었다가 재사용하는 기술이다. 네트워크 대역폭 절감, 응답 속도 향상, 서버 부하 감소를 목적으로 한다.

---

### 캐시의 이점

| 이점 | 설명 |
|------|------|
| **불필요한 데이터 전송 감소** | 동일 문서 반복 전송 없이 캐시된 사본 제공 |
| **대역폭 병목 해소** | 느린 WAN 링크 대신 빠른 로컬 캐시에서 제공 |
| **갑작스러운 요청 폭주(Flash Crowd) 대응** | 갑자기 많은 요청이 몰려도 캐시가 흡수 |
| **거리로 인한 지연 감소** | 멀리 있는 원 서버 대신 가까운 캐시에서 빠르게 제공 |

---

### 캐시 히트와 미스

- **캐시 히트 (Cache Hit)**: 요청한 문서가 캐시에 있고 신선한 상태일 때. 원 서버 접속 없이 캐시에서 응답.
- **캐시 미스 (Cache Miss)**: 요청한 문서가 캐시에 없거나 오래되어 원 서버에서 새로 받아와야 할 때.
- **재검사 히트 (Revalidation Hit)**: 캐시 문서가 오래됐지만 서버에서 변경되지 않았음이 확인됨 (`304 Not Modified`).
- **재검사 미스**: 캐시 문서가 서버에서 변경됨 → 새 문서 수신 (`200 OK`).

#### 히트율 vs 바이트 히트율

- **히트율 (Hit Rate)**: 전체 요청 중 캐시에서 처리된 요청의 비율. 40%만 되어도 훌륭한 수준.
- **바이트 히트율 (Byte Hit Rate)**: 전송된 총 바이트 중 캐시에서 제공된 바이트의 비율. 대용량 파일이 캐시되면 바이트 히트율이 크게 높아진다.

---

### 캐시 토폴로지

#### 프라이빗 캐시 (Private Cache)

개별 사용자 전용 캐시다. 브라우저 캐시가 대표적이다. 한 명의 사용자에게 맞춤화된 콘텐츠나 자주 방문하는 페이지를 저장한다.

#### 퍼블릭 캐시 (Public Proxy Cache)

여러 사용자가 공유하는 캐시 프록시 서버다. 많은 클라이언트가 동일한 리소스를 요청할 경우 중복 트래픽을 크게 줄일 수 있다.

#### 캐시 계층 구조

캐시도 프록시처럼 계층적으로 구성할 수 있다. 클라이언트에서 요청이 발생하면:
1. 로컬(레벨 1) 캐시 확인 → 미스
2. 상위(레벨 2) 캐시 확인 → 미스
3. 원 서버에 요청

#### 캐시 메시와 콘텐츠 라우팅

캐시 계층이 단순 트리 구조가 아닌 **메시(mesh)** 형태를 이룰 수 있다. 이때 캐시들은 서로 **피어(peer) 관계**를 맺고 ICP(Internet Cache Protocol)나 HTCP를 사용해 서로의 캐시 보유 여부를 물어볼 수 있다.

---

### 캐시 처리 단계 (7단계)

| 단계 | 설명 |
|------|------|
| 1. 요청 수신 | 캐시가 클라이언트 요청 수신 |
| 2. 파싱 | URL, 헤더 파싱 |
| 3. 조회 | 캐시 스토리지에서 로컬 사본 검색 |
| 4. 신선도 검사 | 캐시 사본이 충분히 신선한지 확인 |
| 5. 응답 생성 | 신선한 경우 캐시에서 응답; 오래됐거나 없으면 서버에 포워드 |
| 6. 전송 | 클라이언트에 응답 전송 |
| 7. 로깅 | 캐시 트랜잭션 기록 |

---

### 문서 신선도 유지

#### 문서 만료 (Document Expiration)

서버는 응답 헤더에 **유효기간**을 지정해 캐시가 얼마나 오래 사본을 보관할 수 있는지 알려준다.

| 헤더 | 형식 | 설명 |
|------|------|------|
| `Expires` | 절대 날짜/시간 | 구식. 서버 클록 불일치 문제 있음 |
| `Cache-Control: max-age` | 초 단위 상대 시간 | 권장. 서버 응답 이후 초 수로 지정 |

```
Expires: Fri, 05 Jul 2002, 05:00:00 GMT
Cache-Control: max-age=3600
Cache-Control: s-maxage=3600  # 공유(퍼블릭) 캐시에만 적용
```

#### 서버 재검사 (Server Revalidation)

캐시 사본이 만료된 경우, 캐시는 **조건부 요청**으로 원 서버에 문서가 변경됐는지 확인한다.

##### If-Modified-Since (IMS)

마지막으로 캐시된 날짜 이후 변경 여부를 물어본다. 서버는 변경이 없으면 `304 Not Modified`를, 변경됐으면 `200 OK`와 새 문서를 반환한다.

```
GET /page.html HTTP/1.1
If-Modified-Since: Wed, 01 Jan 2025 00:00:00 GMT
```

##### If-None-Match / ETag

**엔터티 태그(ETag)**는 문서의 버전을 나타내는 임의 식별자다. 날짜 기반 검사보다 정확하다.

```
# 서버 응답
ETag: "v2.6"

# 캐시의 재검사 요청
If-None-Match: "v2.6"

# 여러 버전 동시 전달 가능
If-None-Match: "v2.4", "v2.5", "v2.6"
```

##### 약한 검사기 vs 강한 검사기

| 구분 | 설명 | 형식 |
|------|------|------|
| **강한 검사기 (Strong)** | 콘텐츠가 조금이라도 바뀌면 변경됨 | `ETag: "v2.6"` |
| **약한 검사기 (Weak)** | 의미 있는 변화가 있을 때만 변경됨. 조건부 범위 요청 불가 | `ETag: W/"v2.6"` |

---

### 캐시 가능성 제어

서버는 응답 헤더로 캐시의 동작을 세밀하게 제어할 수 있다.

| 헤더 | 효과 |
|------|------|
| `Cache-Control: no-store` | 캐시 저장 자체를 금지. 포워드 후 즉시 삭제 |
| `Cache-Control: no-cache` | 저장은 가능하지만, 서버 재검사 없이 클라이언트에 제공 불가 |
| `Cache-Control: must-revalidate` | 만료 후 반드시 서버 재검사. 서버 접근 불가 시 `504 Gateway Timeout` |
| `Cache-Control: max-age=<초>` | 응답이 신선한 것으로 간주할 최대 시간(초) |
| `Expires: <날짜>` | 구식 만료 헤더. max-age 권장 |
| `Pragma: no-cache` | HTTP/1.0 하위 호환. `Cache-Control: no-cache`와 동일 |

---

### 휴리스틱 만료 (Heuristic Expiration)

서버가 `Cache-Control: max-age`나 `Expires` 헤더를 전혀 제공하지 않는 경우, 캐시는 자체적으로 신선도를 추정한다.

가장 대표적인 알고리즘은 **LM-Factor 알고리즘**이다.

```perl
$time_since_modify = max(0, $server_Date - $server_Last_Modified);
$server_freshness_limit = int($time_since_modify * $lm_factor);
```

- 문서가 오래전에 마지막으로 수정됐다면 → 안정적인 문서 → 더 오래 캐시
- 최근에 수정됐다면 → 자주 바뀌는 문서 → 짧게 캐시

휴리스틱 만료 기간이 24시간을 넘으면 응답에 `Warning 13` 헤더를 추가해야 한다.

---

### 클라이언트 신선도 제약

클라이언트도 `Cache-Control` **요청 헤더**로 캐시 동작을 조정할 수 있다.

| 요청 헤더 | 효과 |
|----------|------|
| `Cache-Control: max-stale[=<초>]` | 만료된 문서도 허용. 지정 시간 이상 오래된 것은 불가 |
| `Cache-Control: min-fresh=<초>` | 지정 시간 이상 신선한 문서만 수락 |
| `Cache-Control: max-age=<초>` | 지정 시간 이상 캐시된 문서는 불가 |
| `Cache-Control: no-cache` | 재검사 없이 캐시된 문서를 수락하지 않음 |
| `Cache-Control: no-store` | 캐시가 응답의 모든 흔적을 즉시 삭제 |
| `Cache-Control: only-if-cached` | 캐시에 있는 경우에만 응답; 없으면 `504` |

---

### 캐시 설정 (Apache)

Apache는 다양한 모듈로 캐시 헤더를 설정할 수 있다.

```apache
# mod_headers: 특정 파일 타입을 캐시 불가로 설정
<Files *.html>
    Header set Cache-control no-cache
</Files>

# mod_expires: 만료 날짜 자동 생성
ExpiresDefault A3600          # 접근 후 1시간
ExpiresDefault M86400         # 수정 후 24시간
ExpiresDefault "access plus 1 week"
ExpiresByType text/html "modification plus 2 days 6 hours 12 minutes"
```

### 상세 알고리즘

#### 나이(Age) 계산

캐시된 문서의 나이는 문서가 서버를 떠난 이후 경과한 시간이다.

```perl
$apparent_age = max(0, $time_got_response - $Date_header_value);
$corrected_apparent_age = max($apparent_age, $Age_header_value);
$response_delay_estimate = $time_got_response - $time_issued_request;
$age_when_arrived = $corrected_apparent_age + $response_delay_estimate;
$age = $age_when_arrived + ($current_time - $time_got_response);
```

- **클록 스큐(Clock Skew)**: 서버와 캐시의 시계가 다를 수 있어 음수 나이가 될 수 있음 → `max(0, ...)` 처리
- **Age 헤더**: HTTP/1.1이 홉 간 상대적 나이를 누적하기 위해 사용

#### 신선도 수명 계산

문서가 신선한지 판단하려면 `나이 < 신선도 수명`을 확인한다. 신선도 수명은 서버 제약(max-age, Expires, LM-Factor)과 클라이언트 제약을 종합해 결정된다.

### 캐시와 광고

광고주는 광고 노출 횟수당 비용을 지불하므로, 캐시가 원 서버 접근을 줄이면 실제 히트 수를 파악할 수 없는 **광고주의 딜레마**가 생긴다.

- **Cache-Busting**: `no-cache` 헤더 사용, CGI를 통해 광고 서빙, 요청마다 URL 변경 등
- **로그 마이그레이션**: 캐시가 로그를 서버에 주기적으로 전달
- **히트 미터링 (RFC 2227)**: `Meter` 헤더로 캐시 히트 수를 서버에 주기적으로 보고

### 원문 기준으로 보완할 개념

#### 캐시 키와 `Vary`

같은 URL이라도 요청 헤더에 따라 서로 다른 표현(representation)이 반환될 수 있다. 이때 캐시는 URL만으로 사본을 식별하면 안 되고, **캐시 키**에 관련 요청 헤더를 포함해야 한다.

```http
Vary: Accept-Encoding, Accept-Language
```

- `gzip` 압축 여부가 다르면 바이트 수준으로 다른 응답이므로 분리 저장해야 한다.
- 언어 협상(`Accept-Language`), 디바이스별 최적화, 이미지 포맷 변환도 `Vary`를 잘못 쓰면 잘못된 콘텐츠가 섞여 나간다.
- `Vary: *`는 사실상 공유 캐시에 적합하지 않다는 뜻에 가깝다.

#### 신선도 판단 우선순위

캐시는 보통 다음 우선순위로 신선도 수명을 계산한다.

1. `Cache-Control: s-maxage` (공유 캐시 전용)
2. `Cache-Control: max-age`
3. `Expires`
4. 휴리스틱 만료

- 여러 지시자가 섞이면 더 구체적인 지시자가 우선한다.
- 공유 캐시와 개인 캐시가 같은 응답을 다르게 해석할 수 있다는 점이 중요하다.

#### `public`, `private`, `no-transform`, `proxy-revalidate`

| 지시어 | 의미 |
|--------|------|
| `public` | 인증이 걸린 응답이어도 공유 캐시 저장을 명시적으로 허용 |
| `private` | 브라우저 같은 개인 캐시만 저장 가능 |
| `no-transform` | 프록시/CDN이 이미지 압축, 포맷 변환 같은 변경을 하지 못하게 함 |
| `proxy-revalidate` | 공유 캐시는 만료 후 반드시 재검사 |

- 개인화된 HTML, 장바구니, 마이페이지 응답은 보통 `private` 또는 `no-store`가 더 안전하다.
- 인증된 API 응답은 캐시 가능 여부를 더 신중히 설계해야 한다.

#### 비안전 메서드와 캐시 무효화

- `PUT`, `POST`, `PATCH`, `DELETE`처럼 상태를 바꾸는 요청이 성공하면, 관련 URL의 기존 캐시 엔트리는 무효화돼야 한다.
- 캐시가 오래된 사본을 계속 제공하면 읽기 일관성이 깨진다.
- 현대 CDN은 이 문제를 다루기 위해 `PURGE`, 태그 기반 무효화, 경로 기반 무효화 기능을 제공한다.

### 현대 웹 환경의 캐싱 전략

#### 브라우저 캐시와 CDN 캐시는 분리해서 설계한다

| 대상 | 일반적인 전략 |
|------|---------------|
| **해시된 정적 자산(JS/CSS/이미지)** | `Cache-Control: public, max-age=31536000, immutable` |
| **HTML 문서** | 짧은 `max-age` 또는 `no-cache` + 재검사 |
| **개인화 API 응답** | `private`, 필요 시 매우 짧은 TTL |
| **공유 가능한 API 응답** | `s-maxage`, `stale-while-revalidate`, `stale-if-error` 활용 |

#### 현대 CDN에서 자주 쓰는 확장 지시어

```http
Cache-Control: public, max-age=60, stale-while-revalidate=30, stale-if-error=600
```

- `stale-while-revalidate`는 오래된 응답을 잠시 제공하면서 백그라운드에서 새 응답을 받아오게 한다.
- `stale-if-error`는 원 서버 장애 시 마지막 정상 사본을 잠시 제공해 가용성을 높인다.
- 일부 CDN은 `Surrogate-Control`이나 자체 캐시 태그 헤더를 추가로 지원한다.

#### 실무에서 자주 놓치는 캐시 포인트

- 압축 방식이 다르면 `ETag`도 달라질 수 있으므로 gzip/br 응답을 분리해야 한다.
- `Set-Cookie`가 포함된 응답은 공유 캐시에 그대로 두면 위험할 수 있다.
- 프런트엔드 빌드 결과물은 파일명에 해시를 포함해 **URL 자체를 바꾸는 캐시 무효화(cache busting)** 가 가장 안전하다.
- 서비스 워커, 브라우저 메모리 캐시, BFCache는 HTTP 캐시와 별개 계층이므로 함께 고려해야 한다.
- 캐시 키에 쿼리 문자열, 호스트, 헤더를 어떻게 포함할지 일관되게 정하지 않으면 **cache poisoning**이나 잘못된 콘텐츠 혼합이 발생할 수 있다.
- 비공개 콘텐츠를 CDN에 올려야 할 때는 signed URL, signed cookie, 짧은 TTL을 함께 사용해 접근 통제를 보완한다.
