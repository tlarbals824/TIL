# 6장. 프록시

프록시(Proxy)는 클라이언트와 서버 사이에 위치하는 **중개자**다. 클라이언트 대신 서버에 요청을 보내고, 서버의 응답을 받아 클라이언트에 전달한다. 이 장에서는 프록시의 종류, 동작 방식, 트래픽 라우팅, 그리고 주요 헤더를 다룬다.

---

### 프록시와 게이트웨이의 차이

| 구분 | 프록시 | 게이트웨이 |
|------|--------|-----------|
| **프로토콜** | 양쪽 모두 HTTP | 클라이언트 측과 서버 측 프로토콜이 다를 수 있음 |
| **역할** | 동일 프로토콜 중개 | 프로토콜 변환 |
| **예시** | HTTP ↔ HTTP | HTTP ↔ FTP |

---

### 프록시의 종류

#### 프라이빗 프록시 (Private Proxy)

개인 사용자 또는 소수의 클라이언트 전용 프록시다. 주로 클라이언트 컴퓨터에서 실행된다. 브라우저 확장 프로그램이나 개인 보안 소프트웨어로 동작하는 경우가 많다.

#### 퍼블릭 프록시 (Public Proxy)

여러 클라이언트가 공유하는 프록시다. 대규모 캐싱 프록시처럼 많은 사용자가 공동으로 이용할 때 효율이 높아진다.

---

### 프록시의 용도

| 용도 | 설명 |
|------|------|
| **캐싱 프록시** | 응답을 저장해두고 재사용해 성능 향상 |
| **콘텐츠 필터링** | 특정 URL이나 콘텐츠 차단 (학교, 기업 등) |
| **보안 방화벽** | 악성 요청/응답 차단, 바이러스 검사 |
| **익명화 프록시** | 클라이언트의 IP 주소, OS, 브라우저 정보 등 개인정보를 제거 |
| **서로게이트 (역방향 프록시)** | 서버 앞에 위치해 서버인 척하며 요청을 받아 부하 분산, SSL 처리 등을 수행 |
| **트랜스코더** | 콘텐츠 변환: 이미지 압축, 언어 번역, 모바일용 변환 등 |
| **수집기 프록시** | 웹 사이트 분석 등을 위해 요청/응답 데이터 수집 |

---

### 프록시 계층 구조

프록시는 계층적으로 연결될 수 있다. 클라이언트 가장 가까이 있는 프록시를 **인바운드 프록시**, 원 서버에 가장 가까운 프록시를 **아웃바운드 프록시**라 부른다.

```
클라이언트 → 프록시 A → 프록시 B → 원 서버
```

- **부모 프록시 (Parent Proxy)**: 현재 프록시 기준으로 원 서버 방향에 있는 상위 프록시
- **자식 프록시 (Child Proxy)**: 클라이언트 방향에 있는 하위 프록시

프록시 계층에서 각 프록시는 항상 부모 프록시로 요청을 전달하는 것이 아니라, **캐시 히트** 여부나 설정에 따라 직접 원 서버로 요청하기도 한다.

---

### 프록시 트래픽 라우팅 방법

클라이언트 요청이 프록시를 거치도록 하는 방법은 여러 가지가 있다.

#### 클라이언트 수동 설정 (Manual Proxy Configuration)

브라우저의 네트워크 설정에서 프록시 서버 주소와 포트를 직접 입력한다. 가장 단순하고 명확하지만, 설정 변경이 번거롭다.

#### PAC 파일 (Proxy Auto-Configuration)

JavaScript 함수(`FindProxyForURL`)가 담긴 PAC 파일을 통해 URL별로 어떤 프록시를 사용할지 자동으로 결정한다.

```javascript
function FindProxyForURL(url, host) {
    if (isInNet(host, "192.168.0.0", "255.255.0.0")) {
        return "DIRECT";
    }
    return "PROXY proxy.company.com:8080";
}
```

#### WPAD (Web Proxy Auto-Discovery Protocol)

브라우저가 DHCP나 DNS를 통해 PAC 파일의 위치를 자동으로 발견하는 프로토콜이다. 사용자가 별도 설정 없이도 네트워크에 맞는 프록시를 자동으로 사용할 수 있게 해준다.

#### 인터셉트 프록시 (투명 프록시, Transparent Proxy)

클라이언트가 프록시의 존재를 모르는 상태에서, 네트워크 인프라(스위치, 라우터 등)가 HTTP 트래픽을 강제로 프록시로 보낸다. 클라이언트 설정 변경이 필요 없어 대규모 환경에 유리하다.

---

### Via 헤더

**Via 헤더**는 요청/응답이 거쳐 온 프록시의 목록을 기록하는 헤더다. 각 프록시가 자신의 정보를 추가하며, 디버깅과 루프 감지에 활용된다.

```
Via: 1.1 proxy1.example.com, 1.0 proxy2.example.com
```

- 각 항목 형식: `프로토콜버전 호스트명[: 포트] [코멘트]`
- 프록시는 받은 요청/응답에 자신의 정보를 Via 헤더에 **추가**해야 한다.

---

### TRACE 메서드

`TRACE` 메서드는 요청이 어떤 경로로 서버에 도달했는지 루프백 테스트를 수행한다. 서버는 요청 메시지를 그대로 응답 바디에 담아 돌려준다.

```
TRACE /index.html HTTP/1.1
Host: www.example.com
Max-Forwards: 2
```

- `Max-Forwards` 헤더로 거칠 수 있는 중개자의 최대 수를 제한할 수 있다.
- 0이 되면 해당 서버(또는 프록시)가 즉시 응답해야 한다.

---

### 프록시 인증

프록시는 접근 제어 목적으로 클라이언트에게 인증을 요구할 수 있다.

```
# 인증 요구
HTTP/1.1 407 Proxy Authentication Required
Proxy-Authenticate: Basic realm="Company Proxy"

# 클라이언트 재요청
GET /page.html HTTP/1.1
Proxy-Authorization: Basic dXNlcjpwYXNz
```

- **407 Proxy Authentication Required**: 프록시가 인증을 요구할 때 반환하는 상태 코드
- **Proxy-Authenticate**: 프록시가 인증 방식을 알려주는 헤더
- **Proxy-Authorization**: 클라이언트가 인증 정보를 전달하는 헤더

---

### 원문 기준으로 보완할 개념

#### 프록시에 보내는 요청 URI 형식

원 서버에 직접 요청할 때와 프록시에 요청할 때는 요청줄의 URI 형식이 다를 수 있다.

```http
# 원 서버에 직접 요청
GET /index.html HTTP/1.1
Host: www.example.com

# 명시적 프록시에 요청
GET http://www.example.com/index.html HTTP/1.1
Host: www.example.com
```

- 클라이언트는 **명시적 프록시**에 요청할 때 절대 URI(absolute-form)를 보내는 경우가 있다.
- 프록시는 목적지 서버를 판별한 뒤 원 서버에는 보통 origin-form으로 다시 요청한다.
- 이 차이를 제대로 이해하지 못하면 프록시 구현 시 라우팅 오류나 캐시 키 오류가 생긴다.

#### 홉별 헤더(Hop-by-Hop)와 종단간 헤더(End-to-End)

프록시는 단순 전달자가 아니라, 어떤 헤더를 다음 홉으로 넘기면 안 되는지 알아야 한다.

| 구분 | 예시 | 처리 원칙 |
|------|------|-----------|
| **홉별 헤더** | `Connection`, `Keep-Alive`, `Proxy-Authenticate`, `Proxy-Authorization`, `TE`, `Trailer`, `Transfer-Encoding`, `Upgrade` | 현재 연결에서만 의미가 있으므로 다음 홉으로 전달하지 않거나 재작성 |
| **종단간 헤더** | `Host`, `Cache-Control`, `ETag`, `Authorization` 등 | 의미를 보존한 채 전달 |

- 특히 `Connection` 헤더에 나열된 항목은 모두 현재 홉에서 소비돼야 한다.
- 이 규칙을 어기면 keep-alive 오동작, 업그레이드 실패, 캐시 오류가 발생한다.

#### 프록시 루프와 익명화 수준

- `Via`는 중간 프록시 경로를 드러내므로 디버깅과 루프 감지에 중요하다.
- 익명 프록시는 `User-Agent`, `X-Forwarded-For`, 쿠키, 인증 헤더 같은 식별 정보를 얼마나 제거하느냐에 따라 익명성 수준이 달라진다.
- 완전한 익명성을 주장하는 프록시도 TLS 핑거프린트, 요청 패턴, 업스트림 로그로 추적 가능할 수 있다.

---

### 현대 웹 환경에서의 프록시 보충

#### 오늘날 자주 보이는 프록시 형태

| 형태 | 현재의 대표 역할 |
|------|------------------|
| **포워드 프록시** | 기업망의 외부 접근 제어, 감사 로그, 아웃바운드 트래픽 정책 |
| **역방향 프록시** | TLS 종료, 부하 분산, 인증 위임, 캐싱 |
| **CDN 엣지** | 전 세계 캐싱, 이미지 최적화, 봇 차단 |
| **서비스 메시 프록시** | 마이크로서비스 간 mTLS, 재시도, 관측성 |
| **API 게이트웨이** | 인증, 속도 제한, 라우팅, 버전 관리 |

#### 실무에서 많이 쓰는 전달 헤더

```http
X-Forwarded-For: 203.0.113.10, 10.0.0.5
X-Forwarded-Proto: https
X-Forwarded-Host: api.example.com
Forwarded: for=203.0.113.10;proto=https;host=api.example.com
```

- 애플리케이션은 보통 이 헤더를 보고 원래 클라이언트 IP와 스킴을 복원한다.
- 다만 **신뢰할 수 있는 프록시 체인** 뒤에서만 사용해야 하며, 외부 클라이언트가 임의로 보낸 값을 그대로 신뢰하면 안 된다.

#### HTTPS 프록시와 보안 고려사항

- HTTPS 사이트 접속 시 명시적 프록시는 대개 `CONNECT`로 터널을 만든다.
- 일부 기업 환경은 TLS를 중간에서 복호화하는 검사 프록시를 두기도 하지만, 이는 신뢰 저장소와 인증서 배포가 함께 관리되어야 한다.
- 잘못 설정된 공개 프록시는 **오픈 프록시**가 되어 스팸, 우회 접속, 공격 중계에 악용될 수 있다.
- WPAD는 편리하지만, 잘못된 DNS/DHCP 설정이나 악성 네트워크 환경에서 의도치 않은 프록시 설정을 주입받을 위험이 있다.

#### HTTP/2, HTTP/3 시대의 프록시

- HTTP/2 프록시는 스트림 단위 다중화와 헤더 압축을 이해해야 한다.
- HTTP/3 프록시는 QUIC 연결 특성 때문에 TCP 프록시와 다른 운영 모델을 가진다.
- WebSocket, gRPC, SSE처럼 장시간 유지되는 연결은 프록시의 타임아웃과 버퍼링 설정에 큰 영향을 받는다.
- `Proxy-Connection`은 비표준 관행이므로 신규 시스템에서는 의존하지 않는 편이 좋다.
- `TRACE`는 진단에는 유용하지만, 잘못 쓰면 요청 헤더 반사를 악용한 XST(Cross-Site Tracing) 문제와 연결될 수 있어 실서비스에서는 비활성화하는 경우가 많다.
