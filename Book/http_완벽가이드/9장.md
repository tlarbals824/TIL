# 9장. 웹 로봇

웹 로봇(Web Robot)은 사람의 개입 없이 자동으로 여러 웹 트랜잭션을 수행하는 소프트웨어 프로그램이다. 크롤러(crawler), 스파이더(spider), 웜(worm), 봇(bot) 등 다양한 이름으로 불린다.

---

### 웹 로봇의 종류

| 종류 | 설명 |
|------|------|
| **주가 수집 로봇** | 주기적으로 주식 서버에 HTTP GET을 보내 주가 데이터 수집 |
| **웹 센서스 로봇** | 웹 전체를 순회하며 페이지 수, 언어, 미디어 타입 등 통계 수집 |
| **검색 엔진 로봇** | 문서를 수집해 검색 인덱스 데이터베이스 구축 |
| **가격 비교 로봇** | 온라인 쇼핑몰을 크롤링해 상품 가격 비교 데이터베이스 구축 |

---

### 크롤러와 크롤링

**웹 크롤러**는 HTML 페이지를 가져오고 → 링크를 추출하고 → 그 링크의 페이지를 가져오는 과정을 재귀적으로 반복한다.

#### 루트 셋 (Root Set)

크롤러가 출발하는 초기 URL 목록이다. 루트 셋을 잘 선정해야 웹의 대부분을 커버할 수 있다.

- 대형 인기 사이트 (포털, 뉴스 등)
- 새롭게 등록된 페이지
- 링크가 없는 고립된 페이지

#### 링크 추출과 상대 URL 정규화

크롤러는 HTML을 파싱해 `<a href="...">` 링크를 추출한다. 상대 URL은 현재 페이지의 기본 URL을 기준으로 절대 URL로 변환해야 한다.

---

### 루프와 중복 문제 (Loops and Dups)

크롤러가 사이클(A → B → C → A...)에 빠지면 세 가지 문제가 발생한다.

1. **무한 루프**: 같은 페이지를 계속 요청해 크롤러가 멈춤
2. **서버 과부하**: 특정 서버에 폭탄처럼 요청 → 사실상 서비스 거부(DoS) 공격
3. **중복 콘텐츠**: 동일 페이지를 다른 URL로 수백 번 수집 → 검색 인덱스 오염

#### URL 방문 기록 관리 (Trails of Breadcrumbs)

수십억 개 URL을 효율적으로 관리하는 기법들:

| 기법 | 설명 |
|------|------|
| **트리/해시 테이블** | 빠른 URL 조회 |
| **손실 있는 비트맵** | URL을 해시로 변환해 비트 배열에 표시. 메모리 절약 (충돌 가능성 있음) |
| **체크포인트** | 크롤러 충돌에 대비해 방문 URL 목록을 주기적으로 디스크에 저장 |
| **파티셔닝** | 여러 크롤러 인스턴스가 URL 공간을 분할해 병렬 처리 |

#### URL 별칭 (Aliasing)

같은 페이지를 가리키는 다른 URL이 존재할 수 있다.

| 케이스 | 예시 |
|--------|------|
| 기본 포트 생략 | `/bar.html` vs `/:80/bar.html` |
| `%XX` 이스케이프 | `/~fred/` vs `/%7Efred/` |
| 앵커 무시 | `/x.html#early` vs `/x.html#middle` |
| 대소문자 | `/readme.htm` vs `/README.HTM` |
| 기본 페이지 | `/` vs `/index.html` |
| IP vs 호스트명 | `www.foo.com` vs `209.231.87.45` |

**URL 정규화(Canonicalization)**: 포트 추가, `%XX` 디코딩, `#` 태그 제거로 명백한 별칭을 사전 제거한다.

#### 파일시스템 링크 사이클

심볼릭 링크가 상위 디렉터리를 가리키면 크롤러가 무한 깊이로 탐색할 수 있다.

```
/subdir/ → /  (심볼릭 링크)
→ /subdir/subdir/index.html
→ /subdir/subdir/subdir/index.html
→ ...
```

#### 동적 가상 웹 공간

악의적 웹마스터가 게이트웨이 애플리케이션으로 무한히 새로운 URL을 생성해 크롤러를 함정에 빠뜨릴 수 있다.

#### 루프/중복 회피 전략

| 전략 | 설명 |
|------|------|
| **URL 정규화** | 명백한 별칭을 표준 형태로 변환 |
| **너비 우선 탐색** | 한 사이트에 깊이 파고드는 대신 광범위하게 크롤링 → 루프의 영향 최소화 |
| **스로틀링** | 사이트별 요청 속도 제한 |
| **URL 크기 제한** | 1KB 이상 URL은 크롤링 거부 (URL이 점점 길어지는 루프 차단) |
| **URL/사이트 블랙리스트** | 알려진 함정 사이트 목록 유지 |
| **패턴 감지** | 반복되는 URL 구성요소 탐지 (`/subdir/subdir/subdir/...`) |
| **콘텐츠 지문** | 페이지 내용의 체크섬(MD5 등)으로 중복 감지 |
| **인간 모니터링** | 로그와 진단 도구로 이상 감지 |

### 로봇과 HTTP

로봇도 HTTP 클라이언트이므로 HTTP 규격을 따라야 한다.

#### 식별 헤더

| 헤더 | 역할 |
|------|------|
| `User-Agent` | 로봇 이름 |
| `From` | 로봇 관리자 이메일 |
| `Accept` | 수용 가능한 미디어 타입 |
| `Referer` | 현재 URL을 담고 있는 문서의 URL |

```
GET /robots.txt HTTP/1.0
Host: www.joes-hardware.com
User-Agent: Slurp/2.0
From: robot-admin@example.com
```

#### 가상 호스팅 지원

로봇은 반드시 `Host` 헤더를 포함해야 한다. 포함하지 않으면 가상 호스팅 서버에서 다른 사이트의 콘텐츠를 가져올 수 있다.

#### 조건부 요청

대규모 크롤러는 이미 수집한 페이지의 변경 여부만 확인할 때 `If-Modified-Since`나 `If-None-Match`를 사용해 불필요한 데이터 전송을 줄인다.

### 로봇의 나쁜 행동 (Misbehaving Robots)

| 문제 | 설명 |
|------|------|
| **폭주 로봇 (Runaway Robot)** | 루프 또는 버그로 서버에 과부하를 주는 로봇 |
| **오래된 URL (Stale URL)** | 삭제된 URL을 계속 요청해 서버 에러 로그 오염 |
| **긴 URL (Long URL)** | 루프로 인해 비정상적으로 긴 URL 요청 → 서버 성능 저하 또는 크래시 |
| **엿보는 로봇 (Nosy Robot)** | 개인 데이터(비밀번호 파일, 신용카드 정보 등)에 접근해 인덱싱 |
| **동적 게이트웨이 접근** | 비싼 연산이 필요한 게이트웨이 리소스를 무자각하게 요청 |

### 로봇 배제 (Excluding Robots)

1994년 제안된 **Robots Exclusion Standard** (로봇 배제 표준)으로 웹마스터가 로봇의 접근을 제어할 수 있다.

#### robots.txt

웹 서버의 document root에 두는 텍스트 파일이다.

```
# 예: Mary's Antiques 사이트의 robots.txt
User-Agent: Suzy-Spider
Disallow: /dynamic
Allow: /private/suzy-stuff
Disallow: /private

User-Agent: Furniture-Finder
Allow: /dynamic/check-inventory
Disallow: /dynamic
Disallow: /private

User-Agent: *
Disallow: /dynamic
Disallow: /private
```

#### robots.txt 규격

| 항목 | 설명 |
|------|------|
| `User-Agent: <이름>` | 규칙을 적용할 로봇 이름 (대소문자 구분 없는 부분 문자열 매칭) |
| `User-Agent: *` | 모든 로봇에 적용되는 기본 규칙 |
| `Disallow: <경로>` | 접근 금지 경로 (대소문자 구분 있는 접두어 매칭) |
| `Allow: <경로>` | 접근 허용 경로 (Disallow보다 우선) |

**응답 코드별 로봇 행동**:
- `2XX` → 파싱 후 규칙 적용
- `404` → 접근 제한 없음으로 간주
- `401/403` → 사이트 전체 접근 제한
- `503` → 나중에 재시도
- `3XX` → 리다이렉션 따라가기

#### robots.txt 캐싱

로봇은 매 요청마다 robots.txt를 재요청하지 않고 캐시해야 한다. 표준 HTTP 캐시 헤더를 사용해 신선도를 유지한다. 실용적으로 대부분 하루에 한 번 재취득한다.

### HTML 로봇 제어 META 태그

개별 페이지 작성자가 HTML META 태그로 로봇의 동작을 직접 제어할 수 있다.

```html
<head>
    <meta name="robots" content="noindex,nofollow">
    <title>...</title>
</head>
```

| 디렉티브 | 의미 |
|---------|------|
| `INDEX` | 콘텐츠를 인덱싱해도 됨 |
| `NOINDEX` | 이 페이지를 인덱싱하지 말 것 |
| `FOLLOW` | 링크를 크롤해도 됨 |
| `NOFOLLOW` | 이 페이지의 링크를 크롤하지 말 것 |
| `NOARCHIVE` | 로컬 사본(캐시)을 보관하지 말 것 |
| `ALL` | `INDEX, FOLLOW`와 동일 |
| `NONE` | `NOINDEX, NOFOLLOW`와 동일 |

### 로봇 에티켓

Martijn Koster가 1993년 작성한 로봇 작성자 가이드라인을 기반으로 정리한 7가지 범주다.

| 범주 | 핵심 지침 |
|------|----------|
| **(1) 신원 공개** | User-Agent, From, 연락처 명확히 제공 |
| **(2) 운영** | 경계하고, 준비하고, 모니터링하고, 지속적으로 개선 |
| **(3) 자기 제한** | URL 필터링, 동적 URL 회피, robots.txt 준수, **스로틀링** |
| **(4) 루프/중복 허용** | 모든 상태 코드 처리, URL 정규화, 공격적인 사이클 회피 |
| **(5) 확장성** | 공간/대역폭/시간 계산, 분할 정복 전략 수립 |
| **(6) 신뢰성** | 철저한 사전 테스트, 체크포인트 기능, 장애 대응 |
| **(7) 공개 관계** | 불만에 빠르게 대응, 블랙리스트에서 즉시 제거 |

### 검색 엔진

검색 엔진은 웹 로봇의 가장 중요한 응용이다.

#### 아키텍처

```
크롤러/인덱서 → 전문 검색 인덱스 DB → 웹 검색 게이트웨이 → 사용자
```

- **크롤러**: 웹을 순회해 문서 수집
- **전문 검색 인덱스 (Full-Text Index)**: 단어 → 문서 목록 매핑 데이터베이스
- **쿼리 게이트웨이**: 사용자 검색어를 인덱스에서 찾아 결과 페이지 생성

#### 관련도 순위 (Relevancy Ranking)

같은 단어가 여러 문서에 있을 때, 링크 수, 단어 빈도, 위치 등을 종합해 순위를 결정한다. 이 알고리즘은 검색 엔진의 핵심 경쟁력이다.

#### 스푸핑 (Spoofing)

상위 노출을 위해 대량의 키워드를 숨기거나, 검색 엔진을 속이는 가짜 페이지를 생성하는 행위다. 검색 엔진은 이를 탐지하기 위해 지속적으로 알고리즘을 개선한다.

### 원문 기준으로 보완할 개념

#### 크롤링 프런티어(Frontier)와 방문 스케줄링

대형 크롤러는 단순히 "발견한 링크를 바로 방문"하지 않고, **방문 대기열(frontier)** 을 관리한다.

| 고려 요소 | 이유 |
|-----------|------|
| **호스트별 요청 간격** | 특정 사이트 과부하 방지 |
| **우선순위** | 중요 페이지, 새 페이지, 자주 바뀌는 페이지를 먼저 방문 |
| **재방문 주기** | 뉴스와 정적 문서는 갱신 주기가 다름 |
| **실패 재시도 정책** | 일시적 오류와 영구 오류를 구분해야 함 |

- 대규모 크롤러는 링크 발견, 중복 제거, 방문 예약, 콘텐츠 파싱을 별도 단계로 분리한다.
- 이 구조가 없으면 빠른 수집과 예의 있는 접근을 동시에 달성하기 어렵다.

#### robots.txt는 접근 통제 장치가 아니다

- robots.txt는 **협조적인 로봇에게만** 의미가 있다.
- 민감한 경로를 robots.txt에 적는 것은 "숨기는 것"이 아니라 오히려 위치를 알려주는 효과가 날 수 있다.
- 진짜 접근 제어는 인증, 권한 검증, 네트워크 제한으로 해야 한다.

#### 페이지 단위 제어와 수집 단위 제어의 차이

- robots.txt는 "가져오지 말라"는 정책에 가깝다.
- HTML의 robots 메타 태그는 페이지를 **가져온 뒤** 인덱싱 여부를 제어한다.
- 따라서 페이지 자체를 요청하지 못하게 하려면 메타 태그만으로는 충분하지 않다.
- 같은 사이트라도 `User-Agent`별로 다른 robots 정책이 적용될 수 있으므로, 로봇은 자신에게 맞는 규칙 블록을 정확히 찾아야 한다.

### 현대 웹 환경에서의 크롤링 보충

#### 오늘날 추가로 자주 쓰는 크롤러 제어 수단

| 수단 | 설명 |
|------|------|
| **`X-Robots-Tag` 응답 헤더** | HTML이 아닌 PDF, 이미지, API 응답에도 인덱싱 정책 부여 가능 |
| **XML Sitemap** | 크롤러에게 중요한 URL과 갱신 시점을 알려줌 |
| **`link rel="canonical"`** | 중복 페이지의 대표 URL을 알려줌 |
| **`nofollow`, `ugc`, `sponsored`** | 링크 의미를 검색엔진에 추가로 전달 |

```http
X-Robots-Tag: noindex, noarchive
```

- robots.txt는 현재 `RFC 9309`로 정리되어 있으며, 보통 **가장 구체적인 경로 규칙이 우선** 적용된다고 이해하면 된다.
- 크롤러는 robots.txt도 무한정 재요청하지 않고 적절히 캐시해야 한다.

#### 현대 검색엔진이 보는 페이지는 더 복잡하다

- 많은 사이트가 자바스크립트 렌더링을 사용하므로, 검색엔진은 단순 HTML 파서만이 아니라 렌더링 파이프라인도 운영한다.
- 그 결과 서버는 HTML 초기 응답, JS 번들, API 응답, 리소스 로딩 경로 전체를 고려해 크롤링 친화성을 설계해야 한다.
- `robots.txt`로 JS/CSS를 막아버리면 렌더링 품질이 떨어져 검색 노출에 악영향을 줄 수 있다.

#### HTTP 관점에서 중요해진 운영 포인트

- `429 Too Many Requests`와 `Retry-After`는 예의 있게 크롤러 속도를 제어하는 현대적 수단이다.
- `ETag`, `Last-Modified`를 잘 제공하면 재크롤링 비용을 크게 줄일 수 있다.
- 대규모 크롤러는 HTTP/2 연결 재사용, 압축, 조건부 요청으로 서버 부담을 줄인다.
- 상태 코드별 재시도 정책을 분리해야 한다. 예를 들어 `404`는 빠르게 포기하고, `503`은 지연 후 재시도하는 편이 합리적이다.

#### 2020년대 이후의 추가 맥락

- 검색엔진 외에도 요약, 추천, 학습 데이터를 모으는 다양한 자동화 크롤러가 존재한다.
- 따라서 사이트 운영자는 "인덱싱 허용 여부"와 "수집 허용 여부"를 구분해서 정책을 설계할 필요가 있다.
- 크롤링 예산(crawl budget)은 대형 사이트에서 여전히 중요한 운영 지표다.
