# 제3장: HTTP 메시지

HTTP 메시지의 구조(시작줄·헤더·본문), 메서드, 상태 코드, 헤더 분류를 다룬다.

---

### 1. 메시지의 흐름

HTTP 메시지는 클라이언트, 서버, 프락시 사이를 흐르는 **데이터 블록**이다. 앞부분에는 메타 정보를 담은 텍스트가 오고, 그 뒤에 선택적으로 데이터가 온다.

#### 1.1 인바운드/아웃바운드

HTTP는 트랜잭션 방향을 설명하기 위해 **인바운드(inbound)**와 **아웃바운드(outbound)**를 사용한다.

- **인바운드**: 메시지가 원 서버(origin server)를 향해 이동하는 방향
- **아웃바운드**: 처리가 끝난 뒤 사용자 에이전트 쪽으로 돌아오는 방향

#### 1.2 업스트림/다운스트림

HTTP 메시지는 강물처럼 흐른다. **모든 메시지(요청·응답 모두)는 다운스트림으로 흐른다.** 메시지의 발송자는 수신자의 업스트림에 있다.

```
클라이언트 → Proxy1 → Proxy2 → Proxy3 → 서버   (요청: 다운스트림)
클라이언트 ← Proxy1 ← Proxy2 ← Proxy3 ← 서버   (응답: 다운스트림)
```

> "upstream/downstream"은 발신자와 수신자의 상대적 관계를 나타낸다. 메시지가 서버를 향하는지 클라이언트를 향하는지는 알 수 없다 — 둘 다 다운스트림이다.

---

### 2. 메시지의 구성

HTTP 메시지는 세 부분으로 이루어진 **단순한 형식의 데이터 블록**이다.

```
시작줄 (Start Line)   ← 요청: 메서드 + URL + 버전 / 응답: 버전 + 상태코드 + 사유구절
헤더 (Headers)        ← 이름: 값 쌍. 빈 줄(CRLF)로 종료
                       ← (빈 줄)
본문 (Entity Body)    ← 선택 사항. 텍스트 또는 이진 데이터
```

시작줄과 헤더는 줄 단위 **ASCII 텍스트**이며, 각 줄은 CRLF(캐리지 리턴 + 라인 피드)로 끝난다. 견고한 구현은 단독 LF도 허용해야 한다.

#### 2.1 메시지 문법

**요청 메시지 형식:**
```
<메서드> <요청-URL> <버전>
<헤더들>

<엔티티-본문>
```

**응답 메시지 형식:**
```
<버전> <상태코드> <사유구절>
<헤더들>

<엔티티-본문>
```

**예시:**
```
(a) 요청 메시지                      (b) 응답 메시지
GET /test/hi-there.txt HTTP/1.1     HTTP/1.0 200 OK
Accept: text/*                      Content-type: text/plain
Host: www.joes-hardware.com         Content-length: 19

                                    Hi! I'm a message!
```

#### 2.2 각 구성 요소

| 요소 | 설명 |
|------|------|
| **메서드** | 클라이언트가 서버에 요청하는 동작 (`GET`, `POST` 등) |
| **요청-URL** | 요청 대상 리소스의 완전한 URL 또는 경로 |
| **버전** | `HTTP/<major>.<minor>` 형식. 예: `HTTP/1.1` |
| **상태코드** | 요청 처리 결과를 나타내는 세 자리 숫자 |
| **사유구절** | 상태 코드의 사람이 읽을 수 있는 설명 (처리에는 무관) |
| **헤더** | `이름: 값` 쌍의 목록. 빈 줄로 종료 |
| **엔티티 본문** | 임의의 데이터 블록 (없을 수도 있음) |

> 버전 번호는 분수가 아니다. `HTTP/2.22`는 `HTTP/2.3`보다 높다 (22 > 3).

#### 2.3 HTTP/0.9 메시지

HTTP/0.9는 메서드와 요청-URL만으로 구성되었고, 응답에는 엔티티만 포함되었다. 버전 정보, 상태 코드, 사유구절, 헤더가 없었다. 매우 단순했으나 유연성이 없었다.

---

### 3. 메서드

HTTP/1.1 준수를 위해 서버는 **GET과 HEAD만 구현하면 된다**. 그 외 메서드는 서버·상황에 따라 제한될 수 있다.

#### 3.1 안전한 메서드 (Safe Methods)

**GET**과 **HEAD**는 서버에서 아무 작용도 일어나지 않는 **안전한 메서드**다. 서버 측에서 어떤 행동도 유발하지 않아야 한다는 의미이며, HTTP 응용 개발자가 사용자에게 안전하지 않은 메서드를 사용할 때 경고할 수 있게 해준다.

#### 3.2 주요 메서드 상세

| 메서드 | 요청 본문 | 설명 |
|--------|-----------|------|
| **GET** | 없음 | 서버에서 문서를 가져온다. 가장 흔한 메서드. HTTP/1.1 필수 |
| **HEAD** | 없음 | GET과 동일하나 **헤더만** 반환, 본문 없음. 리소스 타입 확인, 존재 여부 확인, 수정 여부 확인에 유용. HTTP/1.1 필수 |
| **PUT** | 있음 | 서버의 요청 URL에 데이터를 저장한다. URL이 이미 존재하면 교체, 없으면 생성. 인증 필요 |
| **POST** | 있음 | 서버의 게이트웨이 프로그램에 데이터를 전송한다. HTML 폼 전송에 주로 사용 |
| **TRACE** | 없음 | 요청이 서버에 도달할 때까지 프락시·게이트웨이를 거치며 어떻게 변경되는지 확인하는 루프백 진단. 진단 목적 |
| **OPTIONS** | 없음 | 서버 또는 특정 리소스에 대해 어떤 메서드가 지원되는지 확인 |
| **DELETE** | 없음 | 서버에서 요청 URL 리소스를 삭제 |

> `POST`는 데이터를 서버로 전송한다. `PUT`은 데이터를 리소스(파일)로 서버에 저장한다.

> `TRACE`는 메서드 구분 기능이 없어 (프락시가 GET과 TRACE를 다르게 처리할 수 있음), 제한적으로 사용된다.

**확장 메서드(Extension Methods):** HTTP는 확장 가능하도록 설계되었다. 서버는 자체적인 추가 메서드를 정의할 수 있다 (예: WebDAV의 `LOCK`, `MKCOL`, `COPY`).

### 4. 상태 코드

상태 코드는 응답 메시지의 시작줄에 포함된다. 숫자 코드는 프로그램 처리에, 사유구절은 사람이 읽기 위한 용도이다.

#### 4.1 상태 코드 분류

| 전체 범위 | 정의된 범위 | 분류 |
|-----------|-------------|------|
| 100-199 | 100-101 | 정보성 (Informational) |
| 200-299 | 200-206 | 성공 (Successful) |
| 300-399 | 300-305 | 리다이렉션 (Redirection) |
| 400-499 | 400-415 | 클라이언트 오류 (Client Error) |
| 500-599 | 500-505 | 서버 오류 (Server Error) |

> 정의되지 않은 코드(예: 515)를 받으면 해당 클래스의 일반 응답으로 처리한다 (515 → 5XX 서버 오류).

#### 4.2 주요 100번대 — 정보성

| 코드 | 사유구절 | 의미 |
|------|----------|------|
| `100` | Continue | 요청의 시작 부분을 받았으며 클라이언트는 계속해야 한다 |
| `101` | Switching Protocols | 클라이언트가 Upgrade 헤더로 요청한 프로토콜로 전환 |

#### 4.3 주요 200번대 — 성공

| 코드 | 사유구절 | 의미 |
|------|----------|------|
| `200` | OK | 요청 성공. 응답 본문에 요청한 데이터가 있음 |
| `201` | Created | PUT 요청 성공 시. 생성된 리소스의 URL을 Location 헤더로 반환 |
| `202` | Accepted | 요청은 받았으나 아직 처리 중 |
| `204` | No Content | 응답 본문 없음. 브라우저를 새 문서로 이동시킬 필요가 없을 때 |
| `206` | Partial Content | 부분 요청(Range) 성공 |

#### 4.4 주요 300번대 — 리다이렉션

| 코드 | 사유구절 | 의미 |
|------|----------|------|
| `301` | Moved Permanently | 리소스가 영구적으로 이동됨. Location 헤더에 새 URL |
| `302` | Found | 임시로 다른 URL에 있음. 원래 URL을 계속 사용할 것 |
| `303` | See Other | POST 이후 다른 URL(GET)로 리다이렉트 |
| `304` | Not Modified | 조건부 요청 시 리소스가 변경되지 않음. 본문 없음 |
| `305` | Use Proxy | 반드시 프락시를 통해 접근해야 함 |

#### 4.5 주요 400번대 — 클라이언트 오류

| 코드 | 사유구절 | 의미 |
|------|----------|------|
| `400` | Bad Request | 잘못된 요청 (문법 오류 등) |
| `401` | Unauthorized | 인증이 필요함 |
| `403` | Forbidden | 서버가 요청을 거부 (권한 없음) |
| `404` | Not Found | 서버에 해당 URL 리소스가 없음 |
| `405` | Method Not Allowed | 해당 메서드를 지원하지 않음 |
| `408` | Request Timeout | 요청 시간 초과 |
| `409` | Conflict | 리소스 충돌 (예: 이미 존재) |
| `410` | Gone | 리소스가 영구 삭제됨 (301과 달리 새 URL 없음) |
| `415` | Unsupported Media Type | 지원하지 않는 미디어 타입 |

#### 4.6 주요 500번대 — 서버 오류

| 코드 | 사유구절 | 의미 |
|------|----------|------|
| `500` | Internal Server Error | 서버 내부 오류 |
| `501` | Not Implemented | 서버가 요청한 메서드를 지원하지 않음 |
| `502` | Bad Gateway | 게이트웨이/프락시가 업스트림으로부터 잘못된 응답 수신 |
| `503` | Service Unavailable | 서버가 현재 요청을 처리할 수 없음 (과부하·유지보수) |
| `504` | Gateway Timeout | 게이트웨이가 업스트림 응답을 기다리다 시간 초과 |
| `505` | HTTP Version Not Supported | 지원하지 않는 HTTP 버전 |

---

### 5. 헤더

헤더는 시작줄 다음에 오는 `이름: 값` 쌍의 목록이다. HTTP/1.1은 특정 헤더의 존재를 요구한다(예: `Host`). 긴 헤더 값은 다음 줄에 공백·탭으로 시작하여 연결할 수 있다(**연속줄**).

#### 5.1 헤더 분류

| 분류 | 설명 |
|------|------|
| **일반 헤더** | 요청·응답 메시지 양쪽에 모두 사용 가능 |
| **요청 헤더** | 요청 메시지에서만 의미 있음. 클라이언트 정보·선호도 제공 |
| **응답 헤더** | 응답 메시지에서만 의미 있음. 서버 정보·지시사항 제공 |
| **엔티티 헤더** | 본문의 크기·종류 등 설명. 요청·응답 모두 사용 |
| **확장 헤더** | 명세에 정의되지 않은 비표준 헤더. HTTP 프로그램은 모르더라도 허용·전달해야 함 |

#### 5.2 일반 헤더

| 헤더 | 설명 |
|------|------|
| `Connection` | 커넥션에 관한 옵션 지정 (hop-by-hop) |
| `Date` | 메시지 생성 일시 |
| `MIME-Version` | 발신자가 사용하는 MIME 버전 |
| `Transfer-Encoding` | 메시지 전송 시 적용된 인코딩 방식 |
| `Upgrade` | 발신자가 업그레이드하고자 하는 새 버전/프로토콜 |
| `Via` | 메시지가 거쳐온 중개자(프락시, 게이트웨이) |
| `Cache-Control` | 캐싱 지시사항 |
| `Pragma` | 캐싱 지시사항 전달 (레거시; `Cache-Control`로 대체됨) |

#### 5.3 요청 헤더

**정보성 헤더:**

| 헤더 | 설명 |
|------|------|
| `From` | 클라이언트 사용자의 이메일 주소 |
| `Host` | 요청 대상 서버의 호스트명과 포트 (HTTP/1.1 필수) |
| `Referer` | 현재 요청 URI를 포함하는 문서의 URL |
| `User-Agent` | 요청을 보내는 애플리케이션 이름 |

**Accept 헤더 (클라이언트 선호도):**

| 헤더 | 설명 |
|------|------|
| `Accept` | 수신 가능한 미디어 타입 |
| `Accept-Charset` | 수신 가능한 문자 집합 |
| `Accept-Encoding` | 수신 가능한 인코딩 방식 |
| `Accept-Language` | 수신 가능한 언어 |

**조건부 요청 헤더:**

| 헤더 | 설명 |
|------|------|
| `If-Modified-Since` | 지정 날짜 이후 변경된 경우에만 전송 |
| `If-None-Match` | ETag가 다른 경우에만 전송 |
| `If-Match` | ETag가 일치하는 경우에만 전송 |
| `If-Range` | 범위 요청을 조건부로 적용 |
| `Range` | 리소스의 특정 범위(바이트) 요청 |

**보안 헤더:**

| 헤더 | 설명 |
|------|------|
| `Authorization` | 서버 인증을 위한 클라이언트 자격증명 |
| `Cookie` | 서버로 토큰 전달 (보안 헤더는 아니지만 보안 함의 있음) |

**프락시 요청 헤더:**

| 헤더 | 설명 |
|------|------|
| `Max-Forwards` | TRACE 메서드와 함께 사용; 프락시·게이트웨이를 최대 몇 번 거칠 수 있는지 |
| `Proxy-Authorization` | 프락시 인증 자격증명 |
| `Proxy-Connection` | 프락시와의 커넥션 옵션 |

#### 5.4 응답 헤더

**정보성 헤더:**

| 헤더 | 설명 |
|------|------|
| `Age` | 응답이 얼마나 오래된 것인지 (프락시 캐시 통과 시) |
| `Server` | 서버 소프트웨어의 이름과 버전 |
| `Retry-After` | 리소스 사용 불가 시 언제 다시 요청할지 |
| `Warning` | 사유구절보다 상세한 경고 메시지 |

**보안 헤더:**

| 헤더 | 설명 |
|------|------|
| `Proxy-Authenticate` | 프락시에서 클라이언트에 보내는 인증 요구 |
| `Set-Cookie` | 클라이언트에 토큰 설정 (보안 함의 있음) |
| `WWW-Authenticate` | 서버에서 클라이언트에 보내는 인증 요구 |

**협상 헤더:**

| 헤더 | 설명 |
|------|------|
| `Accept-Ranges` | 서버가 수락할 범위 요청의 타입 |
| `Vary` | 응답을 결정하는 데 사용된 헤더들의 목록 |

#### 5.5 엔티티 헤더

**정보성 헤더:**

| 헤더 | 설명 |
|------|------|
| `Allow` | 이 리소스에 허용되는 요청 메서드 목록 |
| `Location` | 리소스의 실제 위치 URL (리다이렉션에 사용) |

**콘텐츠 헤더:**

| 헤더 | 설명 |
|------|------|
| `Content-Type` | 본문의 객체 타입 (MIME 타입) |
| `Content-Length` | 본문의 길이 또는 크기 (바이트) |
| `Content-Encoding` | 본문에 적용된 인코딩 |
| `Content-Language` | 본문을 이해하기 위한 자연어 |
| `Content-Location` | 리소스의 실제 위치 |
| `Content-MD5` | 본문의 MD5 체크섬 |
| `Content-Range` | 전체 리소스에서 이 엔티티가 차지하는 바이트 범위 |

**캐싱 헤더:**

| 헤더 | 설명 |
|------|------|
| `ETag` | 엔티티 태그. 리소스의 특정 버전 식별자 |
| `Expires` | 이 엔티티가 유효하지 않게 되는 날짜/시간 |
| `Last-Modified` | 이 엔티티가 마지막으로 변경된 날짜/시간 |
