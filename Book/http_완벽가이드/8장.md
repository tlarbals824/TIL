# 8장. 통합 포인트: 게이트웨이, 터널, 릴레이

이 장은 HTTP가 다른 프로토콜이나 애플리케이션과 어떻게 연결되는지를 다룬다. 게이트웨이는 프로토콜 변환, 터널은 비HTTP 트래픽을 HTTP 위에 실어나르는 방법, 릴레이는 단순한 HTTP 중계자를 설명한다.

---

### 게이트웨이

**게이트웨이(Gateway)**는 서로 다른 프로토콜과 애플리케이션 사이에서 변환기 역할을 한다. HTTP 클라이언트가 FTP, 이메일 등 비HTTP 프로토콜의 리소스에 접근할 수 있도록 해준다.

#### 클라이언트 측 게이트웨이와 서버 측 게이트웨이

게이트웨이는 클라이언트 측과 서버 측 프로토콜 표기법으로 구분한다.

```
<클라이언트 프로토콜> / <서버 프로토콜>
```

| 구분 | 설명 |
|------|------|
| **서버 측 게이트웨이 (HTTP/\*)** | 클라이언트와 HTTP로 통신, 서버 측은 다른 프로토콜 |
| **클라이언트 측 게이트웨이 (\*/HTTP)** | 클라이언트는 다른 프로토콜, 서버와는 HTTP로 통신 |

---

### 프로토콜 게이트웨이

#### HTTP/FTP 서버 측 게이트웨이

가장 흔한 게이트웨이 중 하나다. 클라이언트가 `ftp://...` URL을 HTTP 요청으로 보내면, 게이트웨이가 FTP 프로토콜로 실제 FTP 서버에 접속해 파일을 가져와 HTTP로 응답한다.

```
GET ftp://ftp.irs.gov/pub/00-index.txt HTTP/1.0
Host: ftp.irs.gov
User-agent: SuperBrowser 4.2
```

게이트웨이는 FTP 서버에 접속해 다음 과정을 수행한다:
1. `USER` / `PASS` 명령으로 로그인
2. `CWD` 명령으로 디렉터리 이동
3. `TYPE A`로 ASCII 전송 모드 설정
4. `MDTM`으로 최종 수정 시간 확인
5. `PASV`로 패시브 데이터 채널 설정
6. `RETR`로 파일 요청
7. 데이터 수신 후 HTTP 응답으로 포장

#### HTTP/HTTPS 서버 측 보안 게이트웨이

인트라넷 환경에서 조직의 모든 웹 요청을 암호화하고자 할 때 사용한다. 클라이언트는 일반 HTTP 요청을 보내고, 게이트웨이가 SSL로 암호화해 보안 서버에 전달한다.

#### HTTPS/HTTP 클라이언트 측 보안 가속기 게이트웨이

서버 앞에 위치해 HTTPS 요청을 받아 복호화하고, 일반 HTTP로 백엔드 서버에 전달한다. SSL 처리 전용 하드웨어로 서버의 부하를 줄인다. 단, 게이트웨이~서버 구간이 암호화되지 않으므로 내부 네트워크 보안이 중요하다.

---

### 리소스 게이트웨이와 애플리케이션 서버

**애플리케이션 서버**는 가장 일반적인 게이트웨이 형태다. HTTP 클라이언트와 통신하면서, 내부적으로 API를 통해 다양한 애플리케이션(데이터베이스, 전자상거래 엔진, 카메라 등)과 연결된다.

#### CGI (Common Gateway Interface)

최초의, 그리고 여전히 가장 널리 쓰이는 서버 확장 인터페이스다. 서버가 특정 URL에 대한 요청을 받으면 외부 프로그램을 실행하고, 그 출력을 HTTP 응답으로 전달한다.

- 어떤 언어로든 구현 가능 (Perl, Python, C, 셸 스크립트)
- 단점: 요청마다 새 프로세스를 생성하는 오버헤드
- **Fast CGI**: 프로세스를 데몬으로 유지해 프로세스 생성 비용 제거

#### 서버 확장 API

CGI보다 더 직접적으로 서버와 통합하는 방식이다. Apache의 모듈 API, Microsoft의 ISAPI 등이 있다. 개발자가 직접 서버 코드에 모듈을 삽입하거나 교체할 수 있다.

#### 웹 서비스 (Web Services)

HTTP를 기반으로 애플리케이션 간 정보를 교환하는 표준이다.
- **SOAP (Simple Object Access Protocol)**: XML 형식으로 데이터를 HTTP 메시지에 담아 전달
- 서로 다른 플랫폼의 애플리케이션이 HTTP를 공통 언어로 통신할 수 있게 해준다

---

### 터널

**터널(Tunnel)**은 HTTP 연결을 통해 비HTTP 트래픽을 전달하는 방법이다. 주로 방화벽을 통과해 암호화된 SSL 트래픽을 전달하기 위해 사용된다.

#### CONNECT 메서드로 터널 개설

웹 터널은 HTTP의 `CONNECT` 메서드로 개설한다. CONNECT는 HTTP/1.1의 핵심 스펙은 아니지만 사실상 표준으로 구현된다.

```
# 클라이언트 → 게이트웨이
CONNECT orders.joes-hardware.com:443 HTTP/1.0
User-agent: SuperBrowser 4.2
```

1. 클라이언트가 게이트웨이에 CONNECT 요청
2. 게이트웨이가 목적지 서버(포트 443)에 TCP 연결 개설
3. 게이트웨이가 클라이언트에 `200 Connection Established` 응답

```
HTTP/1.0 200 Connection established
Proxy-agent: Netscape-Proxy/1.1
```

4. 이후 클라이언트가 보내는 모든 데이터는 게이트웨이를 통해 그대로 릴레이된다 (내용 검사 없음)

#### CONNECT 요청/응답 형식

- **요청 URI**: 일반 URI 대신 `호스트명:포트` 형식
- **응답**: Content-Type 헤더 불필요 (이후 연결이 바이트 스트림 릴레이가 되므로)

#### SSL 터널링

SSL 트래픽은 암호화되어 있어 일반 프록시가 내용을 검사할 수 없다. 터널은 이런 SSL 트래픽을 포트 80 HTTP 연결을 통해 방화벽 너머로 전달할 수 있게 해준다.

| 방식 | 설명 |
|------|------|
| **HTTPS/HTTP 게이트웨이** | 프록시가 SSL 세션을 시작해 복호화 후 HTTP로 전달. X.509 인증 불가 |
| **SSL 터널** | 클라이언트가 직접 SSL 세션 수립. 프록시는 암호화된 데이터만 릴레이. 인증 가능 |

#### 터널 인증

터널을 사용하기 전에 프록시 인증을 요구할 수 있다.

```
# 게이트웨이의 인증 요구
HTTP/1.0 407 Proxy Authentication Required
Proxy-authenticate: Basic realm="wormhole"

# 클라이언트 재요청
CONNECT orders.joes-hardware.com:443 HTTP/1.0
Proxy-authorization: Basic YnJpYW46...
```

#### 터널 보안 고려사항

터널 게이트웨이는 전달하는 프로토콜이 실제로 의도된 것인지 확인할 수 없다. 악의적 사용자가 SSL 터널을 게임 트래픽이나 이메일 발송에 악용할 수 있다. 따라서 **잘 알려진 포트(443 등)만** 터널을 허용하도록 제한해야 한다.

### 릴레이

**릴레이(Relay)**는 HTTP 명세를 완전히 따르지 않는 단순한 HTTP 프록시다. HTTP를 파싱하지 않고 연결을 맺은 후 바이트를 그대로 전달한다.

- 구현이 단순하고 가볍다
- 간단한 필터링, 진단, 콘텐츠 변환에 사용

#### 릴레이의 문제점: Keep-Alive와 Connection 헤더

릴레이가 `Connection: Keep-Alive` 헤더를 제대로 처리하지 않으면 연결이 영구히 대기 상태(hang)에 빠질 수 있다.

1. 클라이언트가 `Connection: Keep-Alive` 헤더를 포함해 릴레이에 요청
2. 릴레이는 헤더를 이해 못 해 그대로 서버에 전달
3. 서버는 릴레이가 keep-alive를 지원한다고 오해하고 연결 유지
4. 릴레이는 서버가 연결을 닫기를 기다리며 대기
5. 클라이언트는 다음 요청을 보내지만 릴레이는 처리하지 못함 → **hang**

> 릴레이를 광범위하게 배포할 때는 HTTP 명세를 완전히 따르는 프록시 서버 사용을 권장한다.

---

### 원문 기준으로 보완할 개념

#### 게이트웨이, 터널, 릴레이의 핵심 차이

| 구성 요소 | HTTP 의미를 이해하는가 | 주된 역할 |
|-----------|------------------------|-----------|
| **게이트웨이** | 이해함 | 프로토콜 변환, 애플리케이션 연동 |
| **터널** | 초기 설정 이후에는 거의 이해하지 않음 | 바이트 스트림 전달 |
| **릴레이** | 매우 제한적으로만 이해 | 단순 중계 |

- 게이트웨이는 요청 의미를 해석해 다른 프로토콜이나 내부 API 호출로 바꾼다.
- 터널은 `CONNECT`가 성립한 뒤 사실상 TCP 파이프처럼 동작한다.
- 릴레이는 구현이 단순하지만 HTTP 세부 규칙을 놓치기 쉽다.

#### 터널이 열린 뒤의 동작

- `200 Connection Established`가 반환된 이후에는 HTTP 메시지가 아니라 **양방향 바이트 스트림**이 흐른다.
- 중간 장비는 바이트를 그대로 전달하므로, 그 안에서 오가는 TLS 핸드셰이크나 애플리케이션 프로토콜을 이해하지 못할 수 있다.
- 따라서 터널은 강력하지만, 동시에 감사·필터링·보안 통제가 어려워진다.
- 한쪽이 연결을 닫으면 다른 쪽에도 종료를 전파해야 하며, 타임아웃과 half-close 처리를 잘못하면 연결 누수가 생긴다.

#### 애플리케이션 서버로 이어지는 확장 모델

- CGI와 서버 확장 API는 현대 프레임워크의 조상 격 개념이다.
- 오늘날에는 FastCGI, WSGI, Rack, Servlet 컨테이너, ASGI처럼 웹 서버와 애플리케이션을 분리하는 인터페이스가 널리 쓰인다.
- 이 분리는 정적 파일 처리와 동적 로직 실행을 분리해 성능과 배포 유연성을 높인다.

### 현대 웹 환경에서의 통합 포인트

#### 오늘날의 게이트웨이 패턴

| 패턴 | 설명 |
|------|------|
| **TLS 종료 게이트웨이** | 외부 HTTPS를 받아 복호화 후 내부 서비스에 전달 |
| **API 게이트웨이** | 인증, 속도 제한, 라우팅, 응답 변환 |
| **BFF(Backend for Frontend)** | 클라이언트별로 API 응답 형태를 조정 |
| **서비스 메시 사이드카** | 서비스 간 mTLS, 재시도, 관측성 |

- 책에서 설명하는 게이트웨이 개념은 오늘날 API 게이트웨이, 인그레스 컨트롤러, L7 로드밸런서로 확장되었다고 보면 이해가 쉽다.

#### WebSocket, gRPC, HTTP/2/HTTP/3와의 연결

- WebSocket은 HTTP 업그레이드 이후 별도 양방향 프로토콜처럼 동작하므로, 프록시와 게이트웨이가 `Upgrade` 처리를 정확히 해야 한다.
- gRPC는 HTTP/2 스트림 위에서 동작하므로, 단순 HTTP/1.1 프록시로는 제대로 중계되지 않을 수 있다.
- HTTP/3 환경에서는 QUIC 위에서 `CONNECT` 계열 메커니즘이 확장되어 더 다양한 터널링 시나리오가 가능해졌다.
- FTP 게이트웨이는 책의 역사적 예시로는 유용하지만, 현대 브라우저는 FTP 지원을 대부분 제거했기 때문에 실무에서는 API 게이트웨이와 TLS 프록시가 훨씬 중요하다.

#### 보안 관점에서의 보충

| 항목 | 이유 |
|------|------|
| `CONNECT` 허용 포트 제한 | 오픈 터널 악용 방지 |
| 목적지 화이트리스트 | 내부망으로의 SSRF 우회 차단 |
| 헤더 정리(sanitization) | 내부 인증/추적 헤더 외부 노출 방지 |
| 백엔드 구간 암호화(mTLS) | TLS 종료 후 내부 평문 노출 최소화 |

- HTTPS 종료 지점 이후 내부 구간이 평문이면, "외부만 안전하고 내부는 취약한" 구조가 되기 쉽다.
- 특히 멀티테넌트 환경에서는 게이트웨이의 라우팅 오류가 곧 테넌트 간 데이터 노출로 이어질 수 있다.
