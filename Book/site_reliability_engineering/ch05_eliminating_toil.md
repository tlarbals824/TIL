## 들어가며

SRE는 운영 업무 대신 **장기적인 엔지니어링 프로젝트**에 시간을 쓰고 싶어 한다. "운영 업무(operational work)"라는 표현은 오해의 소지가 있어, Google SRE는 이를 **toil(토일)** 이라는 구체적인 용어로 부른다.

---

## Toil Defined: 토일이란 무엇인가

토일은 단순히 "하기 싫은 일"이 아니다. 행정 업무나 지저분한(grungy) 작업과도 동일하지 않다.

| 구분 | 설명 |
|------|------|
| **Toil** | 프로덕션 서비스 운영에 직접 연결된 반복적·수동적 작업 |
| **Overhead** | 서비스 운영과 직접 연결되지 않은 행정 업무 (팀 미팅, OKR 설정, HR 서류, snippets 등) |
| **Grungy work** | 지저분하지만 장기적 가치가 있으면 토일이 아님 (예: 알림 설정 전체 정리) |

토일은 프로덕션 서비스 운영에 연결된 작업 중 다음 특성을 가진 것이다. 하나 이상에 가까울수록 토일일 가능성이 높다.

### 토일의 6가지 속성

**1. Manual (수동)**
- 스크립트를 수동으로 실행하는 것도 토일이다. 스크립트 실행으로 경과 시간은 줄일 수 있지만, **사람이 직접 손을 대는 시간** 자체가 토일 시간이다.

**2. Repetitive (반복적)**
- 처음이나 두 번째로 하는 작업은 토일이 아니다. 토일은 **반복해서 수행하는 작업**이다. 새로운 문제를 해결하거나 새 솔루션을 발명하는 작업은 토일이 아니다.

**3. Automatable (자동화 가능)**
- 기계가 사람만큼 잘 수행할 수 있거나, 설계 변경으로 작업 자체를 없앨 수 있다면 토일이다. **인간의 판단이 본질적으로 필요**하면 토일이 아닐 가능성이 높다.

> 단, "인간 판단이 필요하니 토일이 아니다"라고 말할 때는 신중해야 한다. 하루에 여러 번 알림이 오고, 각각 복잡한 대응이 필요한 서비스는 **설계가 잘못된 것**이다. 근본 원인을 제거하거나 자동 처리하도록 재설계할 때까지, 알림에 대응하는 작업은 토일이다.

**4. Tactical (전술적)**
- 토일은 **인터럽트 기반·반응적**이며, 전략 기반·선제적이 아니다. 페이저 알림 처리는 토일이다. 완전히 제거할 수는 없지만, 지속적으로 최소화해야 한다.

**5. No enduring value (지속적 가치 없음)**
- 작업 후 서비스 상태가 동일하면 토일일 가능성이 높다. 서비스에 **영구적 개선**을 남겼다면 토일이 아니다 (레거시 코드 정리 같은 grunt work가 포함되어도).

**6. O(n) with service growth (서비스 성장에 비례)**
- 서비스 크기, 트래픽, 사용자 수에 따라 작업량이 선형으로 증가하면 토일이다. 이상적으로 잘 설계·관리된 서비스는 **한 자릿수 이상 성장해도 추가 작업 없이** 운영 가능해야 한다 (리소스 추가 같은 일회성 작업 제외).

## Why Less Toil Is Better: 토일을 줄여야 하는 이유

### 50% 규칙

SRE 조직의 공식 목표: **각 SRE의 토일(운영 업무)을 50% 이하**로 유지한다. 최소 50%는 토일을 줄이거나 서비스 기능을 추가하는 **엔지니어링 프로젝트**에 써야 한다.

- 토일을 방치하면 빠르게 100%를 채운다
- 토일 감소와 서비스 스케일업 작업이야말로 Site Reliability **Engineering**의 "Engineering"
- 이를 통해 SRE 조직은 서비스 규모에 **준선형(sublinear)** 으로 스케일링된다

신규 SRE 채용 시 "일반 Ops 조직이 아니다"라는 약속을 지키기 위해 50% 규칙을 반드시 지켜야 한다.

### 토일 계산

6인 on-call 로테이션 기준:
- 주당 1주 primary + 1주 secondary on-call → 6주 중 2주 = **최소 33%** 토일
- 8인 로테이션 → **최소 25%**

Google SRE 분기별 설문 결과, 평균 토일 비율은 약 **33%** (목표 50%보다 낮음). 하지만 평균은 이상치를 숨긴다 — 0% 토일(순수 개발)부터 80% 토일까지 존재한다. 개인이 과도한 토일을 보고하면, 매니저가 팀 전체에 토일 부하를 분산하고 만족스러운 엔지니어링 프로젝트를 찾도록 해야 한다.

**토일의 주요 원인 (순서대로):**
1. **Interrupts** — 긴급하지 않은 서비스 관련 메시지·이메일
2. **On-call response** — 긴급 대응
3. **Releases and pushes** — 자동화가 있어도 개선 여지가 많음

## What Qualifies as Engineering?: 엔지니어링이란 무엇인가

엔지니어링 작업은 **새롭고 본질적으로 인간의 판단을 요구**한다. 서비스에 **영구적 개선**을 남기며, **전략에 의해 안내**된다. 창의적·혁신적이며, 일반화될수록 좋다. 같은 인원으로 더 많은 서비스를 관리할 수 있게 돕는다.

### SRE 활동 분류

| 카테고리 | 설명 | 예시 |
|---------|------|------|
| **Software engineering** | 코드 작성·수정, 설계, 문서화 | 자동화 스크립트, 도구/프레임워크, 확장성·신뢰성 기능, 인프라 코드 개선 |
| **Systems engineering** | 일회성 노력으로 지속적 개선을 만드는 프로덕션 시스템 구성 | 모니터링 설정, 로드밸런싱, 서버/OS 튜닝, 아키텍처 컨설팅 |
| **Toil** | 반복적·수동적 서비스 운영 작업 | — |
| **Overhead** | 서비스 운영과 직접 무관한 행정 업무 | 채용, HR, 미팅, 버그 큐 정리, peer review, 교육 |

분기 또는 연 단위 평균으로 **최소 50%를 엔지니어링**에 써야 한다. 토일은 변동이 크므로 일부 분기는 50% 미만일 수 있지만, 장기적으로 크게 밑도는 팀은 원인을 파악해야 한다.

## Is Toil Always Bad?: 토일은 항상 나쁜가

소량의 토일은 예측 가능하고 반복적이라 오히려 차분함과 성취감, 빠른 승리를 준다. 일부 사람은 토일 작업을 선호하기도 한다.

**소량의 토일은 SRE 역할에서 불가피하다.** 문제는 **대량의 토일**이다.

### 개인에게 미치는 영향

| 문제 | 설명 |
|------|------|
| **Career stagnation** | 프로젝트 시간이 부족하면 경력 성장이 느려진다. 불가피하고 큰 영향을 주는 grungy work는 인정받지만, grunge로 경력을 쌓을 수는 없다 |
| **Low morale** | 사람마다 한계가 다르지만, 과도한 토일은 번아웃·지루함·불만을 유발한다 |

### SRE 조직에 미치는 영향

| 문제 | 설명 |
|------|------|
| **Creates confusion** | SRE는 엔지니어링 조직이라는 인식을 훼손한다 |
| **Slows progress** | 수동 작업·소방전에 치여 feature velocity가 느려진다 |
| **Sets precedent** | Dev 팀이 SRE에 더 많은 토일을 넘기는 인센티브가 생긴다 |
| **Promotes attrition** | 팀에 토일이 내장되면 최고의 엔지니어가 떠난다 |
| **Causes breach of faith** | 프로젝트 작업을 약속받고 합류한 신규 인력이 배신감을 느낀다 |

## 핵심 정리

| 개념 | 핵심 |
|------|------|
| **Toil** | 수동·반복·자동화 가능·전술적·지속 가치 없음·서비스 성장에 선형 비례 |
| **50% 규칙** | 토일 ≤ 50%, 나머지는 엔지니어링 프로젝트 |
| **Engineering** | 영구적 개선, 전략 기반, 인간 판단 필요, 스케일 가능 |
| **토일의 독성** | 소량은 괜찮지만, 대량은 경력·사기·조직 정체성·이탈을 해친다 |

매주 조금씩 토일을 엔지니어링으로 제거하면, 서비스가 정리되고 스케일 엔지니어링, 차세대 아키텍처, 크로스-SRE 툴체인 구축에 집중할 수 있다. **더 많이 발명하고, 덜 토일하자.**
