Written by Marc Alvidrez / Edited by Kavita Guliani

## 핵심 전제: 100% 신뢰성은 목표가 아니다

Google이 100% 신뢰성을 추구할 것 같지만, 실제로는 **일정 수준 이상의 신뢰성은 오히려 해롭다**는 것이 이 장의 핵심 주장이다.

이유는 두 가지다.

1. **신뢰성이 높아질수록 비용이 기하급수적으로 증가한다.** 한 단계 추가적인 신뢰성 향상은 이전 단계보다 100배 더 비쌀 수 있다.
2. **사용자는 99%와 99.999% 차이를 느끼지 못한다.** 사용자 경험은 스마트폰, 셀룰러 네트워크 등 더 불안정한 구성 요소에 의해 이미 제한된다. 99% 신뢰성 스마트폰을 사용하는 사용자는 서비스가 99.99%인지 99.999%인지 구분할 수 없다.

따라서 SRE의 목표는 **업타임을 단순히 최대화하는 것이 아니라, 가용 불가 리스크와 빠른 혁신·효율적 운영 목표를 균형 있게 맞추는 것**이다. 사용자의 전반적인 행복(기능, 서비스, 성능)을 최적화한다.

## Managing Risk: 리스크 관리의 비용

신뢰성을 높이는 것은 두 가지 차원에서 비용이 발생한다.

### 1. 중복 리소스 비용 (Cost of redundant resources)
- 예비 장비를 유지하거나, 패리티 코드 블록을 저장하거나, 오프라인 유지보수를 위한 여유 공간을 확보하는 비용

### 2. 기회비용 (Opportunity cost)
- 리스크를 줄이는 시스템을 만드는 데 할당된 엔지니어링 자원은 사용자에게 직접적으로 보이는 기능과 제품에 투자될 수 없다

SRE에서는 리스크를 **연속체(continuum)**로 개념화한다. Search, Ads, Gmail, Photos 등 각 서비스를 비선형적 리스크 연속체 위에 배치하고 비용/편익 분석을 수행한다. 핵심은 **"서비스는 필요한 만큼만 신뢰성 있으면 된다"**는 것이다. 99.99% 가용성 목표를 설정했다면, 이를 초과하는 것은 낭비다 — 기능 추가, 기술 부채 해소, 운영 비용 절감의 기회를 날리는 것이다. 이 관점에서 **가용성 목표는 최솟값이자 동시에 최댓값**이다.

## Measuring Service Risk: 서비스 리스크 측정

리스크를 수치화하기 위해 Google은 **비계획 다운타임(unplanned downtime)**에 집중한다.

### Time-based availability (전통적 방법)

```
availability = uptime / (uptime + downtime)
```

연간 기준으로 계산하면 각 가용성 목표별 허용 다운타임을 구할 수 있다.

| 가용성 | 연간 허용 다운타임 |
|--------|-----------------|
| 99% | 87.6시간 |
| 99.9% | 8.76시간 |
| 99.99% | 52.56분 |
| 99.999% | 5.26분 |

### Aggregate availability (Google 방식)

Google은 전 세계 분산 서비스를 운영하므로 시간 기반 지표는 의미가 없다. 어딘가에서는 항상 일부 트래픽을 처리 중이기 때문이다. 대신 **요청 성공률(request success rate)**로 가용성을 정의한다.

```
availability = successful requests / total requests
```

예시: 하루 2.5M 요청을 처리하는 시스템의 일일 가용성 목표가 99.99%라면, 하루 최대 250건의 에러까지 허용된다.

이 방식은 배치 처리, 파이프라인, 트랜잭션 시스템 등 비서빙(non-serving) 시스템에도 적용 가능하다 — 성공/실패 단위의 작업이 있으면 된다. 분기별로 가용성 목표를 설정하고 주간 또는 일간 단위로 추적한다.

## Risk Tolerance of Services: 서비스의 리스크 허용도 파악

서비스 리스크 허용도를 파악하기 위해 SRE는 제품 오너와 협력하여 비즈니스 목표를 엔지니어링 가능한 명시적 목표로 전환해야 한다.

### 소비자 서비스 (Consumer Services)

소비자 서비스는 보통 제품 팀과 제품 관리자가 있어 리스크 요건을 논의할 수 있다. 리스크 허용도를 평가할 때 고려하는 주요 요소들:

**가용성 목표 수준 결정 시 고려사항:**
- 사용자가 기대하는 서비스 수준은?
- 이 서비스가 수익(Google 또는 고객)에 직접 연결되는가?
- 유료 서비스인가, 무료 서비스인가?
- 경쟁사의 서비스 수준은?
- 소비자 대상인가, 기업 대상인가?

**구체적 사례:**
- **Google Apps for Work**: 기업 사용자들이 의존 → 외부 분기 목표 99.9%, 내부 더 강한 목표 + 위반 시 패널티 계약
- **YouTube (2006년 인수 직후)**: 소비자 중심, 빠른 성장 단계 → 기업 제품보다 낮은 가용성 목표 설정 (빠른 기능 개발 중요)

**장애 유형(Types of failures)의 중요성:**
- 동일한 총 에러 수라도 형태가 다를 수 있다 — 지속적 낮은 실패율 vs 간헐적 전체 장애
- 예시: 연락처 앱에서 프로필 사진 렌더링 실패 → 빠른 수정; 다른 사용자의 비공개 연락처 노출 → 신뢰 훼손, 서비스 전체 종료가 적절

**비용(Cost):**
- Ads 시스템: 요청 성공/실패 = 수익 gain/loss로 직접 환산 가능
- 예시: 가용성 99.9% → 99.99% 개선, 서비스 수익 $1M → 개선 가치 = $1M × 0.0009 = $900. 개선 비용이 $900 미만이면 투자 가치 있음
- ISP 배경 에러율(0.01%~1%)을 기준으로 — 서비스 에러율이 배경 노이즈 이하면 추가 개선 의미 없음

**기타 서비스 지표(Other metrics):**
- AdWords: 지연시간이 핵심 불변량 — 광고가 검색 경험을 느리게 하면 안 됨
- AdSense: 서드파티 페이지 렌더링 속도에 종속 → 수백 밀리초 느려도 괜찮음 → 더 적은 지역에 통합 운영 가능 → 운영 오버헤드 절감

### 인프라 서비스 (Infrastructure Services)

인프라 서비스는 소비자 제품과 다르다. **다수의 클라이언트가 있고 각자 니즈가 다르다.**

**Bigtable 사례:**
- 저지연 사용자: 요청 큐가 항상 비어있길 원함 (즉시 처리)
- 처리량(throughput) 사용자: 큐가 절대 비어있지 않길 원함 (최대 처리)
- 두 사용자의 성공 조건은 정반대

**해결책: 명시적 서비스 수준으로 분리**
- **저지연 클러스터**: 슬랙 용량 확보, 강한 클라이언트 격리, 높은 신뢰성
- **처리량 클러스터**: 여유 없이 빡빡하게 운영, 낮은 중복성, 비용 10~50% 절감
- 이를 통해 클라이언트가 자신의 니즈에 맞는 수준을 선택하도록 유도

인프라 서비스의 핵심 전략: **명시적으로 구분된 서비스 수준을 제공하여, 비용 차이를 외부화하고 클라이언트가 최저 비용으로 자신의 니즈를 충족하는 옵션을 선택하게 한다.**

**Frontend infrastructure 사례:**
- 역방향 프록시와 로드 밸런싱 시스템 — 사용자로부터의 연결 엔드포인트
- 요청이 여기서 실패하면 완전히 손실 → 극도로 높은 신뢰성으로 설계

## Motivation for Error Budgets

*Written by Mark Roth / Edited by Carmela Quinito*

### Dev와 SRE 사이의 구조적 긴장

- **Product Development**: 기능 속도(velocity)로 평가 → 빠른 코드 배포 인센티브
- **SRE**: 서비스 신뢰성으로 평가 → 높은 변경률에 저항하는 인센티브

정보 비대칭이 이 긴장을 악화시킨다: Dev는 자신의 코드 개발 시간/노력을 잘 알고, SRE는 프로덕션 신뢰성 상태를 잘 안다. 이 긴장이 다음 영역에서 충돌로 표출된다.

| 영역 | 긴장 |
|------|------|
| 소프트웨어 내결함성 | 너무 적으면 취약한 제품, 너무 많으면 아무도 안 쓰는 제품 |
| 테스트 | 너무 적으면 장애, 너무 많으면 시장 기회 손실 |
| 배포 빈도 | 각 배포는 리스크 — 얼마나 줄일 것인가? |
| 카나리 기간/크기 | 얼마나 기다리고, 얼마나 크게 카나리를 잡을 것인가? |

Google SRE의 비공식 모토: **"Hope is not a strategy."** 결정은 정치, 두려움, 희망이 아니라 객관적 지표로 이루어져야 한다.

### Error Budget 형성

Error Budget은 두 팀이 분기별로 공동으로 정의하며, SLO 기반으로 계산된다.

**프로세스:**
1. Product Management가 SLO를 정의 (분기 내 서비스 기대 업타임)
2. 실제 업타임은 중립적 제3자인 모니터링 시스템이 측정
3. 두 숫자의 차이 = 분기 내 남은 "허용 비신뢰성 예산"
4. Error Budget이 남아있는 한 → 새 릴리스 배포 가능

**예시:**
- SLO: 분기당 쿼리의 99.999% 성공
- Error Budget = 실패율 0.001% (분기 기준)
- 문제 발생으로 0.0002%의 쿼리 실패 → 분기 에러 예산의 20% 소진

### Error Budget의 이점

1. **공통 인센티브 제공**: Dev와 SRE 모두 혁신과 신뢰성 사이의 올바른 균형을 찾는 데 집중
2. **자기 규제(self-policing)**: 에러 예산이 충분할 때 Dev는 더 많은 리스크를 감수하고, 예산이 거의 소진되면 Dev 스스로 더 많은 테스트와 느린 배포 속도를 원하게 된다 — 예산을 다 써서 런칭이 중단되길 원하지 않기 때문
3. **정치 제거**: "Ops 팀의 주관적 판단"이 아닌 "에러 예산 잔액"이라는 객관적 지표가 결정
4. **공동 책임**: 네트워크 장애나 데이터센터 장애도 에러 예산을 소진 → 팀 전체가 업타임에 대한 책임을 공유
5. **과도한 신뢰성 목표의 비용 가시화**: SLO를 낮추면(에러 예산 증가) 혁신 속도가 높아진다

**핵심 인사이트:**
- 서비스 신뢰성 관리는 곧 리스크 관리이며, 리스크 관리에는 비용이 따른다
- 100%는 올바른 신뢰성 목표가 될 수 없다 — 달성 불가능하고, 사용자도 원하지 않는다
- Error Budget은 SRE와 Product Development 간 인센티브를 정렬하고 공동 소유권을 강조한다
