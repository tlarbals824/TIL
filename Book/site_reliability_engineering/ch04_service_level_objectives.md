## 들어가며

서비스를 올바르게 관리하려면 **어떤 동작이 실제로 중요한지, 그리고 그것을 어떻게 측정하고 평가할지** 이해해야 한다. 이를 위해 SRE는 특정 *서비스 수준(level of service)*을 정의하고 제공한다.

**직관, 경험, 사용자가 원하는 것에 대한 이해**를 바탕으로 세 가지 개념을 정의한다:
- **SLI (Service Level Indicator)**: 지표 — 어떤 값을 측정할 것인가
- **SLO (Service Level Objective)**: 목표 — 그 값이 어느 수준이어야 하는가
- **SLA (Service Level Agreement)**: 계약 — 목표를 충족하지 못하면 무슨 일이 생기는가

적절한 지표를 선택하면 문제 발생 시 올바른 행동을 유도하고, 서비스가 건강한지 SRE 팀이 자신있게 판단할 수 있다.

## Service Level Terminology

### Indicators (SLI)

SLI는 **서비스 수준의 특정 측면을 나타내는 신중하게 정의된 정량적 지표**다.

**가장 일반적인 SLI:**
- **요청 지연시간 (Request latency)**: 응답을 반환하는 데 걸리는 시간
- **에러율 (Error rate)**: 전체 요청 중 실패한 비율
- **시스템 처리량 (System throughput)**: 초당 처리 요청 수

측정값은 종종 집계된다 — 측정 윈도우에서 수집된 원시 데이터를 비율, 평균, 백분위수로 변환.

**가용성(Availability)**도 중요한 SLI다 — 서비스가 사용 가능한 시간의 비율. 잘 형성된 요청이 성공하는 비율(yield)로 정의된다. **내구성(Durability)**도 데이터 스토리지 시스템에서 중요하다 — 장기간에 걸쳐 데이터가 유지될 가능성.

SLI는 이상적으로는 원하는 서비스 수준을 직접 측정해야 하지만, **프록시(proxy)만 가능한 경우**도 있다. 예: 클라이언트 측 지연시간이 사용자 관련성이 더 높지만, 서버에서만 측정 가능할 수 있다.

### Objectives (SLO)

SLO는 SLI로 측정되는 **서비스 수준의 목표값 또는 범위**다.

- `SLI ≤ target` 형태, 또는
- `lower bound ≤ SLI ≤ upper bound` 형태

예시: "평균 검색 요청 지연시간이 100ms 미만이어야 한다"

SLO를 선택하는 것은 복잡하다. 일부 지표는 사용자가 결정한다 — 예를 들어 외부에서 들어오는 HTTP 요청의 QPS는 사용자 욕구로 결정되므로 SLO를 설정할 수 없다. 반면 평균 지연시간은 목표를 설정할 수 있고, 이는 프론트엔드를 저지연 방식으로 구현하거나 특정 장비를 구매하는 동기가 된다.

**SLO를 사용자에게 공개하면** 서비스 동작에 대한 기대치를 설정하여, 근거 없는 불만을 줄일 수 있다. 명시적 SLO가 없으면 사용자는 직접 믿음을 형성하고 이는 실제 설계/운영 목표와 다를 수 있다.

**Chubby 사례 — The Global Chubby Planned Outage:**
Chubby(Google의 분산 락 서비스)는 워낙 신뢰성이 높아 서비스 오너들이 "절대 다운되지 않는다"고 가정하고 의존성을 추가했다. 이는 Chubby가 실제로 드물게 장애가 발생할 때 연쇄 장애를 일으켰다. 해결책: SRE가 Chubby의 SLO를 충족하되 *크게 초과하지 않도록* 보장. 분기에 자연적 장애가 SLO 이하로 떨어지지 않으면, 의도적으로 계획된 다운타임을 만들어 불합리한 Chubby 의존성을 조기에 강제 제거.

### Agreements (SLA)

SLA는 **SLO의 충족/불충족에 대한 결과를 포함하는 명시적/묵시적 계약**이다. 결과는 금전적(환불 또는 패널티)이 가장 명확하지만 다른 형태도 있다.

SLO와 SLA를 구분하는 쉬운 방법: "SLO를 충족하지 못하면 무슨 일이 생기는가?" — 명시적 결과가 없으면 SLO다.

**SRE의 역할:**
- SLA 구성에는 보통 직접 개입하지 않음 (비즈니스/법무 영역)
- SLA에 포함된 SLO를 충족하지 못하는 결과를 *피하도록* 돕는 역할
- SLO 측정을 위한 SLI 정의를 도움

**Google Search**는 공개 SLA 없음 — 하지만 비가용성은 명성 훼손 + 광고 수익 감소라는 결과 발생. **Google for Work**는 명시적 SLA 보유. SLA 유무와 관계없이 SLI와 SLO를 정의하고 서비스 관리에 활용하는 것이 중요하다.

## Indicators in Practice

### 무엇을 측정할 것인가?

모니터링 시스템으로 추적할 수 있는 모든 지표를 SLI로 사용하면 안 된다. 사용자가 시스템에서 원하는 것에 대한 이해가 지표 선택을 이끌어야 한다. **너무 많은 지표**는 중요한 지표에 충분한 주의를 기울이기 어렵게 만들고, **너무 적은 지표**는 중요한 시스템 동작을 확인하지 못하게 한다.

**서비스 유형별 핵심 SLI:**

| 서비스 유형 | 핵심 SLI |
|------------|---------|
| User-facing serving (Shakespeare 검색 등) | 가용성, 지연시간, 처리량 |
| Storage | 지연시간, 가용성, 내구성 |
| Big data (데이터 처리 파이프라인) | 처리량, end-to-end 지연시간 |
| 모든 시스템 | 정확성 (correctness) |

**정확성(Correctness)**: 올바른 응답이 반환되었는가, 올바른 데이터가 검색되었는가. 인프라가 아닌 데이터의 속성이므로 보통 SRE 책임이 아니지만, 시스템 건강의 지표로 추적할 가치가 있다.

### 지표 수집 (Collecting Indicators)

- 대부분의 지표는 **서버 사이드**에서 수집 (Borgmon, Prometheus, HTTP 로그 분석 등)
- **클라이언트 사이드** 수집도 중요 — 서버 지표만 보면 JavaScript 문제 등 사용자에게 영향을 미치지만 서버 지표에는 나타나지 않는 문제를 놓친다
  - 예: Shakespeare 검색 백엔드 응답 지연시간은 괜찮지만, 페이지 JavaScript 문제로 사용자의 실제 경험은 느릴 수 있다. 이 경우 페이지가 사용 가능해지는 데 걸리는 시간이 더 나은 프록시

### 집계 (Aggregation)

원시 측정값을 집계할 때 주의가 필요하다.

**평균의 함정:**
- "초당 요청 수"조차도 측정 윈도우에 따라 의미가 달라진다. 짝수 초에 200 req/s, 홀수 초에 0인 시스템은 평균 100 req/s이지만, 순간 부하는 2배
- **요청 지연시간의 평균**은 중요한 세부사항을 숨긴다 — 대부분 요청이 빠르지만 긴 꼬리(long tail)가 훨씬 느릴 수 있다

**분포로 생각하기:**
- 지표는 평균보다 **분포(distribution)**로 생각하는 것이 낫다
- 예: 전형적 요청이 50ms이지만 요청의 5%는 20배 느림. 평균만 모니터링하면 하루 종일 변화 없는 것으로 보일 수 있지만 꼬리 지연시간은 크게 변하고 있다

**백분위수(Percentile) 사용:**
- **99번째 또는 99.9번째 백분위수**: 최악의 경우(plausible worst-case) 표시
- **50번째 백분위수(중앙값)**: 전형적인 경우 표시
- 응답 시간 분산이 클수록 사용자 경험에 더 많은 영향을 미침 (높은 부하에서 큐잉 효과로 악화)
- 일부 SRE 팀은 99.9번째 백분위수만 집중 — 그 값이 좋으면 전형적 경험은 확실히 좋다

**통계적 오류 주의:**
- 평균(산술 평균)보다 백분위수를 선호 — 긴 꼬리의 데이터 포인트를 고려 가능
- 컴퓨팅 시스템 데이터는 종종 치우침(skewed) — 응답은 0ms 미만 불가, 타임아웃(1000ms)을 초과하는 성공 응답도 불가. 평균과 중앙값이 같다고 가정하면 안 됨
- 데이터가 정규 분포라고 가정하지 말 것 — 먼저 검증해야 한다

### 지표 표준화 (Standardize Indicators)

SLI 정의를 표준화하여 매번 기본 원칙부터 다시 생각할 필요가 없도록 한다. 표준 정의 템플플릿을 따르는 기능은 개별 SLI 명세에서 생략 가능하다.

표준화 예시:
- 집계 간격: "1분 평균"
- 집계 범위: "클러스터 내 모든 태스크"
- 측정 빈도: "10초마다"
- 포함 요청: "블랙박스 모니터링 작업의 HTTP GET"
- 데이터 수집 방식: "서버에서 측정된 모니터링"
- 데이터 접근 지연시간: "마지막 바이트까지의 시간"

각 공통 지표에 대한 재사용 가능한 SLI 템플릿 세트를 구축 → 효율성 향상 + 누구나 특정 SLI의 의미를 쉽게 이해

## Objectives in Practice

### 목표 정의하기 (Defining Objectives)

**출발점**: 측정 가능한 것이 아니라 **사용자가 무엇을 원하는지**부터 생각하라. 사용자가 원하는 것을 측정하기 어려우면 사용자 니즈를 근사하는 방법을 찾게 된다. 측정하기 쉬운 것부터 시작하면 덜 유용한 SLO가 된다.

때로는 **원하는 목표에서 역방향으로 특정 지표를 도출**하는 방식이 더 효과적이다.

**명확한 SLO 명세:**
- SLO는 어떻게 측정하는지, 어떤 조건에서 유효한지 명시해야 한다
- 예시: "99%의 Get RPC 호출이 100ms 미만에 완료 (모든 백엔드 서버에 걸쳐 1분 평균 측정)"

**여러 목표 지정:**
- 성능 곡선의 형태가 중요하다면 여러 SLO 목표를 지정할 수 있다:
  - 90%의 Get RPC 호출: 1ms 미만
  - 99%의 Get RPC 호출: 10ms 미만
  - 99.9%의 Get RPC 호출: 100ms 미만

- 처리량과 지연시간 두 워크로드가 섞여 있다면 각각 별도 목표:
  - 처리량 클라이언트의 95%: Set RPC 1초 미만
  - 지연시간 클라이언트(1kB 미만 페이로드)의 99%: Set RPC 10ms 미만

**에러 예산:** SLO를 100%의 시간 동안 충족하는 것은 비현실적이고 바람직하지 않다. 혁신과 배포 속도를 줄이고, 과도하게 보수적인 솔루션이 필요하다. 대신 SLO를 위반할 수 있는 비율(에러 예산)을 허용하고 일/주 단위로 추적한다.

### 목표 선택하기 (Choosing Targets)

SLO 선택은 순수한 기술적 활동이 아니다 — 제품과 비즈니스 함의가 반영되어야 한다.

**Google이 배운 교훈들:**

**현재 성능 기반으로 목표를 선택하지 말 것**
- 시스템의 장점과 한계를 이해하는 것은 중요하지만, 성찰 없이 현재 값을 채택하면 상당한 재설계 없이는 개선 불가능한 목표를 지원해야 하는 상황에 갇힐 수 있다

**단순하게 유지할 것**
- SLI의 복잡한 집계는 시스템 성능 변화를 숨기고 추론하기 어렵게 만든다

**절댓값을 피할 것**
- 지연 증가 없이 무한히 스케일하고 "항상" 가용한 시스템을 요구하는 것은 비현실적
- 이런 이상에 근접한 시스템도 설계/구축에 오래 걸리고 운영 비용이 높다

**SLO를 최대한 적게 유지할 것**
- 시스템 속성에 대해 충분한 커버리지를 제공하는 만큼만 선택
- 특정 SLO를 인용해서 우선순위 논쟁에서 이길 수 없다면, 그 SLO는 보유할 가치가 없을 것이다

**완벽함은 나중에**
- SLO 정의와 목표는 시스템 동작을 배우면서 시간이 지남에 따라 개선할 수 있다
- 달성 불가능한 엄격한 목표보다 나중에 강화하는 느슨한 목표로 시작하는 것이 낫다

SLO는 SRE와 제품 개발자의 작업 우선순위를 이끄는 주요 동력이 될 수 있고 그래야 한다. 잘 설계된 SLO는 개발팀에게 유용한 강제 함수(forcing function)가 된다. 하지만 잘못 설계된 SLO는 팀이 무의미하게 영웅적 노력을 기울이거나 너무 느슨해서 나쁜 제품을 만들 수 있다.

### Control Measures: 제어 루프

SLI와 SLO는 시스템을 관리하는 제어 루프의 핵심 요소다:

1. 시스템의 SLI를 모니터링하고 측정한다
2. SLI를 SLO와 비교하고, 행동이 필요한지 결정한다
3. 행동이 필요하다면, 목표를 충족하기 위해 *무엇*이 필요한지 파악한다
4. 그 행동을 취한다

예시: 2단계에서 요청 지연시간이 증가하고 몇 시간 안에 SLO를 위반할 것으로 보이면, 3단계에서 서버가 CPU 바운드인지 가설을 테스트하고 더 많이 추가하기로 결정할 수 있다. SLO 없이는 언제 (또는 어떻게) 행동할지 알 수 없다.

### SLO가 기대치를 설정한다 (SLOs Set Expectations)

SLO를 공개함으로써 시스템 동작에 대한 기대치를 설정한다. 사용자(및 잠재 사용자)는 종종 서비스를 자신의 유스케이스에 적합한지 이해하기 위해 기대치를 알고 싶어한다.

**사용자 기대치를 현실적으로 설정하는 전술:**

**Safety margin (안전 마진) 유지:**
- 사용자에게 광고하는 SLO보다 내부적으로 더 엄격한 SLO를 사용하면 만성적 문제가 외부에 가시화되기 전에 대응할 여유가 생긴다
- 성능을 비용 또는 유지보수 용이성과 맞바꾸는 재구현을 사용자를 실망시키지 않고 수용 가능

**과도한 달성(Overachieve) 금지:**
- 사용자는 서비스가 제공한다고 *말하는 것*이 아니라 *실제로 제공하는 것*을 기준으로 구축한다
- 서비스의 실제 성능이 명시된 SLO보다 훨씬 좋으면, 사용자는 현재 성능에 의존하게 된다
- 의도적으로 가끔 시스템을 오프라인으로 만들거나(Chubby 계획된 다운타임), 일부 요청을 쓰로틀하거나, 낮은 부하에서 더 빠르게 만들지 않는 방식으로 과도한 의존을 방지

시스템이 기대치를 얼마나 잘 충족하는지 이해하면 시스템을 더 빠르고, 더 가용하고, 더 탄력적으로 만드는 데 투자할지, 아니면 기술 부채 해소, 새 기능 추가, 다른 제품 도입에 시간을 쓸지 결정하는 데 도움이 된다.

---

## Agreements in Practice

SLA를 만드는 것은 비즈니스와 법무 팀이 위반에 대한 적절한 결과와 패널티를 선택해야 한다. SRE의 역할은 SLA에 포함된 SLO를 충족할 가능성과 난이도를 이해하도록 돕는 것이다.

SLO 구성에 관한 대부분의 조언은 SLA에도 적용된다. 사용자에게 광고하는 내용에 보수적이어야 한다 — 구성원이 많을수록 현명하지 않거나 다루기 어렵다는 것을 입증하는 SLA를 변경하거나 삭제하기가 더 어렵다.

## 핵심 정리

| 개념 | 정의 | 예시 |
|------|------|------|
| SLI | 서비스 수준의 정량적 지표 | 99번째 백분위수 지연시간 = 200ms |
| SLO | SLI에 대한 목표값 | 99%의 요청이 200ms 미만 |
| SLA | SLO 충족/불충족에 대한 계약 | SLO 미충족 시 사용자에게 크레딧 환불 |

**실무 가이드라인:**
- 사용자가 신경 쓰는 것을 측정하라 — 측정 가능한 것이 아니라
- 평균이 아닌 분포(백분위수)를 사용하라
- 단순하게 유지하고, 적게 유지하고, 시간이 지남에 따라 개선하라
- SLO는 최솟값이 아니라 최솟값이자 최댓값이다 (과도한 달성도 문제)
- SLA가 없더라도 SLI와 SLO는 정의하고 사용해야 한다
