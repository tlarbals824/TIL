# Chapter 13: Emergency Response

Written by Corey Adam Baye

> "Things break; that's life."

---

## 들어가며

시스템은 깨진다. 그것이 현실이다. 조직의 규모나 이해관계와 관계없이, 장기적으로 조직을 차별화하는 핵심 요소는 **비상 상황에 어떻게 대응하느냐**이다.

대부분의 사람은 비상 상황에서 자연스럽게 잘 대응하지 못한다. 적절한 대응은 **준비**와 **정기적이고 관련성 있는 실습(hands-on training)**이 필요하다. 철저한 훈련·테스트 프로세스를 구축·유지하려면 경영진·이사회의 지원과 직원의 세심한 관심이 모두 필요하다. 이를 통해 팀이 돈, 시간, 에너지, 심지어 uptime을 투자해 비상 시 효율적으로 대응할 수 있는 환경이 조성된다.

이 장은 postmortem 문화(Chapter 15)에서 다루는 학습 기회와 연결되는 **구체적인 비상 사례**를 제공한다.

---

## What to Do When Systems Break

**첫 번째: 당황하지 마라.**

- 하늘이 무너지는 것이 아니다. 당신은 이 상황을 처리하도록 훈련된 전문가다.
- 보통 물리적 위험은 없다 — 전자만 위험에 처한다. 최악의 경우에도 인터넷의 절반이 다운될 뿐이다.
- 압도되면 더 많은 사람을 끌어들여라. 필요하면 회사 전체에 페이지를 보낼 수도 있다.
- 인시던트 대응 프로세스(Chapter 14)가 있다면 반드시 숙지하고 따를 것.

## Test-Induced Emergency (테스트로 인한 비상)

Google은 재해·비상 테스트에 **선제적(proactive)** 접근을 채택한다. SRE는 시스템을 의도적으로 깨뜨리고, 실패 방식을 관찰하며, 재발을 막기 위한 변경을 한다. 대부분은 계획대로 진행되지만, 가끔 가정과 실제 결과가 완전히 다르다.

### 사례: 테스트 DB 접근 차단

**Details**
- 대규모 분산 MySQL DB 중 100개 중 1개 테스트 DB에 대한 숨겨진 의존성을 찾아내기 위해 접근을 차단하는 테스트 계획.

**Response**
- 테스트 시작 몇 분 만에 수많은 의존 서비스가 외부·내부 사용자의 핵심 시스템 접근 불가 보고.
- SRE는 즉시 테스트 중단. 롤백 시도했으나 실패.
- 이미 검증된 방식으로 replica/failover에 권한 복원. 핵심 개발자에게 DB 애플리케이션 레이어 라이브러리 수정 요청.
- **1시간 이내** 전체 접근 복원.

**Findings — What went well**
- 영향받은 서비스가 즉시 사내 에스컬레이션.
- 테스트가 통제 불능이 된 것을 올바르게 가정하고 즉시 중단.
- 일부 팀은 테스트 DB를 피하도록 시스템 재구성하는 병렬 노력.
- 후속 조치 빠르게 해결, 주기적 재테스트 도입.

**What we learned**
- 테스트는 철저히 검토됐지만 의존 시스템 간 상호작용에 대한 이해가 불충분했음.
- 몇 주 전에 도입된 인시던트 대응 프로세스를 따르지 않음 → 고객·서비스에 outage 인지 통보 누락.
- 롤백 절차를 테스트 환경에서 검증하지 않아 롤백이 결함 있었고 outage가 길어짐.
- **대규모 테스트 전 롤백 절차의 철저한 테스트를 필수화.**

## Change-Induced Emergency (변경으로 인한 비상)

Google은 방대하고 복잡한 설정을 지속적으로 변경한다. 변경 전 수많은 테스트를 수행하지만, 규모와 복잡성 때문에 모든 의존성·상호작용을 예측하는 것은 불가능하다.

### 사례: 금요일 글로벌 설정 변경

**Details**
- 서비스 남용 방지 인프라에 대한 설정 변경이 금요일에 글로벌 푸시.
- 외부 facing 시스템 전반과 상호작용하는 인프라에서 crash-loop 버그 발생.
- 전체 fleet이 거의 동시에 crash-loop 시작. Google 내부 인프라도 자체 서비스에 의존하므로 내부 앱 다수 사용 불가.

**Response**
- 수 초 내 모니터링 알림 발화.
- 일부 on-call 엔지니어는 corporate network 장애로 판단, backup access가 있는 secure room(panic room)으로 이동.
- 5분 내 push 엔지니어가 롤백 푸시 → 서비스 복구 시작.
- 10분 내 인시던트 선언, 사내 통보 시작.
- 일부 서비스는 원래 이벤트로 인한 무관한 버그/설정 오류로 1시간까지 완전 복구 안 됨.

**Findings — What went well**
- 모니터링이 거의 즉시 문제 감지·알림 (단, 알림이 반복·과다해 on-call과 커뮤니케이션 채널을 압도).
- 인시던트 관리는 전반적으로 잘 진행, 업데이트가 자주·명확하게 전달.
- out-of-band 커뮤니케이션 시스템이 복잡한 소프트웨어 스택이 사용 불가해도 연결 유지.
- CLI 도구·대체 접근 방법으로 다른 인터페이스가 불가해도 업데이트·롤백 가능.
- 영향받은 시스템이 rate-limit으로 전체 업데이트 속도 제한 → crash-loop 완화, 완전 outage 방지 가능.
- **운(luck)**: push 엔지니어가 실시간 커뮤니케이션 채널을 모니터링 중이었고, complaint를 보고 즉시 롤백.

**What we learned**
- 이전 canary는 희귀한 설정 키워드+신기능 조합을 테스트하지 않아 버그 미발견.
- 이번 변경은 위험하지 않다고 판단되어 덜 엄격한 canary 프로세스 적용 → 글로벌 푸시 시 미테스트 조합으로 실패.
- canary·자동화 개선이 다음 분기 우선순위였으나, 이 인시던트로 즉시 상향.
- 모든 location이 오프라인이라 알림이 폭주 → 실제 작업과 커뮤니케이션 방해.
- Google은 자체 도구에 의존 — troubleshooting·커뮤니케이션 스택도 crash-looping job 뒤에 있어 outage가 더 길었다면 디버깅 심각히 저해됨.

## Process-Induced Emergency (프로세스로 인한 비상)

자동화로 fleet 전체에서 수많은 job을 쉽게 시작·중지·재구성할 수 있다. 자동화의 효율성이 계획과 다르게 진행될 때는 무서울 수 있다.

### 사례: turndown 자동화 버그

**Details**
- 폐기 예정 소규모 서버 설치에 대해 연속 두 번의 turndown 요청 제출.
- 자동화의 미묘한 버그로 **전 세계 모든 설치의 모든 머신**이 Diskerase 큐로 전송 → 하드 드라이브 wipe 예정.

**Response**
- 첫 소규모 설치가 offline되자 on-call 페이지.
- 머신이 Diskerase 큐로 이동한 것 확인 → 정상 절차대로 해당 location 트래픽 drain.
- 곧 전 세계 모든 해당 설치에서 pager 발화.
- 모든 팀 자동화 비활성화, 추가 자동화·프로덕션 유지보수 중지·동결.
- **1시간 내** 모든 트래픽 다른 location으로 우회. 사용자는 latency 증가했지만 요청 처리됨.
- 복구: 네트워크 혼잡 완화, 한 설치를 먼저 재구축 → 3시간 내 복구.
- US 팀이 EU 팀에 핸드오프. 수동 재설치 프로세스로 3파트 분할 → **3일 내** 대부분 capacity 복구.

**Findings — What went well**
- 대규모 설치의 reverse proxy는 다르게 관리되어 영향 없음.
- 소규모 → 대규모 설치로 트래픽 빠르게 이동. 대규모 설치는 full load 처리 가능.
- turndown 프로세스 자체는 효율적 (1시간 이내 wipe 완료).
- 모니터링 변경 revert로 피해 범위 파악.
- 1년 전 첫 사례 이후 성숙한 인시던트 대응 프로토콜 준수, 커뮤니케이션·협업 우수.

**What we learned**
- **Root cause**: turndown 자동화 서버에 명령에 대한 적절한 sanity check 부재.
- 실패한 turndown 재실행 시 빈 rack 응답 → 필터링 없이 machine DB에 전달 → **"zero does mean all"** — 모든 머신 Diskerase.
- 재설치가 느리고 불안정: TFTP + 최저 QoS + BIOS의 poor failure handling.
- 재설치 인프라가 수천 대 동시 setup 처리 불가 (regression: worker당 2 setup task 제한, 잘못된 QoS, 부적절한 timeout).

## All Problems Have Solutions

시간과 경험이 보여주듯, 시스템은 예상치 못한 방식으로 깨진다. Google의 가장 큰 교훈 중 하나: **해결책은 반드시 존재한다** — 당장 떠오르지 않더라도.

- 해결책이 떠오르지 않으면 범위를 넓혀라. 더 많은 팀원, 도움 요청, 빠르게.
- **최우선 순위: 문제를 빠르게 완화(mitigate).**
- 종종 이벤트를 트리거한 사람이 가장 많은 state를 가지고 있음 — 그 사람을 활용하라.

비상이 완화되면 반드시:
- 정리(clean up)
- 인시던트 문서화
- **과거에서 배우고 반복하지 말 것**

## Learn from the Past. Don't Repeat It.

### Keep a History of Outages

과거에 무엇이 깨졌는지 문서화하는 것보다 더 나은 학습 방법은 없다. 역사는 **모든 사람의 실수**로부터 배우는 것이다.

- 철저하고, 정직하게, **어려운 질문**을 하라.
- outage 재발을 막을 구체적 행동을 찾되, 전술적·전략적 모두.
- postmortem을 발행·조직화해 회사 전체가 학습하게 하라.
- postmortem의 구체적 후속 조치에 책임을 지게 하라.

### Ask the Big, Even Improbable, Questions: What If…?

현실보다 더 큰 시험은 없다.

- 건물 전력이 끊기면…?
- 네트워크 랙이 2피트 물에 잠기면…?
- primary datacenter가 갑자기 꺼지면…?
- 웹 서버가 침해되면…?
- 누구를 부를 것인가? 어떻게 반응할 것인가? 시스템은 어떻게 반응할 것인가?
- 옆자리 동료도 같은 대응을 할 수 있는가?

### Encourage Proactive Testing

이론과 현실은 매우 다른 영역이다. 시스템이 실제로 실패하기 전까지는 시스템, 의존 시스템, 사용자가 어떻게 반응할지 진정으로 알 수 없다.

- 가정이나 테스트하지 못한 것에 의존하지 마라.
- 토요일 새벽 2시(회사 대부분이 team-building offsite 중)에 실패가 일어나는 것과, 최고의 엔지니어들이 면밀히 검토한 테스트를 모니터링하는 것 중 어느 쪽을 선호하겠는가?

## Conclusion

세 가지 비상 사례 — proactive 테스트, 설정 변경, turndown 자동화 — 는 트리거가 달랐지만 대응은 공통점이 많았다:

- 당황하지 않음
- 필요 시 더 많은 사람 동원
- 이전 outage에서 학습
- 시스템을 outage 유형에 더 잘 대응하도록 구축
- 새로운 failure mode 문서화
- proactive 테스트로 수정 검증 및 약점 사전 발견

시스템이 진화할 때마다 이 사이클이 반복되며, 프로세스와 시스템 모두 점진적으로 개선된다.

---

## 핵심 정리

| 유형 | 트리거 | 핵심 교훈 |
|------|--------|-----------|
| Test-induced | 의도적 장애 주입 테스트 | 롤백 절차 사전 검증, 인시던트 프로세스 준수, 의존성 이해의 한계 |
| Change-induced | 설정/릴리스 변경 | 철저한 canary(위험도와 무관), 알림 폭주 관리, out-of-band 도구 |
| Process-induced | 자동화 버그 | sanity check, "zero = all" 같은 엣지 케이스, 복구 인프라 용량 |

**비상 대응 원칙**
1. 당황하지 마라 — 전문가로서 훈련받았다
2. 빠르게 완화 — root cause는 그 다음
3. 해결책은 존재한다 — 범위를 넓혀라
4. 과거 outage 기록·학습 — postmortem + proactive testing
5. What-if 질문 — 재해 시나리오 사전 계획

## 추가로 알면 좋은 내용

### Disaster Recovery Testing (DiRT)

Google의 proactive 테스트 문화는 **DiRT (Disaster Recovery Testing)** 프로그램으로 체계화되어 있다. 정기적으로 실제 프로덕션 환경에서 장애를 주입해 시스템·프로세스·사람의 대응 능력을 검증한다. Netflix의 **Chaos Monkey**와 유사한 철학이지만, Google은 조직 전체 차원의 재해 복구 훈련에 더 초점을 둔다.

### Rollback은 First-Class Citizen

이 장의 반복되는 교훈: **롤백 절차를 배포 전에 반드시 테스트**해야 한다. "배포는 잘 되는데 롤백은 안 된다"는 흔한 안티패턴이다. Blue-green, canary 배포에서 롤백 경로를 별도로 검증하는 것이 필수다.

### Out-of-Band (OOB) 통신·접근

프로덕션 시스템이 다운되면 Slack, 이메일, 내부 도구도 함께 죽을 수 있다. **별도의 통신 채널**(전화, SMS, 물리적 secure room)과 **CLI 기반 대체 접근**을 미리 구축·정기 테스트해야 한다.

### 자동화의 이중성

자동화는 toil을 줄이지만, 버그가 있으면 **인간보다 훨씬 빠르게 대규모 피해**를 낸다 (process-induced emergency 사례). 자동화에는 반드시:
- **Sanity check** (입력 검증, 범위 제한)
- **Circuit breaker** (연쇄 실행 방지)
- **Kill switch** (즉시 중단 가능)
- **Idempotency** (중복 실행 안전)

### FEMA Incident Command System (ICS) 연결

Chapter 14에서 다루는 ICS는 미국 연방재난관리청(FEMA)에서 개발한 표준이다. 소방·의료·재난 대응에서 검증된 프레임워크로, IT 인시던트 관리의 기반이 되었다. Chapter 13의 "당황하지 마라, 사람을 끌어들여라"는 ICS의 기본 정신과 일치한다.

### PagerDuty / Incident.io 등 현대 도구

Google의 Escalator·Outalator(Chapter 16)와 유사한 기능을 제공하는 상용 도구들:
- **PagerDuty**: 에스컬레이션, on-call 스케줄, 인시던트 워크플로
- **Incident.io**: 인시던트 선언, 역할 할당, 타임라인 자동 기록
