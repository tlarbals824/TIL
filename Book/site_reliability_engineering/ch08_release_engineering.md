## 들어가며

Release engineering(릴리즈 엔지니어링)은 **소프트웨어를 빌드하고 전달하는** 비교적 새롭고 빠르게 성장하는 소프트웨어 엔지니어링 분야다.

릴리즈 엔지니어는 다음에 대한 탄탄한(전문가 수준은 아니어도) 이해를 갖는다:
- 소스 코드 관리, 컴파일러, 빌드 설정 언어
- 자동화 빌드 도구, 패키지 매니저, 인스톨러
- 개발, 구성 관리, 테스트 통합, 시스템 관리, 고객 지원

신뢰할 수 있는 서비스 운영에는 **신뢰할 수 있는 릴리즈 프로세스**가 필요하다. SRE는 사용하는 바이너리와 설정이 **재현 가능하고 자동화된 방식**으로 빌드되었는지 알아야 한다. 릴리즈는 "눈송이(unique snowflake)"가 아니라 **반복 가능**해야 한다. 소스 코드부터 배포까지 모든 변경이 **의도적**이어야 한다.

## The Role of a Release Engineer: 릴리즈 엔지니어의 역할

Google은 데이터 기반 회사이며, 릴리즈 엔지니어링도 마찬가지다.

**측정하는 메트릭:**
- 코드 변경 → 프로덕션 배포까지 걸리는 시간 (release velocity)
- 빌드 설정 파일에서 사용되는 기능 통계

릴리즈 엔지니어는 **일관되고 반복 가능한 방법론**으로 프로젝트가 릴리즈되도록 모범 사례와 도구를 정의한다:
- 컴파일러 플래그, 빌드 식별 태그 형식, 빌드 중 필수 단계

도구가 기본적으로 올바르게 동작하고 문서화가 잘 되면, 팀은 릴리즈 방식을 재발명하는 대신 **기능과 사용자**에 집중할 수 있다.

**SRE와의 협업:**
- 카나리 변경 전략
- 서비스 중단 없는 새 릴리즈 push
- 문제가 있는 기능의 롤백

## Philosophy: 4대 원칙

### 1. Self-Service Model (셀프 서비스)

스케일에서 작동하려면 팀이 **자립적**이어야 한다. 수천 명의 엔지니어와 제품이 있어도, 각 팀이 **언제·얼마나 자주** 릴리즈할지 결정할 수 있어 높은 release velocity를 달성한다.

- 릴리즈 프로세스를 엔지니어 개입 최소화까지 자동화
- 많은 프로젝트가 자동 빌드 시스템 + 배포 도구로 **완전 자동** 릴리즈
- 문제 발생 시에만 엔지니어 개입

### 2. High Velocity (높은 속도)

사용자 대면 소프트웨어(Google Search 등)는 **빈번하게** 재빌드된다.

> **빈번한 릴리즈 = 버전 간 변경이 적음** → 테스트와 트러블슈팅이 쉬워진다.

| 모델 | 설명 |
|------|------|
| **Hourly builds + selection** | 시간당 빌드 후 테스트 결과·기능 기반으로 배포 버전 선택 |
| **Push on Green** | 모든 테스트 통과 빌드를 자동 배포 |

### 3. Hermetic Builds (헤르메틱 빌드)

같은 리비전을 다른 머신에서 빌드하면 **동일한 결과**가 나와야 한다.

- 빌드 머신에 설치된 라이브러리·소프트웨어에 **둔감**
- 알려진 버전의 빌드 도구·의존성에만 의존
- **외부 서비스에 의존하지 않는** 자체 완결적 빌드

**Cherry pick:** 프로덕션 버그 수정 시 원래 빌드와 동일 리비전에서 재빌드 + 이후 특정 변경만 포함. 빌드 도구 자체도 프로젝트 리비전에 따라 버전 관리 → 지난달 빌드는 지난달 컴파일러 사용.

### 4. Enforcement of Policies and Procedures (정책·절차 강제)

릴리즈 시 특정 작업에 대한 **다층 보안·접근 제어:**

| 작업 | 설명 |
|------|------|
| 소스 코드 변경 승인 | 코드베이스 전체의 설정 파일로 관리 |
| 릴리즈 프로세스 동작 지정 | — |
| 새 릴리즈 생성 | — |
| 통합 제안(integration proposal) 승인 + cherry pick | 특정 리비전에서 빌드 요청 |
| 새 릴리즈 배포 | — |
| 빌드 설정 변경 | — |

거의 모든 변경은 **코드 리뷰 필수**. 자동 릴리즈 시스템이 릴리즈에 포함된 **모든 변경 보고서**를 생성·보관 → SRE가 문제 발생 시 빠른 트러블슈팅 가능.

## Continuous Build and Deployment: 지속적 빌드·배포

Google의 자동 릴리즈 시스템: **Rapid**

### Building — Blaze

Google의 빌드 도구 (오픈소스: **Bazel**).

- C++, Java, Python, Go, JavaScript 지원
- 빌드 타겟(JAR 등)과 의존성 정의 → Blaze가 의존 타겟 자동 빌드
- 모든 바이너리는 빌드 날짜, 리비전 번호, 빌드 ID 표시 플래그 지원

### Branching — 브랜칭

모든 코드는 **mainline(메인 브랜치)** 에 체크인. 대부분의 주요 프로젝트는 mainline에서 직접 릴리즈하지 않는다.

**Branch + Cherry pick 방식:**
1. 특정 리비전에서 mainline에서 브랜치 생성
2. 브랜치를 mainline에 **merge back 하지 않음**
3. 버그 수정은 mainline에 제출 후 cherry pick으로 릴리즈 브랜치에 포함
4. 원래 빌드 이후 mainline의 무관한 변경이 섞이지 않음 → **각 릴리즈의 정확한 내용을 알 수 있음**

### Testing — 테스트

**Continuous test:** mainline 변경마다 unit test 자동 실행 → 빌드·테스트 실패 빠른 감지.

**권장 사항:**
- continuous build의 테스트 타겟 = 릴리즈를 게이트하는 테스트 타겟
- **마지막으로 모든 테스트를 통과한 continuous test 빌드의 리비전**에서 릴리즈 생성
- 릴리즈 시 unit test 재실행 + 모든 테스트 통과 audit trail 생성 (cherry pick으로 릴리즈 브랜치가 mainline에 없는 코드를 포함할 수 있으므로)

**System-level test:** 패키징된 빌드 아티팩트에 대한 독립 테스트 환경. Rapid에서 수동 또는 자동 실행.

### Packaging — MPM (Midas Package Manager)

Blaze 규칙 기반으로 패키지 조립 (아티팩트, 소유자, 권한).

- 패키지 이름 (예: `search/shakespeare/frontend`)
- 고유 해시로 버전 관리
- 서명으로 진위 보장
- Rapid가 **빌드 ID 라벨** 적용 → 이름 + 라벨로 고유 참조
- `dev`, `canary`, `production` 등 릴리즈 단계 라벨

**라벨 이동:** 기존 라벨을 새 패키지에 적용하면 자동으로 이전 패키지에서 이동.

### Rapid — 릴리즈 워크플로우

**Blueprint:** 내부 설정 언어로 빌드/테스트 타겟, 배포 규칙, 프로젝트 소유자 정의. RBAC로 작업 권한 제어.

**전형적 릴리즈 프로세스:**
1. continuous test에서 integration revision 번호로 **릴리즈 브랜치** 생성
2. Blaze로 바이너리 컴파일 + unit test (종종 병렬). 전용 환경에서 실행
3. 빌드 아티팩트를 system test·canary 배포에 사용. canary = 프로덕션에 소수 job 시작
4. 각 단계 결과 로깅 + 마지막 릴리즈 이후 **모든 변경 보고서** 생성

Cherry pick 요청을 승인/거부할 수 있다.

### Deployment — 배포

**Rapid 직접 배포:** blueprint의 배포 정의에 따라 Borg job을 새 MPM 패키지로 업데이트.

**Sisyphus (SRE 개발):** 복잡한 배포용 범용 롤아웃 자동화 프레임워크.
- Python 클래스 확장으로 모든 배포 프로세스 지원
- 롤아웃 진행 대시보드
- Rapid가 Sisyphus에 rollout 생성 → MPM 빌드 라벨로 배포 버전 지정
- 즉시 전체 업데이트 ~ 수 시간에 걸쳐 클러스터 순차 업데이트

**서비스 리스크 프로파일에 맞는 배포:**

| 환경 | 전략 |
|------|------|
| 개발/프리프로덕션 | 시간당 빌드, 테스트 통과 시 자동 push |
| 대규모 사용자 대면 서비스 | 1개 클러스터에서 시작 → 지수적 확장 |
| 민감한 인프라 | 수일에 걸쳐 지역 간 인스턴스에 인터리빙 |

## Configuration Management: 설정 관리

설정 변경은 **불안정성의 잠재적 원인**. 릴리즈 엔지니어와 SRE가 밀접히 협업.

모든 방식에서 설정을 **주 소스 코드 저장소**에 저장하고 **엄격한 코드 리뷰** 적용.

### 4가지 설정 배포 모델

| 모델 | 설명 | 장단점 |
|------|------|--------|
| **Mainline for configuration** | main 브랜치 head에서 설정 파일 수정 → 리뷰 → 실행 시스템에 적용. 바이너리 릴리즈와 **분리** | 개념·절차 단순. job 업데이트 전까지 checked-in vs running **skew** 발생 |
| **설정을 MPM 패키지에 포함** | 바이너리와 같은 패키지 | 배포 단순 (1개 패키지). 바이너리·설정 **강결합**, 유연성 제한 |
| **MPM "configuration packages"** | 바이너리·설정을 별도 MPM 패키지로, 빌드 ID로 재구성 가능 | 각각 독립 변경 가능. 라벨(예: `much_ado`)로 함께 설치할 패키지 버전 지정 |
| **외부 저장소에서 읽기** | Chubby, Bigtable, 소스 기반 파일시스템 | 빈번·동적 변경 필요 시. 바이너리 실행 중 설정 변경 가능 |

**Cherry pick 설정 예시:** `first_folio` → `bad_quarto` 플래그 변경만 cherry pick → 설정 패키지만 재빌드·배포 (새 바이너리 빌드 불필요).

프로젝트 오너가 케이스별로 최적 모델을 선택한다.

## Conclusions: 결론

### Google만의 것이 아니다

올바른 도구, 적절한 자동화, 잘 정의된 정책이 있으면 개발자와 SRE는 릴리즈를 걱정하지 않아도 된다. 릴리즈는 **버튼 하나**만 누르면 될 수 있다.

모든 회사가 동일한 문제를 다룬다:
- 패키지 버전 관리 방법?
- Continuous build/deploy vs 주기적 빌드?
- 얼마나 자주 릴리즈?
- 어떤 설정 관리 정책?
- 어떤 릴리즈 메트릭이 관심 대상?

Google은 규모 때문에 자체 도구를 개발했지만, **정책을 먼저 정의**하는 것은 모든 회사에 해당한다.

### Start Release Engineering at the Beginning

릴리즈 엔지니어링은 종종 **사후 고려**되었지만, 플랫폼·서비스가 커지면서 초기부터 계획해야 한다.

- **제품 개발 초기**에 릴리즈 엔지니어링 리소스 예산
- 나중에 소급 적용하는 것보다 **초기에 좋은 관행**이 저렴
- 개발자, SRE, 릴리즈 엔지니어가 **함께** 일해야 한다
- 개발자가 빌드·배포 의도를 릴리즈 엔지니어에게 넘기면 안 된다

| **4대 원칙** | Self-service, High velocity, Hermetic builds, Policy enforcement |

| **Blaze/Bazel** | 헤르메틱 빌드, 의존성 자동 해결 |

| **시작 시점** | 릴리즈 엔지니어링은 제품 초기부터 투자 |
