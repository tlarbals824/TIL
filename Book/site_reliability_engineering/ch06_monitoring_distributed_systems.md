## 들어가며

Google SRE 팀은 성공적인 모니터링·알림 시스템을 구축하기 위한 기본 원칙과 모범 사례를 갖고 있다. 이 장은 **어떤 이슈가 사람을 페이징해야 하는지**, 그리고 **페이징할 만큼 심각하지 않은 이슈를 어떻게 다룰지**에 대한 가이드라인을 제시한다.

---

## Definitions: 용어 정의

모니터링 관련 용어는 업계 전반에 통일된 어휘가 없다. Google 내부에서도 사용이 다르지만, 가장 일반적인 해석은 다음과 같다.

| 용어 | 의미 |
|------|------|
| **Monitoring** | 쿼리 수·에러 수·처리 시간·서버 수명 등 실시간 정량 데이터를 수집·처리·집계·표시하는 것 |
| **White-box monitoring** | 로그, JVM Profiling Interface, 내부 통계 HTTP 핸들러 등 시스템 내부에서 노출되는 메트릭 기반 모니터링 |
| **Black-box monitoring** | 사용자 관점에서 보이는 외부 동작을 테스트 |
| **Dashboard** | 서비스 핵심 메트릭 요약 뷰. 티켓 큐 길이, 고우선순위 버그, on-call 담당자, 최근 push 등 팀 정보도 포함 가능 |
| **Alert** | 사람이 읽도록 의도된 알림. 티켓 큐, 이메일, 페이저로 전달 (각각 ticket, email alert, page로 분류) |
| **Root cause** | 수정하면 같은 방식으로 재발하지 않을 것이라는 확신을 주는 결함. 하나의 사건에 여러 근본 원인이 있을 수 있음 |
| **Node / Machine** | 물리 서버, VM, 컨테이너에서 실행 중인 단일 커널 인스턴스. 한 머신에 여러 서비스가 공존할 수 있음 |
| **Push** | 실행 중인 소프트웨어나 설정에 대한 모든 변경 |

## Why Monitor?: 왜 모니터링하는가

| 목적 | 예시 |
|------|------|
| **장기 트렌드 분석** | DB 크기와 성장 속도, DAU 증가율 |
| **시간/실험군 비교** | 스토리지 엔진 A vs B 쿼리 속도, memcache 노드 추가 효과 |
| **Alerting** | 무언가 깨졌거나 곧 깨질 것 — 즉시 또는 곧 조치 필요 |
| **대시보드 구축** | 서비스 기본 질문에 답 (4 Golden Signals 포함) |
| **Ad hoc 사후 분석 (디버깅)** | 레이턴시 급증 시 같은 시점에 무슨 일이 있었는지 |

모니터링은 비즈니스 분석, 보안 침해 분석에도 유용하지만, 이 책은 SRE 전문 영역에 집중한다.

### 페이징의 비용

사람을 페이징하는 것은 **직원 시간의 매우 비싼 사용**이다.
- 업무 중: 워크플로우 중단
- 퇴근 후: 개인 시간·수면 중단

페이지가 너무 자주 오면 직원들이 알림을 의심하고, 훑어보거나, 심지어 무시한다. 노이즈 속에 숨은 **진짜 페이지**를 놓쳐 장애가 길어진다.

> 효과적인 알림 시스템은 **높은 신호(signal), 매우 낮은 노이즈(noise)** 를 갖는다.

"뭔가 좀 이상한 것 같다"는 이유만으로 알림을 트리거해서는 안 된다 (매우 좁은 범위의 보안 감사 제외).

## Setting Reasonable Expectations: 모니터링에 대한 현실적 기대

복잡한 애플리케이션 모니터링 자체가 상당한 엔지니어링 작업이다. 10~12명 규모 SRE 팀에 보통 1~2명이 모니터링 시스템 구축·유지를 주 업무로 담당한다. 공통 인프라 일반화로 이 수는 줄었지만, 최소 1명의 "monitoring person"은 여전히 필요하다.

Google의 추세:
- **더 단순하고 빠른** 모니터링 시스템
- **사후 분석**을 위한 더 나은 도구
- 임계값을 학습하거나 인과관계를 자동 감지하는 **"마법" 시스템 회피**

장기 관측 실험(수개월~수년, 낮은 샘플링)이나 용량 계획·트래픽 예측은 더 복잡해도 허용된다.

**복잡한 의존성 계층**은 제한적 성공만 거둠. "DB가 느리면 DB 알림, 아니면 웹사이트 전체 느림 알림" 같은 규칙은 인프라가 지속적으로 리팩토링되므로 유지하기 어렵다.

> 프로덕션 문제 발생 → 페이지 → 인간 → 기본 트리아지 → 심층 디버깅까지의 **크리티컬 패스**는 모두가 이해할 수 있을 만큼 단순하고 명확해야 한다.

## Symptoms Versus Causes: 증상 vs 원인

모니터링 시스템은 두 가지 질문에 답해야 한다: **무엇이 깨졌는가(what)?** 왜 깨졌는가(why)?

| 증상 (Symptom) | 원인 (Cause) |
|---------------|-------------|
| HTTP 500/404 서빙 | DB 서버가 연결 거부 |
| 응답이 느림 | CPU 과부하(bogosort) 또는 랙 아래 눌린 이더넷 케이블(부분 패킷 손실) |
| 남극 사용자가 GIF 못 받음 | CDN이 특정 IP 블랙리스트 |
| 비공개 콘텐츠가 전 세계 공개 | 새 소프트웨어 push로 ACL 누락 |

**"what" vs "why" 구분은 신호 대 노이즈를 극대화하는 모니터링 작성의 핵심**이다.

---

## Black-Box Versus White-Box

| | Black-box | White-box |
|---|-----------|-----------|
| **관점** | 증상 지향 — "지금 시스템이 제대로 작동하지 않음" | 시스템 내부(로그, HTTP 엔드포인트) 검사 |
| **감지** | 이미 발생 중인 활성 문제 | 임박한 문제, 재시도로 가려진 실패 등 |
| **페이징** | 이미 진행 중이고 실제 증상에 기여하는 문제만 알림 — **규율 강제** | — |
| **디버깅** | — | 필수 (웹 서버와 DB 각각의 관점 필요) |

다층 시스템에서 **한 사람의 증상은 다른 사람의 원인**이다. DB SRE에게 느린 읽기는 증상이지만, 프론트엔드 SRE에게는 원인이다.

## The Four Golden Signals: 4대 황금 신호

사용자 대면 시스템에서 4가지만 측정할 수 있다면 이것에 집중하라.

### 1. Latency (지연시간)
- 요청 처리 시간
- **성공 요청과 실패 요청의 지연시간을 구분**해야 한다
- HTTP 500이 DB 연결 끊김으로 매우 빨리 반환될 수 있음 — 500을 전체 지연시간에 포함하면 오해의 소지
- 느린 에러는 빠른 에러보다 더 나쁨 → **에러 지연시간도 추적**

### 2. Traffic (트래픽)
- 시스템에 가해지는 수요의 고수준 메트릭
- 웹: HTTP req/s (정적/동적 구분), 스트리밍: 네트워크 I/O 또는 동시 세션, KV 스토리지: 트랜잭션/조회 수

### 3. Errors (에러)
- 실패 요청 비율 — 명시적(HTTP 500), 암묵적(HTTP 200이지만 잘못된 콘텐츠), 정책적(1초 SLA 초과)
- LB에서 500 캐치 vs E2E 테스트로 잘못된 콘텐츠 감지 — 난이도가 다름

### 4. Saturation (포화)
- 서비스가 얼마나 "찼는가"
- 가장 제약된 리소스 강조 (메모리 제약 → 메모리, I/O 제약 → I/O)
- 많은 시스템은 100% 이용률 전에 성능 저하 → **이용률 목표 필수**
- 99번째 백분위수 응답 시간은 포화의 **선행 지표**
- "DB 하드 드라이브가 4시간 후 가득 참" 같은 **임박 포화 예측**도 포함

4가지 신호를 모두 측정하고, 문제가 있거나(포화의 경우 거의 문제) 사람을 페이징하면 최소한 괜찮은 커버리지를 갖게 된다.

---

## Worrying About Your Tail: 꼬리 지연시간

평균(mean) 기반 모니터링의 위험:
- 평균 지연 100ms, 1000 req/s에서 **1%가 5초** 걸릴 수 있음
- 여러 백엔드의 99번째 백분위수가 프론트엔드의 **중앙값**이 될 수 있음

**해결:** 실제 지연시간 대신 **지연시간 버킷별 요청 수**를 수집 (히스토그램). 버킷 경계를 대략 3배씩 지수적으로 (0-10ms, 10-30ms, 30-100ms, 100-300ms...).

## Choosing an Appropriate Resolution: 측정 해상도

| 상황 | 권장 |
|------|------|
| CPU 부하를 1분 단위로 관찰 | 긴 스파이크를 놓칠 수 있음 |
| 99.9% 가용성 목표 웹 서비스 | 200 상태를 분당 1~2회 이상 프로빙은 과도 |
| 99.9% 가용성 하드 드라이브 | 1~2분마다 확인은 불필요할 수 있음 |

초당 CPU 측정은 비용이 크다. 서버에서 내부 샘플링(5% 버킷) 후 외부에서 집계하면 비용 절감 가능.

## As Simple as Possible: 가능한 한 단순하게

요구사항이 쌓이면 모니터링이 복잡·취약·유지보수 부담이 된다.

**단순화 가이드라인:**
- 실제 사건을 가장 자주 잡는 규칙은 **단순·예측 가능·신뢰**해야 한다
- 분기 1회 미만으로 실행되는 수집/집계/알림 설정 → 제거 검토
- 대시보드·알림에 사용되지 않는 신호 → 제거 후보

모니터링을 프로파일링, 디버깅, 로그 분석, 트래픽 검사 등과 **과도하게 결합하지 말 것**. 명확하고 느슨하게 결합된 별도 시스템 + 웹 API 통합이 더 낫다.

## Tying These Principles Together: 알림 규칙 작성 시 질문

새 알림을 작성하거나 검토할 때:

1. 이 규칙이 **긴급하고, 조치 가능하며, 능동적이거나 임박한 사용자 가시적** 조건을 감지하는가?
2. 이 알림을 **무시할 수 있는가**? 언제, 왜? 어떻게 피할 수 있는가?
3. 이 알림이 **사용자에게 부정적 영향**을 확실히 나타내는가? 드레인된 트래픽·테스트 배포 등 필터링 필요?
4. 이 알림에 **대응 조치**가 있는가? 긴급한가, 아침까지 기다릴 수 있는가? 자동화 가능한가? 장기 수정인가 임시 우회인가?
5. 같은 이슈로 **다른 사람도 페이징**되는가?

### 페이지 철학

- 페이지가 울릴 때마다 **긴급감**으로 반응할 수 있어야 한다. 하루 몇 번 이상이면 피로
- **모든 페이지는 조치 가능**해야 한다
- **모든 페이지 대응은 지능을 요구**해야 한다. 로봇 대응이면 페이지가 아니다
- 페이지는 **새로운 문제**이거나 **이전에 본 적 없는 이벤트**여야 한다

**증상(symptom)에 훨씬 더 많은 노력을 쏟고**, 원인(cause)은 **매우 확실하고 임박한 것**만 걱정하라.

## Monitoring for the Long Term: 장기적 모니터링

모니터링 결정은 **장기 목표**를 염두에 두고 내려야 한다. 오늘의 모든 페이지는 내일 시스템 개선에서 사람을 빼앗는다. 때로는 **단기 가용성·성능을 의도적으로 희생**해 장기 전망을 개선하는 것이 맞다.

### Bigtable SRE: 과도한 알림의 사례

Bigtable SLO가 synthetic client의 **평균 성능** 기반이었고, 하위 5%가 평균을 끌어내렸다. 이메일·페이징 알림이 대량 발생 → 엔지니어링 시간 소모, 실제 사용자 영향 사건 놓침.

**3-pronged 접근:**
1. Bigtable 성능 개선 (장기)
2. SLO 목표를 **75번째 백분위수 지연시간**으로 일시 완화
3. 이메일 알림 비활성화

→ on-call 엔지니어가 실제 작업 가능, 장기 문제 해결 가속. **일시적으로 알림을 줄인 것이 더 빠른 개선을 가능하게 했다.**

### Gmail: 예측 가능한 인간 대응

초기 Gmail은 Workqueue 기반. 개별 task de-schedule 시 알림 → 수천 개 task 중 하나씩 알림, 유지 불가.

SRE가 스케줄러를 "살짝 밀어주는" 도구를 만들었고, 전체 루프 자동화 논의가 있었다. 하지만 **해킹이 영구 수정을 무한정 지연**할 수 있다는 우려.

> **정형화된(rote) 알고리즘적 대응이 필요한 페이지는 경고 신호.** 자동화를 꺼리는 팀은 기술 부채 정리에 대한 자신감이 부족하다는 뜻 — 에스컬레이션 필요.

### The Long Run

모든 페이지를 고립된 이벤트가 아닌, **전체 페이징 수준이 건강한 시스템·팀·장기 전망**을 향하는지 관점에서 봐야 한다. 분기 보고서에 shift당 incident 수를 경영진과 공유한다.

## 핵심 정리

| 원칙 | 내용 |
|------|------|
| **페이징** | 증상(symptom) 중심. 원인(cause)은 디버깅 보조 |
| **4 Golden Signals** | Latency, Traffic, Errors, Saturation |
| **단순함** | 크리티컬 패스는 모두가 이해 가능해야 함 |
| **이메일 알림** | 가치 제한적, 노이즈에 쉽게 잠김 → 대시보드 선호 |
| **장기 관점** | 단기 가용성 희생이 장기 안정성에 도움이 될 수 있음 |

건강한 모니터링·알림 파이프라인은 단순하고 추론하기 쉽다. 달성 가능한 목표에 맞추고, 신속한 진단을 지원하는 모니터링을 갖추면 성공적인 on-call과 제품 운영이 가능하다.
