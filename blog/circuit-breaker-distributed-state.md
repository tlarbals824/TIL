# 분산 환경에서 서킷브레이커, 파드마다 관리하면 생기는 일

결제 시스템을 개발하면서 서킷브레이커를 적용하다 보니 분산 환경에서 예상치 못한 문제를 마주쳤습니다. 그 고민의 과정과 해결 방법을 정리해보려 합니다.

---

## 서킷브레이커란?

서킷브레이커는 전기 회로의 차단기에서 이름을 따온 패턴입니다. 외부 서비스 호출이 연속으로 실패하면 더 이상 호출을 시도하지 않고 즉시 에러를 반환(fail fast)해서, 장애가 전파되는 것을 막습니다.

상태는 세 가지입니다.

- **Closed**: 정상. 요청을 그대로 통과시킵니다.
- **Open**: 차단. 요청을 보내지 않고 즉시 fallback을 반환합니다.
- **Half-Open**: 회복 탐색. 제한적으로 요청을 보내보고, 성공하면 Closed로, 실패하면 다시 Open으로 전환합니다.

실패율이 임계치를 넘으면 Closed → Open으로 열리고, 일정 시간이 지나면 Half-Open을 거쳐 다시 닫힙니다.

---

## 상황 설정: PG사 장애와 fallback 결제

결제 시스템에서 PG사(Payment Gateway) 장애는 꽤 현실적인 시나리오입니다. PG사 API 호출이 연속으로 타임아웃 나거나 에러를 반환하기 시작하면, 단순히 재시도를 반복하는 것은 상황을 악화시킵니다. 이미 장애 중인 서버에 요청을 계속 보내봐야 소용없고, 커넥션 풀만 고갈됩니다.

그래서 서킷브레이커를 도입했습니다. 실패율이 임계치를 초과하면 서킷이 열리고, 이 시점에 이벤트가 발행됩니다. 이 이벤트를 통해 **대체 결제 수단으로 fallback**되며, 운영팀 알림이나 모니터링 파이프라인과도 연동합니다.

```
결제 요청
    │
    ▼
[서킷브레이커]
    │
    ├─ Closed ──────────────► PG사 API 호출
    │
    └─ Open ────────────────► 이벤트 발행
                                     │
                                     ▼
                              대체 결제 수단 (fallback)
                              운영팀 알림 / 모니터링
```

여기까지는 교과서적인 서킷브레이커 적용입니다.

## "각 파드에서 서킷을 관리하면 되지 않나?"

서비스는 쿠버네티스 위에서 여러 파드로 수평 확장되어 운영됩니다. Resilience4j 같은 서킷브레이커 라이브러리를 붙이면 각 파드 내 메모리에서 상태를 관리합니다.

언뜻 보면 문제없어 보입니다.

- 각 파드가 독립적으로 PG사 호출 실패를 감지
- 임계치 초과 시 해당 파드의 서킷이 열림
- PG사 장애가 전 파드에 고르게 영향을 준다면, 결국 모든 파드의 서킷이 열림

실제로 **장애 발생 직후**에는 이 구조가 잘 동작합니다. PG사가 터지면 모든 파드가 비슷한 시점에 실패를 쌓고, 임계치를 넘어 서킷이 열립니다.

문제는 **장애 복구 시점**에 발생합니다.

---

## 문제: 파드마다 서킷이 닫히는 시점이 다르다

서킷브레이커는 Open 상태에서 일정 시간이 지나면 Half-Open으로 전환되고, 소수의 요청을 실제로 보내봅니다. 성공하면 Closed로 돌아옵니다.

그런데 각 파드의 서킷이 열린 시점이 정확히 같지 않습니다.

```
파드 A: 14:00:00 서킷 Open → 14:01:00 Half-Open → 14:01:05 Closed
파드 B: 14:00:03 서킷 Open → 14:01:03 Half-Open → 14:01:08 Closed
파드 C: 14:00:07 서킷 Open → 14:01:07 Half-Open → 14:01:12 Closed
```

PG사가 14:01:00에 복구됐다고 가정하면:

- 파드 A: 복구 직후 Half-Open 진입, PG사 호출 성공 → Closed
- 파드 B, C: 아직 Open 상태 → 계속 fallback 처리

이 구간 동안 **동일한 요청이 어느 파드로 라우팅되느냐에 따라 결과가 달라집니다.** 파드 A로 가면 정상 결제, 파드 B나 C로 가면 여전히 fallback. 사용자 입장에서는 같은 시점에 결제를 시도해도 결과가 다를 수 있습니다.

더 심각한 경우도 있습니다. 서킷 상태 변화에 따라 외부 상태(예: DB에 저장된 결제 수단 설정)를 함께 업데이트하는 구조라면, 파드마다 서킷이 닫히는 시점이 달라 **외부 상태와 서킷 상태가 불일치**하는 구간이 생깁니다. 이미 복구된 외부 상태를 보고 원래 PG로 요청을 시도하지만, 아직 Open 상태인 파드에서는 그 요청이 차단되어 **실제 결제 실패**로 이어질 수 있습니다.

## 근본 원인: 분산된 서킷 상태

파드별 서킷 관리의 근본 문제는 **서킷 상태가 분산되어 있다**는 것입니다.

각 파드는 자신이 관측한 실패만을 기준으로 서킷을 열고 닫습니다. 클러스터 전체에서 PG사 호출이 얼마나 실패했는지는 모릅니다. 그래서:

- 실패가 특정 파드에 몰리면 → 일부 파드만 서킷이 열릴 수 있습니다
- 복구 시점에 → 파드마다 다른 시점에 서킷이 닫힙니다

---

## 잠깐, 원래 서킷브레이커는 파드별로 관리하는 게 맞지 않나?

맞습니다. 서킷브레이커는 원래 **각 인스턴스가 독립적으로 외부 호출을 보호**하기 위한 패턴입니다. Resilience4j도 이 철학을 따라 인메모리에서 상태를 관리하도록 설계되어 있고, 파드마다 서킷 상태가 다른 것은 버그가 아니라 의도된 동작입니다.

그런데 이번 상황은 조금 다릅니다. 서킷 상태 변화가 **외부 상태(DB에 저장된 결제 수단)를 함께 변경**하는 구조이기 때문에, 파드 간 서킷 상태 불일치가 곧 외부 상태와의 불일치로 이어집니다. 서킷브레이커 자체의 문제가 아니라, 서킷 상태에 연동된 부수 효과(side effect)가 파드 간에 일관되지 않는 것이 문제입니다.

일반적인 서킷브레이커 용도라면 파드별 관리로 충분합니다. 이 고민은 서킷 상태가 단순한 요청 차단을 넘어 외부 상태를 구동하는 특수한 상황에서 비롯된 것입니다.

---

## 해결: Redis pub/sub으로 상태 전파

핵심 문제는 **파드마다 서킷이 열린 시점이 달라 닫히는 시점도 달라진다**는 것입니다. 실패율 집계를 통일하는 것보다, 상태 전환 자체를 동기화하는 것이 훨씬 단순하게 해결됩니다.

Resilience4j는 분산 환경을 공식 지원하지 않습니다. 내부 슬라이딩 윈도우는 JVM 메모리에 고정되어 있고 교체할 수 없습니다. 하지만 상태 변화 이벤트 API는 제공하므로, 이를 활용해 Redis pub/sub으로 상태 전환을 전파할 수 있습니다.

```
파드 A: 서킷 Open 감지
    │
    ├─ Redis channel에 "OPEN" publish
    │
파드 B, C: 메시지 수신
    │
    └─ transitionToForcedOpenState() 즉시 호출
```

파드 A가 Open되는 순간 모든 파드가 동일 시점에 Open 상태가 됩니다. 이후 Half-Open → Closed 타이밍도 자연스럽게 동일해집니다.

**상태 변화 감지 → Redis publish**

```kotlin
circuitBreaker.eventPublisher
    .onStateTransition { event ->
        val state = event.stateTransition.toState.name
        redisTemplate.convertAndSend("circuit:pg:state", state)
    }
```

**Redis subscribe → 강제 상태 동기화**

```kotlin
@Component
class CircuitBreakerSyncListener(
    private val circuitBreaker: CircuitBreaker
) : MessageListener {

    override fun onMessage(message: Message, pattern: ByteArray?) {
        when (String(message.body)) {
            "OPEN" -> circuitBreaker.transitionToForcedOpenState()
            "CLOSED" -> circuitBreaker.transitionToClosedState()
            "HALF_OPEN" -> circuitBreaker.transitionToHalfOpenState()
        }
    }
}
```

publish한 파드 자신도 메시지를 수신하지만, Resilience4j는 이미 해당 상태이면 전환을 무시하므로 별도 처리가 필요하지 않습니다.

---

## 트레이드오프

**Redis가 SPOF가 됩니다.** pub/sub 채널이 다운되면 상태 전파가 되지 않아 파드 간 불일치가 다시 발생할 수 있습니다. Redis 자체의 가용성을 높이거나, 장애 시 각 파드가 독립적으로 동작하도록 fallback 정책을 정해둬야 합니다.

**실패율 집계는 여전히 파드별입니다.** 이 방식은 상태 전환 동기화만 해결합니다. 트래픽이 특정 파드에 몰리는 상황에서는 그 파드만 먼저 임계치에 도달해 서킷을 열 수 있습니다. 다만 Open 이벤트가 전파되므로 다른 파드도 즉시 따라 열립니다.

---

## 정리

| | 파드별 서킷 관리 | Redis pub/sub 상태 동기화 |
|---|---|---|
| 서킷 전환 시점 | 파드마다 다름 | 클러스터 전체 동일 |
| 실패율 기준 | 해당 파드의 관측값 | 해당 파드의 관측값 |
| 복구 시 상태 동기화 | 시점 불일치 발생 | 자동 동기화 |
| 기존 라이브러리 변경 | 없음 | 없음 (이벤트 리스너 추가만) |
| 구현 복잡도 | 낮음 | 낮음 |
| 단일 장애 지점 | 없음 | Redis pub/sub |

단순한 서비스에서 서킷브레이커는 파드별로 관리해도 충분합니다. 하지만 PG사 fallback처럼 **서킷 상태와 외부 상태가 함께 연동되는 구조**에서는 파드 간 상태 불일치가 실제 장애로 이어질 수 있습니다. Redis pub/sub을 활용하면 Resilience4j를 거의 건드리지 않으면서 이 문제를 해결할 수 있습니다.
