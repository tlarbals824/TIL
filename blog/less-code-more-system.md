# 더 적은 코드로 해결하기까지

분산 환경에서 서킷브레이커 파드 간 상태를 동기화하는 코드를 작성하다가 멈추게 됐습니다. Redis pub/sub으로 상태를 전파하는 구현을 절반쯤 마쳤을 때, 문득 이 코드가 앞으로 얼마나 많은 비용을 만들어낼지 떠올랐습니다. 그 고민의 과정을 정리해보려 합니다.

---

## 코드로 모든 문제를 해결하던 시절

개발자로서 문제를 마주하면 자연스럽게 코드를 작성했습니다. 요구사항이 생기면 구현하고, 버그가 발생하면 처리 로직을 추가하고, 상태가 맞지 않으면 동기화 코드를 붙였습니다.

이 접근은 서비스 규모가 작을 때는 잘 작동했습니다. 문제가 명확하고, 해결책도 명확했습니다.

---

## 스케일이 커지면서 코드가 복잡도를 만들어냈다

서비스가 분산 환경으로 확장되면서 상황이 달라졌습니다. 특히 **글로벌 상태를 일관되게 유지하는 것**이 문제가 됐습니다.

단일 인스턴스에서는 메모리 안에서 상태를 관리하면 됩니다. 하지만 파드가 여러 개로 늘어나면, 각 인스턴스가 서로 다른 상태를 갖게 됩니다. 이를 맞추기 위해 코드를 추가하면, 그 코드가 또 다른 상태와 의존성을 만들어냅니다.

코드가 문제를 해결하는 게 아니라, 코드가 복잡도를 생산하는 상황이 됐습니다.

---

## 서킷브레이커 파드 동기화: 코드로 해결하려 했을 때

[분산 환경에서 서킷브레이커, 파드마다 관리하면 생기는 일](https://velog.io/@tlarbals824/%EB%B6%84%EC%82%B0-%ED%99%98%EA%B2%BD%EC%97%90%EC%84%9C-%EC%84%9C%ED%82%B7%EB%B8%8C%EB%A0%88%EC%9D%B4%EC%BB%A4-%ED%8C%8C%EB%93%9C%EB%A7%88%EB%8B%A4-%EA%B4%80%EB%A6%AC%ED%95%98%EB%A9%B4-%EC%83%9D%EA%B8%B0%EB%8A%94-%EC%9D%BC)에서 다뤘던 문제입니다. 각 파드가 독립적으로 서킷 상태를 관리하면, 장애 복구 시점에 파드마다 서킷이 닫히는 시간이 달라져 동일한 요청도 결과가 다르게 됩니다.

처음 떠올린 해결책은 Redis pub/sub이었습니다. 한 파드에서 서킷이 전환되면 채널에 메시지를 발행하고, 다른 파드들이 구독해서 상태를 강제 전환하는 구조입니다.

```
파드 A: 서킷 Open
    │
    ├─ Redis channel에 "OPEN" publish
    │
파드 B, C: 메시지 수신 → 상태 강제 전환
```

구현을 시작했습니다. 상태 변화 감지, publish 로직, subscribe 리스너, 자기 자신이 발행한 메시지 무시 처리, Redis 장애 시 fallback... 코드가 쌓이기 시작했습니다. 그리고 이 코드는 앞으로 계속 관리해야 할 대상이 됩니다.

이 시점에서 멈추게 됐습니다.

---

## 이미 있는 시스템을 활용하면 어떨까

서킷브레이커 상태는 이미 Prometheus 메트릭으로 수집되고 있었습니다. Grafana에서 PromQL로 각 파드의 서킷 상태를 한눈에 조회할 수 있었습니다.

여기서 한 단계 더 나아갔습니다. Grafana Alert를 설정해 서킷 상태 변화를 감지하고, 웹훅으로 전달하는 방식입니다.

```
Prometheus 메트릭 수집
    │
Grafana Alert (PromQL 조건)
    │
웹훅 전달 → 상태 동기화 처리
```

결과적으로 새로운 코드가 추가되지 않았고, 오히려 기존에 장애 복구를 위해 작성하던 코드가 줄었습니다. 상태 추적, 알림, 파이프라인이 이미 모니터링 시스템 안에 있었고, 그것을 연결하기만 하면 됐습니다.

---

## 코드를 만드는 것이 다가 아니다

이 경험이 하나의 패턴으로 보이기 시작했습니다.

코드를 작성하는 것은 문제를 해결하는 동시에 미래의 유지보수 비용을 예약하는 일입니다. 기능이 추가될수록, 인프라가 복잡해질수록, 이 비용은 선형이 아닌 방식으로 증가합니다.

반면 기존 시스템 간 통합은 다릅니다. 이미 검증된 시스템들이 각자의 역할을 맡고, 그 사이를 연결하는 것이 우리의 역할이 됩니다. 코드가 적고, 검증된 동작에 기댈 수 있으며, 운영 가시성도 이미 확보되어 있습니다.

---

## AI 시대에 이 감각이 흐려진다

AI로 코드 생산성이 크게 올랐습니다. 몇 줄의 설명으로 수십 줄의 코드가 만들어집니다. Redis pub/sub 구현도 AI를 활용하면 빠르게 완성할 수 있습니다.

문제는 AI가 생산한 코드도 유지보수 비용은 똑같이 발생한다는 것입니다. 오히려 코드를 손쉽게 만들 수 있게 되면서, "코드를 추가하는 것이 옳은가"를 자문하는 순간이 줄어듭니다. 작성 비용이 낮아지면 당연히 더 많이 만들게 되고, 그 코드들은 시간이 지나며 조용히 부채로 쌓입니다.

AI를 활용하면 할수록, **무엇을 만들지 않을 것인가**를 판단하는 능력이 더 중요해집니다.

그리고 이 판단은 평소에 시스템을 공부하지 않으면 하기 어렵습니다. 기존 시스템이 무엇을 할 수 있는지 모르면, 코드로 직접 구현하는 것 외에 선택지가 없습니다. Grafana가 alert와 웹훅을 지원한다는 것을 알고 있어야, Redis pub/sub 대신 그것을 선택할 수 있습니다.

---

## 정리

| | 코드로 직접 구현 | 기존 시스템 통합 |
|---|---|---|
| 초기 구현 | 자유도 높음 | 시스템 이해 필요 |
| 코드량 | 많음 | 적음 |
| 유지보수 비용 | 직접 부담 | 시스템이 흡수 |
| 운영 가시성 | 별도 구축 필요 | 기본 제공 |
| AI 활용 시 | 빠르게 만들 수 있음 | 판단이 선행되어야 함 |

코드를 잘 작성하는 것만큼, 코드를 작성하지 않는 선택도 실력입니다. AI가 코드를 쉽게 만들어주는 지금, 그 선택의 가치는 오히려 더 높아지고 있다고 생각합니다.
