# TIL 블로그 작성 지침

이 저장소는 velog(@tlarbals824) 기술 블로그 초안을 관리합니다.

---

## 말투 및 문체

- **존댓말(합쇼체)** 사용. 반말 혼용 금지 ("함정이다" → "함정입니다", "적용하자" → "적용하시길 권장합니다")
- 친근하면서도 전문적인 톤 유지
- 짧은 문장과 긴 설명 문장을 교차하는 혼합형
- 불필요한 연결 문구 최소화 — 다이어그램이나 코드는 설명 없이 바로 제시

## 1. 기술 블로그 글 구조 (기준: `blog/circuit-breaker-distributed-state.md`)

아래 흐름을 따릅니다.

1. **제목**: 문제 상황을 암시하는 짧고 구체적인 제목
2. **도입부**: `## 들어가며` 헤딩 없이 바로 본문. 개인 경험 → 고민 → "정리해보려 합니다" 형식
3. **개념 설명**: 교과서적 정의가 필요할 때만 포함
4. **상황 설정**: 실무 맥락 + ASCII 다이어그램
5. **문제 제기**: "~하면 되지 않나?" 형태의 반문으로 독자를 끌어들임
6. **문제 발견**: 실제로 발생하는 문제를 구체적으로 기술
7. **근본 원인**: 왜 그 문제가 생기는지 설명
8. **해결**: 코드 예시 포함
9. **트레이드오프**: 해결책의 한계와 주의사항
10. **정리**: 비교표 + 한두 문장 마무리

### 기술 블로그 도입부 원칙

- 교과서적 정의로 시작하지 않음
- 실무에서 어떤 문제를 맞닥뜨렸는지 먼저 제시
- 어떤 서비스/라이브러리를 도입했는지, 왜 직접 구현하게 됐는지 등 맥락을 구체적으로 포함

### 기술 블로그 맥락 수집

글을 개선하기 전에 아래를 확인합니다.

- 실제 도입한 서비스/라이브러리 이름 (예: GrowthBook)
- 직접 구현하게 된 이유
- 실무에서 마주친 구체적인 문제 상황
- 인프라 구성 (CDN 유무, 분산 환경 등)

## 2. 회고 블로그 글 구조 (기준: `blog/less-code-more-system.md`)

아래 흐름을 따릅니다.

1. **제목**: 과도하게 자극적이지 않게. 경험이나 변화를 담은 짧은 제목
2. **도입부**: 헤딩 없이 바로 본문. 특정 경험에서 촉발된 고민 → "정리해보려 합니다" 형식
3. **이전 접근**: 기존에 어떻게 생각하고 행동했는지
4. **전환점**: 무엇이 생각을 바꾸게 했는지 — 구체적인 사례로 제시
5. **깨달음**: 사례에서 도출한 패턴이나 인사이트
6. **확장**: 더 넓은 맥락(AI, 조직, 성장 등)으로 연결
7. **정리**: 비교표 또는 한두 문장 마무리

### 회고 블로그 작성 원칙

- 구현 코드는 최소화 — 흐름 다이어그램 수준으로만 표현
- 내부 아키텍처나 시스템 구조가 드러나는 코드·API 노출 금지
- 기술적 사실보다 **판단의 변화**를 중심에 놓음
- 외부 글(본인 velog 포함) 링크로 기술 맥락을 보완 가능

---

## 파일 작업 규칙

- 글 구조를 대폭 변경할 때는 **반드시 `.backup.md`로 원본 백업** 후 작업
  - 예: `sse-server-to-server.md` → `sse-server-to-server.backup.md`
- AI 멘트("추가해드릴 수 있어요" 등) 제거
- 제목에 "블로그 초안" 등 메타 표현 포함 금지

## 태그 선정 기준

- **기술 블로그**: 핵심 기술 스택 위주로 3~5개. 예: `SSE`, `Spring WebClient`, `Reactor`, `Feature Flag`
- **회고 블로그**: 주제 키워드 위주로 3~5개. 예: `회고`, `Grafana`, `분산시스템`
- 너무 광범위한 태그(`백엔드`, `개발`) 지양
